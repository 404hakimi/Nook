# 独享 ISP IP 代理服务 - 技术实现文档

## 一、项目概述

### 业务定义

为 AI 开发者用户提供独享机房 ISP IP 的代理服务。核心特征:

- 一人一 IP 不超售
- 下行限速 30Mbps
- 用户通过 CDK 卡密充值获取使用时长
- 不限制用户使用场景,只做技术服务提供

### 用户接入方式

- Web 用户中心(注册、充值、查看 IP/订阅信息、流量监控)
- 订阅链接(Clash Verge、v2rayN、sing-box 等主流客户端导入即用)
- PC 客户端(第二阶段开发,内嵌 sing-box 内核做 TUN 模式接入)

### 销售渠道(第一阶段)

闲鱼挂单 + 手动发卡:

- 闲鱼商品包装为"开发者节点服务""海外开发环境配置"等,规避敏感词
- 用户拍下后联系卖家,人工生成 CDK 发给买家
- 买家在 Web 用户中心注册账号并兑换 CDK 开通服务
- 管理后台提供 CDK 批量生成 + 单个生成功能,方便运营人员快速发卡

### 技术栈选型

- 后端: Java 17 + Spring Boot 3.x + MyBatis-Plus + MySQL 8 + Redis 7
- 代理内核: Xray-core(服务端)
- 前端: Vue 3 + Element Plus + TypeScript
- 通信协议: gRPC(Java ↔ Xray)、HTTPS REST(前端 ↔ Java)
- 部署: Docker Compose + Nginx 反代

## 二、整体架构

### 拓扑结构

整个系统分为四层:

**第一层:销售入口**

闲鱼挂单引流。买家拍下后通过闲鱼私聊联系,运营人员在管理后台生成 CDK 并发送给买家。

**第二层:业务后端层**

Java Spring Boot 应用提供所有核心业务逻辑,包括用户管理、CDK 生成与兑换、套餐管理、订阅链接生成、流量统计、节点调度等。

**第三层:代理控制层**

Java 后端通过 gRPC 协议调用各台出口服务器上的 Xray 实例,动态管理 inbound 和 user。每台出口服务器一个 Xray 实例,承载多个独立的 inbound(每个独享 IP 用户对应一个 inbound)。

**第四层:出口节点层**

每台 VPS 上跑 Xray,绑定多个独立 IP 地址,通过 inbound 的 listen 字段实现 IP 隔离。同时配置 tc/iptables 实现带宽限制。

### 网络拓扑

入口与出口分离的双层架构:

- **入口机**: DMIT 美西 GIA 服务器,国内访问优化,作为用户的接入点
- **出口机**: RackNerd / HostHatch 等便宜 IDC,提供大量独享 IP

国内用户通过 Clash 连接到入口机,入口机通过内网或公网中转到出口机的指定 IP,再从出口机的独享 IP 出网到目标网站。这样既保证国内访问速度,又能利用便宜 IDC 的 IP 资源。

第一阶段可以简化为单层架构: 用户直接连接出口机的 IP,等用户量上来后再加入口中转层。

## 三、数据库设计思路

### 核心表清单

**用户系统相关**

- 用户表: 基础账户信息、密码哈希、邮箱手机号、当前套餐 ID、套餐到期时间、注册时间、最后登录时间、设备指纹、状态字段
- 用户会话表: JWT token 的服务端记录,用于强制下线
- 用户操作日志表: 登录、兑换、换 IP 等关键操作的审计日志

**资源管理相关**

- 服务器表: 每台 VPS 的连接信息(host、SSH 端口、Xray gRPC 端口、gRPC 认证凭据)、总带宽、可用 IP 数、状态
- IP 池表: 每个独立 IP 一行,记录 IP 地址、所属服务器 ID、绑定的端口、当前状态(available / occupied / testing / blacklisted / cooling)、当前分配用户、AI 平台兼容性测试结果(chatgpt_ok、claude_ok、gemini_ok 三个布尔字段)、scamalytics 评分、最后健康检查时间、被分配次数
- Xray inbound 表: 每个 inbound 一行,对应一个用户的独享通道,记录 inbound tag、监听 IP、监听端口、协议类型、UUID、关联用户 ID、关联 IP ID

**商业系统相关**

- 套餐表: 套餐 ID、名称、类型(试用 / 月 / 季 / 半年 / 年)、时长天数、价格、附带的 IP 类型、是否上架、显示顺序
- CDK 表: 卡密码、关联套餐 ID、生成批次、状态(unused / used / disabled / expired)、生成时间、使用时间、使用者用户 ID、备注(可选,记录是哪个闲鱼买家、订单号等)
- 订单流水表: 所有用户操作的财务流水,包括 CDK 兑换、IP 切换、套餐变更等
- 续费记录表: 用户每次续费的明细,用于统计 LTV 和续费率

**监控统计相关**

- 流量日志表: 按用户按天聚合的上下行流量统计
- IP 健康日志表: 每次 IP 健康检查的结果记录,用于趋势分析
- 系统告警日志表: 服务器宕机、IP 大批量被封等异常事件

### 关键设计决策

- 用户与 IP 的绑定关系存在 IP 池表的 assigned_user_id 字段中,而不是单独建关联表(原因是一对一关系,用关联表反而复杂)
- CDK 不预生成 UUID,而是用算法生成短码(如 16 位数字字母混合),方便用户输入。生成后明文存数据库(便于运营查询),原文返回给运营人员
- 订阅链接不存数据库,每次请求时根据用户当前绑定的 inbound 实时生成,避免数据不一致
- 流量统计采用增量上报模式: Java 后端每 5 分钟从 Xray 拉取每个用户的累计流量,与上次拉取的差值就是这 5 分钟内的流量,写入流量日志表的当天累计字段

## 四、Xray 服务端配置思路

### 单台服务器的 Xray 配置结构

每台出口服务器装一个 Xray 实例,配置文件结构如下:

- **api 模块**: 开启 HandlerService 和 StatsService,监听本地 127.0.0.1 的固定端口(如 10085),通过 dokodemo-door 协议接受 gRPC 调用
- **stats 模块**: 开启,policy 配置中为不同 level 设置流量统计开关和限速参数。Level 0 用于普通用户,开启 statsUserUplink 和 statsUserDownlink,bufferSize 设为合理值
- **inbounds 数组**: 初始为空(除了 api 那个),所有用户 inbound 都通过 gRPC 动态添加
- **outbounds**: 配置直接出网,不需要复杂路由。如果某些 IP 需要走特定出口,再配置 routing 规则按 inboundTag 分流到不同 outbound

### 用户 inbound 的设计

每个独享 IP 用户对应一个独立的 inbound,关键字段:

- **tag**: 唯一标识,建议格式 `user_{userId}_{ipId}`,方便定位
- **listen**: 绑定到该用户独享的 IP 地址(VPS 上要先把这些 IP 通过 ip addr add 添加到网卡上)
- **port**: 协议监听端口,建议每个用户用不同端口避免冲突,可以从 10000 开始递增
- **protocol**: 建议 vless 协议,配合 reality 或 ws+tls 传输方式
- **settings.clients**: 包含一个 client,UUID 随机生成,level 设为 0,email 字段填用户标识用于流量统计
- **streamSettings**: 根据选择的传输方式配置,reality 模式不需要域名和证书最省心

### IP 绑定到网卡

VPS 默认只有一个主 IP 在网卡上激活,附加 IP 需要手动添加:

通过 `ip addr add 1.2.3.4/24 dev eth0` 命令添加每个附加 IP,但重启后会丢失。永久生效需要写入 netplan 配置(Ubuntu 22+)或 /etc/network/interfaces(Debian)。

建议写一个初始化脚本,从数据库读取该服务器分配的所有 IP,开机自动绑定。

### 限速实现

Xray 自带的 policy 限速精度一般,更可靠的方案是 Linux tc:

为每个独享 IP 创建一个 tc class,使用 HTB 队列规则,设置 ceil 30Mbit。通过 iptables 给经过该 IP 的流量打 mark,tc 根据 mark 分流到对应 class 实现按 IP 限速。

具体实现:

1. 开机时为网卡创建 root qdisc(htb 类型),为每个独享 IP 创建一个子 class,限速 30Mbit
2. iptables 的 OUTPUT 链根据源 IP 打 mark
3. tc filter 根据 mark 分类

这套配置写成 shell 脚本,IP 增删时 Java 后端通过 SSH 调用脚本动态调整。

## 五、Java 后端核心模块设计

### 模块划分

按业务领域划分包结构:

- **auth 模块**: 注册、登录、JWT 签发与校验、密码加密、邮箱/短信验证码
- **user 模块**: 用户信息管理、套餐查询、流量查询
- **cdk 模块**: CDK 生成、批次管理、兑换逻辑、运营查询接口
- **plan 模块**: 套餐 CRUD、价格管理
- **node 模块**: 服务器管理、IP 池管理、IP 健康检查
- **xray 模块**: Xray gRPC 客户端封装、inbound 管理、用户管理
- **subscription 模块**: 订阅链接生成(Clash YAML、v2rayN 链接、sing-box JSON)
- **traffic 模块**: 流量统计采集、聚合、查询
- **monitor 模块**: 系统监控、告警、日志
- **admin 模块**: 管理后台 API(包括 CDK 生成与发放界面)

### Xray gRPC 对接的实现思路

#### 第一步: proto 文件的获取与编译

从 Xray-core 仓库拿到所需的 proto 文件,主要是:

- `app/proxyman/command/command.proto` (HandlerService)
- `app/stats/command/command.proto` (StatsService)
- `common/protocol/user.proto` (用户定义)
- `proxy/vless/account.proto` (VLESS 账户定义)
- `common/serial/typed_message.proto` (通用序列化)

把这些 proto 文件放到 Java 项目的 `src/main/proto` 目录,配置 protobuf-maven-plugin 编译生成 Java stub。

#### 第二步: gRPC 连接管理

封装一个 XrayClient 类,每个服务器一个实例,包含:

- 一个 ManagedChannel 连接到该服务器的 Xray gRPC 端口
- HandlerServiceBlockingStub 用于 inbound 和 user 管理
- StatsServiceBlockingStub 用于流量查询
- 连接保活与重连逻辑(gRPC 自带 keepalive)
- 失败重试机制

由于 Xray gRPC 默认监听 127.0.0.1 不对外,Java 后端要么部署在同一台机器(不推荐,混部不利于运维),要么通过 SSH 隧道转发,要么用 Nginx mTLS 反代。**建议用 SSH 隧道方案**:

Java 后端启动时,对每台远程 Xray 服务器建立 SSH 端口转发,把远程的 127.0.0.1:10085 映射到本地的不同端口(10085 / 10086 / 10087...)。XrayClient 连接到本地的对应端口即可。

SSH 隧道用 jsch 或 sshj 库实现,需要处理断线重连、心跳保活。

#### 第三步: 核心操作的封装

XrayManager 提供高层 API:

- `addInbound(serverId, inboundConfig)`: 添加新 inbound,对应给用户开通服务
- `removeInbound(serverId, tag)`: 删除 inbound,对应用户到期
- `updateInboundUser(serverId, tag, user)`: 更新 inbound 的用户配置
- `queryUserTraffic(serverId, email)`: 查询单用户流量(uplink + downlink)
- `queryUserTrafficAndReset(serverId, email)`: 查询并重置(用于按周期统计)

每个操作内部调用对应的 gRPC stub 方法,处理异常和重试。

#### 第四步: 配置同步(Reconciler)

数据库中的 inbound 配置是真实状态,Xray 内存中的配置是运行状态,两者可能不一致(Xray 重启后内存配置清空)。需要一个 Reconciler:

每次 Xray 服务器重连或定期(如每小时)触发一次 reconcile: 从数据库读取该服务器应有的 inbound 列表,从 Xray 读取实际运行的 inbound 列表,对比差异,缺失的补上,多余的删除。

这个机制对运维至关重要,Xray 进程重启或被 OOM kill 后能自动恢复。

### CDK 生成与兑换流程

#### CDK 生成(管理端)

运营人员通过管理后台生成 CDK,支持两种模式:

**单个生成**: 闲鱼买家拍下后,运营人员选择对应套餐(月卡/季卡/年卡),点击"生成 CDK",系统返回卡密。运营人员可在备注栏填闲鱼订单号或买家昵称便于追溯。

**批量生成**: 一次生成 N 个相同套餐的 CDK,导出 Excel 或文本文件。适合预先囤库存,运营人员发卡时直接从库存挑一个未用的发出。

CDK 生成算法:

- 长度 16 位,字母大写 + 数字混合,排除易混淆字符(0/O、1/I)
- 随机生成,数据库唯一索引保证不重复
- 生成时记录: 生成人、生成时间、关联套餐、状态置为 unused

管理后台 CDK 页面提供以下功能:

- CDK 列表查看(筛选: 状态、套餐、生成时间、批次)
- 单个生成 / 批量生成
- 失效作废(误生成时手动作废)
- 备注编辑(填写买家信息)
- 导出 Excel(批量发货时使用)

#### CDK 兑换流程(用户端)

兑换是最容易出问题的环节,必须设计成幂等且原子的事务:

1. 用户提交 CDK 码,后端先做基本校验: CDK 是否存在、是否已使用、是否过期、用户是否被封禁

2. 开启数据库事务,加锁查询该 CDK: 使用 SELECT FOR UPDATE 防止并发兑换同一个 CDK

3. 根据 CDK 关联的套餐,决定动作:
   - 如果用户当前没有有效套餐: 分配新 IP(从 IP 池中找一个 available 的 IP),创建对应的 inbound,更新用户的套餐和到期时间
   - 如果用户已有有效套餐且类型相同: 续期,到期时间累加
   - 如果用户已有有效套餐但类型不同: 根据业务规则处理(建议禁止跨类型续费,要求先到期)

4. 调用 Xray gRPC 创建 inbound(如果是新开通)或更新(如果是续期不需要动)

5. 标记 CDK 为已使用,写入订单流水

6. 提交事务

**异常处理**:

- 如果 Xray gRPC 调用失败(网络异常、服务器宕机),数据库事务回滚,CDK 保持未使用状态,用户重试
- 如果数据库已提交但 Xray 失败(不太可能但要防),通过 Reconciler 后续会修复
- IP 池耗尽时,CDK 兑换失败给出明确提示("当前节点已满,请联系客服分配新节点")

### 订阅链接生成

用户在 Web 端点"复制订阅链接",后端返回一个唯一的订阅 URL,如 `https://your-domain.com/sub/{subscription_token}`。

订阅 token 是用户独立的随机字符串,存在用户表中,可以通过"重置订阅链接"功能更换(防泄露)。

订阅 URL 被客户端访问时,根据 User-Agent 判断客户端类型:

- v2rayN / Shadowrocket: 返回 base64 编码的 vless:// 链接列表,每行一个节点
- Clash Verge / Clash Meta: 返回 YAML 格式的配置
- sing-box: 返回 JSON 格式配置

如果分不清,默认返回 base64 vless 链接(兼容性最好)。

每个用户的订阅内容根据其当前绑定的 inbound 动态生成。如果用户买了双 IP 套餐,订阅里就有两个节点。

为了让用户感知"还有节点可选",可以在订阅里加上几个备用节点(虽然实际只有一个能连)。但建议第一阶段只返回真实可用的节点,避免用户混淆。

订阅接口要做防爬虫: 限制单个 token 每分钟最多 10 次请求,超过暂时返回 429。

### 流量统计采集

定时任务每 5 分钟跑一次:

1. 遍历所有活跃用户,按服务器分组
2. 对每台服务器,批量调用 StatsService 的 queryStats 接口,获取该服务器上所有 user 的 uplink/downlink 流量。Xray 的统计是累积值(自启动以来)
3. 与上次的累积值对比,差值就是这 5 分钟的流量。如果累积值小于上次(说明 Xray 重启过),把当前值作为新基准
4. 将差值写入流量日志表,更新用户的当月累计流量字段(用于配额检查)
5. 检查异常: 如果 5 分钟内某用户流量超过阈值(如 10GB),触发告警;如果月累计超过套餐上限(如果有上限),暂停服务
6. 将关键指标写入 Redis: 用户当日流量、当月流量、最近 1 小时速率,供 Web 端实时展示

### IP 健康检查

后台定时任务每 12 小时跑一次(或每 6 小时根据资源情况):

针对 IP 池中所有状态为 occupied 或 available 的 IP:

1. 通过该 IP 发起对几个目标的 HTTP 请求:
   - ChatGPT 的某个端点(如 `https://chat.openai.com/api/auth/csrf` 或类似的不需要登录的接口)
   - Claude 的健康检查端点
   - Gemini 的检查端点

2. 根据响应判断:
   - 200/正常响应 = 平台兼容
   - 403/Cloudflare 拦截 = 被风控
   - 超时/连接失败 = IP 或网络问题

3. 查询第三方服务的 IP 评分(可选):
   - Scamalytics 的 fraud score
   - IPQualityScore 的 risk score
   - AbuseIPDB 的滥用举报数

4. 更新 IP 池表的兼容性字段和健康日志表

5. 处理异常 IP:
   - 被多平台拦截的 IP 标记为 degraded,不再分配给新用户
   - 已分配给用户的 degraded IP,主动通知该用户"建议申请换 IP"
   - 评分极差的 IP 进入 cooling 状态,停用一段时间观察

健康检查实现方案: 给每台服务器装一个 healthcheck 微服务,由该服务器自己测试本机所有 IP 的可用性,定时上报结果到 Java 后端。这样避免 Java 后端跨网络发起测试的复杂性。

## 六、闲鱼运营流程

### 商品包装

闲鱼直接卖"代理"会被秒下架,必须包装:

- ✗ "美国独享 IP / VPN / 翻墙 / 机场 / 节点"
- ✓ "海外开发者节点 / Claude Cursor 加速 / GitHub 加速服务"
- ✓ "OpenAI ChatGPT 国外环境配置咨询"
- ✓ "海外开发环境技术支持 - 月付/季付/年付"

商品图用代码截图、Cursor 截图、Claude 对话截图,避开任何"代理"字样。

### 多账号策略

开 3-5 个闲鱼小号,不同 ID 卖不同套餐和价位测试,一个被封不影响整体。

不同账号之间不要互相点赞、收藏,避免被关联识别。

### 发卡流程

1. 买家拍下商品后通过闲鱼私聊联系
2. 运营人员核对订单信息(套餐类型、价格)
3. 登录管理后台,选择对应套餐,点击"生成 CDK",填入备注(闲鱼订单号 + 买家昵称)
4. 复制 CDK 通过闲鱼私聊发给买家
5. 同时发送使用说明: 注册地址、Clash 配置教程链接、常见问题
6. 买家在 Web 端注册账号 → 输入 CDK → 自动开通服务

### 售后处理

常见售后场景:

- **IP 被 OpenAI/Claude 拦截**: 后台为该用户分配新 IP,免费换号
- **客户端配置不会**: 引导查看使用文档,或远程协助
- **退款请求**: 7 天内未使用超过 50% 时长可退,已用 IP 进入 cooling 池

### 防黑产

闲鱼上有专门买代理转卖的黑产用户,识别特征:

- 拍下后催促快速发货
- 只买年卡(囤货转卖)
- 收货后立刻确认(避免售后)

应对:

- 同一闲鱼账号一个月内复购超过 3 次需人工审核
- 大单(年卡)分批发货
- 服务条款明确"禁止转售,违者封号不退款"

## 九、安全设计

### 认证与授权

- 所有 API(除注册、登录、订阅)都需要 JWT 认证。JWT 有效期 7 天,支持 refresh token 续期
- 管理后台用独立的 admin 账号系统,与用户账号完全隔离
- 敏感操作(兑换、修改密码、删除账户)记录审计日志

### 通信安全

- Web 端到 Java 后端: HTTPS(Let's Encrypt 证书)
- Java 后端到 Xray gRPC: SSH 隧道(前面已说明)
- Java 后端到 MySQL/Redis: 内网通信,不对公网暴露

### 数据保护

- 用户密码用 BCrypt 哈希存储,cost factor 12
- CDK 在数据库中存原文(因为要支持兑换时的明文校验,且运营人员需要查询发卡历史),但要求 CDK 足够长且随机(16+ 位字母数字混合,熵足够)
- 订阅 token 是高熵随机字符串,泄露后用户可一键重置

### 防 DDoS 与爬虫

- Java 后端前面挂 Cloudflare(免费版即可),开启基本的 DDoS 防护和速率限制
- 订阅接口、兑换接口设单独的速率限制规则
- 登录接口连续失败 5 次锁定 15 分钟

### 用户行为管控

虽然不管用户用 IP 干什么,但要防止极端行为牵连服务器:

- Xray 配置中加 sniffing,识别 BT/PT 流量,routing 规则中将这类流量 block
- 监控异常流量模式: 单用户短时间内连接数异常(>1000/分钟,疑似扫描)、流量异常(>100GB/天,疑似挂代理给别人用)、目标地址异常(大量连接到 25 端口,疑似垃圾邮件)
- 触发异常时不立即封号(避免误伤),先发邮件警告,重复触发则暂停服务

## 十、运维与监控

### 部署架构

- **主控服务器**: Java 后端、MySQL、Redis、Nginx 部署在同一台(4C8G 即可),用 Docker Compose 编排
- **出口节点**: Xray 单独部署在多台 VPS,每台跑独立的 Xray 实例

### 日志体系

- Java 后端日志用 Logback,按日轮转,保留 30 天
- Xray 日志开启 access 和 error 级别,但不记录用户 IP(隐私保护),按日轮转,保留 7 天
- 关键操作(兑换、换 IP、登录)单独记审计日志,永久保留
- 所有日志可以通过 Loki + Grafana 集中收集(可选,用户量大了再上)

### 监控告警

- **服务可用性**: 每分钟从外部访问 Web 接口和订阅接口,失败超过 3 次告警
- **服务器健康**: 每台 VPS 跑 node_exporter,主控用 Prometheus 抓取,CPU > 80%、内存 > 90%、磁盘 > 85%、带宽 > 80% 告警
- **业务指标**: 每小时统计活跃用户数、新增用户数、CDK 兑换数、IP 健康率,写入时序数据库(InfluxDB 或 Prometheus)
- **异常告警**: IP 大批量被封(如 1 小时内 3+ 个 IP 健康检查失败)、用户大批量异常(10+ 用户同时反馈无法连接)、CDK 库存告急(后期上自动发卡时)
- **告警渠道**: Telegram Bot 推送(最及时) + 邮件备份

### 备份策略

- MySQL 每天凌晨全量备份,保留 14 天。增量备份用 binlog,保留 7 天
- Redis 不需要备份(缓存数据可重建)
- Xray 配置不需要备份(从数据库 reconcile 重建)
- 备份文件加密后上传到对象存储(如 Cloudflare R2 免费额度够用)

## 十三、关键风险与应对

### 技术风险

- **Xray 进程崩溃**: systemd 自动拉起 + Java 后端 reconcile 自动恢复 inbound 配置
- **服务器宕机**: Java 后端检测到不可达后,将该服务器上的用户标记为"待迁移",运维介入处理。建议永远保持 1-2 台服务器作为备用容量
- **数据库故障**: 主从复制 + 定期备份。第一阶段单实例可以接受,但要做好备份

### 业务风险

- **IP 被大批量封禁**: 分散使用多家 IDC,不要把鸡蛋放一个篮子。每月监控各 IDC 的 IP 健康率,及时切换
- **闲鱼账号封禁**: 多个小号轮换,不要把所有流量押在一个账号上
- **黑产囤货转售**: 通过限购、分批发货、服务条款约束

### 法律合规

国内法律红线: 服务器、域名、备案、实名信息都用境外资源,主体不在国内。

**这个项目本质上是灰色业务,要做好心理准备和应急预案**。建议从一开始就把所有运营痕迹与个人身份隔离: 境外注册的支付账号、独立的运营手机号、专用的设备和网络环境。
