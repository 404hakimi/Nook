## 一、产品形态

Nook 卖**独享 IP 代理服务**, 用户按"区域 + IP 数量 + 月流量"购买套餐.

- **客户**: 跨境电商 / LLM 开发者 / 远程办公; 对"住宅 IP / ISP IP / 干净 IP"敏感
- **商品**: 套餐 = (区域, IP 数, 设备数, 周期, 流量档位); 售价随档位变
- **价位带**: 闲鱼小作坊级 SaaS, 99-299 元/月切入
- **客户感知**: 一份订阅 URL 包含多个节点, 在 v2rayN / Clash 等客户端导入后看到多节点列表

---

## 二、技术总览

```
[客户 v2rayN/Clash]
       │ vmess+ws (用户独占 UUID + 套餐时长 + 套餐流量)
       ▼
[中转线路服务器 (按区域 N 台)] ── 共享 inbound :443 + per-user routing rule
       │
       │ N 条 socks5 outbound, 1 outbound = 1 IP
       ▼
[落地 IP 节点 (按区域 N 个 SOCKS5)]
       │
       ▼
   Internet
```

业务流程:
1. 用户买套餐 → 后端 **资源分配器** 按区域从 IP 池挑 N 个 IP → 批量 provision N 个 xray_client → 拼成一份**聚合订阅 URL** 回给用户
2. 用户在客户端导入订阅 → 看到 N 个节点 → 任选
3. 流量按"总池"语义计费, 跑哪个节点都从同一池子扣

## 四、核心算法: 资源分配器

### 4.1 触发场景

| 场景 | 行为 |
|---|---|
| 新购订阅 | 调用 `allocate(memberId, planId, planTierId)`; 分配 N 个 IP, 创建 N 个 xray_client |
| 续费 | **不重新分配**; 只延长 `expires_at` + 调 `xray api adu` 重置 totalBytes |
| 流量包加购 | **不分配**; 把购买流量平均加到 N 个 client.totalBytes, 调 `xray api adu` 立即生效 |
| 节点故障自动切换 | 调用 `reallocate(subscriptionId, failedClientId)`; 单独换一个 IP |

### 4.2 硬约束矩阵 (候选过滤)

每个候选 `(server, ip)` 对必须全部通过:

| 维度 | 条件 |
|---|---|
| IP 状态 | `status = 1` (available) |
| IP 区域 | `region = plan.region` |
| IP 月流量余量 | `monthly_used_gb / traffic_quota_gb < 0.85` |
| IP 过期保护 | `expires_at >= now + plan.period_days` |
| Server 客户数 | `current_clients < max_concurrent_clients` |
| Server 月流量余量 | `server_used_gb / monthly_traffic_gb < 0.85` |
| Server 过期保护 | `expires_at >= now + plan.period_days` |

### 4.3 评分函数 (top-N 选择)

```
score(server, ip) = 
    0.30 × (1 - server_客户数水位)        ← 客户均衡, 防热点
  + 0.25 × (1 - server_流量水位)          ← 流量均衡
  + 0.20 × (1 - ip_流量水位)              ← 单 IP 不烧爆
  + 0.15 × ip_类型优先级                  ← residential / ISP > datacenter
  + 0.10 × (1 - 最近 24h 分配次数惩罚)    ← 反羊群效应
```

每项归一化 [0, 1], 总分越高优先级越高.

### 4.4 多 IP 分布策略

`plan.ip_count > 1` 时:
- 取评分 top N 自然排序结果
- **不强制跨 server**: 单 server 多 IP 也 OK (xray inbound 复用率高, ops 简化)
- **不强制单 server**: 跨 server 也 OK (高可用)
- 若 top N 都在同一 server, 由评分中的"客户数水位"自然产生分散偏好

### 4.5 资源不足处理

候选 < N → **硬拒**:
- 抛 `BusinessException(INSUFFICIENT_RESOURCES, region, available, required)`
- 自动写入 `alert` 表 (type=IP_POOL_LOW, severity=WARN)
- 触发邮件通知 admin

不做软降级 (不自动改换其他区域分配).

### 4.6 算法伪代码

```java
Subscription allocate(memberId, planId, planTierId) {
  Plan plan = planMapper.selectById(planId);
  PlanTier tier = planTierMapper.selectById(planTierId);
  
  // 1. 拉候选 (含硬约束过滤)
  List<Candidate> candidates = candidateMapper.findEligible(
      plan.region, plan.periodDays);
  
  if (candidates.size() < plan.ipCount) {
    alertService.raise(IP_POOL_LOW, plan.region, candidates.size(), plan.ipCount);
    throw new BusinessException(INSUFFICIENT_RESOURCES, plan.region);
  }
  
  // 2. 评分 + 排序
  candidates.forEach(c -> c.score = scoreOf(c));
  candidates.sort(byScoreDesc());
  
  // 3. 取 top N
  List<Candidate> picked = candidates.stream()
      .limit(plan.ipCount)
      .toList();
  
  // 4. 事务: 占 IP + 建订阅 + provision N client
  return transactionTemplate.execute(status -> {
    Subscription sub = createSubscription(memberId, plan, tier);
    long totalBytes = tier.trafficGb * 1024L * 1024L * 1024L;
    long perClientBytes = totalBytes / plan.ipCount;  // 平均分
    long expiryMillis = sub.expiresAt.toEpochMilli();
    
    for (Candidate c : picked) {
      ipPoolService.occupyById(c.ipId, memberId);
      XrayClient client = clientService.provision(
          c.serverId, c.ipId, memberId,
          perClientBytes, expiryMillis, plan.deviceLimit);
      subscriptionClientMapper.link(sub.id, client.id);
    }
    return sub;
  });
}
```

## 五、流量统计

### 5.1 数据源 (单一: 客户端层级)

维持现状: `XrayClientTrafficSampleServiceImpl` 每 5min 调 `xray api statsquery` 拉每个 user 的累计流量, 写 `xray_client_traffic`.

**不**采集服务器网卡 (`vnstat` / `/proc/net/dev`), **不**采集 IP 池机器网卡. 服务器 / IP 层级流量全部由 SQL 聚合派生.

### 5.2 派生统计

```sql
-- 某 server 本月已用
SELECT SUM(up_bytes + down_bytes) / 1024 / 1024 / 1024 AS gb
FROM xray_client_traffic 
WHERE server_id = ? AND sampled_at >= DATE_FORMAT(CURDATE(), '%Y-%m-01');

-- 某 IP 本月已用
SELECT SUM(up_bytes + down_bytes) / 1024 / 1024 / 1024 AS gb
FROM xray_client_traffic xct JOIN xray_client xc ON xc.id = xct.xray_client_id
WHERE xc.ip_id = ? AND xct.sampled_at >= DATE_FORMAT(CURDATE(), '%Y-%m-01');

-- 某 subscription 本月已用 (用于"总池"配额 enforce)
SELECT SUM(up_bytes + down_bytes) / 1024 / 1024 / 1024 AS gb
FROM xray_client_traffic xct JOIN subscription_client sc ON sc.xray_client_id = xct.xray_client_id
WHERE sc.subscription_id = ? AND xct.sampled_at >= sub.last_reset_at;
```

### 5.3 "总池"语义实现

套餐 200GB / 2 IP → 每个 xray_client.totalBytes = 100GB (平均分).

**问题**: 单 IP 撞 100GB 时 xray 拒连, 但用户总池可能还没用完.

**解法**: 调度 (每 5min) 扫所有活订阅:
- 算 `subscription.current_used_gb = SUM(关联 client 流量)`
- 若 < `monthly_traffic_gb × 1.0` → 不动
- 若 ≥ `monthly_traffic_gb × 1.0` → 触发"用尽": 暂停所有 client (`xray api rmu`) + 通知用户

**单 IP 撞顶的可恢复性**: xray 拒连后, 调度自动平衡? **不做**——简化模型, 接受单 IP 100GB 拒连; 用户可在 client 切到没用完的 IP. 实际撞顶概率随时间趋近平均使用模式而很低.

## 六、订阅生命周期

### 6.1 状态机

```
[新购] → PENDING_PAYMENT
       │   (订单付款)
       ▼
       ACTIVE  ←─────────────────┐
       │                          │  (续费)
       │   (expires_at 到)       │
       ▼                          │
       GRACE  (3 天宽限期)        │
       │                          │
       │   (3 天内续费 ✓)         │
       │   (3 天内未续 ×)         │
       ▼                          │
       EXPIRED  ─→  CANCELLED (手动吊销 / 系统 14 天后清理)
                    ↓
                    (revoke 所有关联 client + IP 进冷却池)
```

### 6.2 周期计算 (滚动周期, 每用户独立)

- **新购**: `started_at = now`, `expires_at = now + period_days`
- **月中购买**: 15 号买 30 天套餐 → 16 号开始算, 下月 14 号过期 (按购买日 + period_days, 不按自然月对齐)
- **续费**: `expires_at = max(expires_at, now) + period_days`
  - 在 ACTIVE 期续: 累加到现有到期日 (不亏天数)
  - 在 GRACE 期续: 同上, 从现有 expires_at 累加
  - EXPIRED 后续: 视为新购, started_at 重置
- **流量包加购**: 不影响 `expires_at`; 只增加 `monthly_traffic_gb` (累加到当前周期)

### 6.3 调度任务

| Cron | 任务 |
|---|---|
| 每日 00:30 | 扫描 `expires_at < now AND status='ACTIVE'`: 标 GRACE; 发邮件通知用户 |
| 每日 01:00 | 扫描 `status='GRACE' AND expires_at + 3 days < now`: 标 EXPIRED + 暂停 client |
| 每日 02:00 | 扫描 `status='EXPIRED' AND expires_at + 14 days < now`: 标 CANCELLED + revoke client + 回收 IP |
| 每 5min | 扫描活订阅: 算 `current_used_gb`, 超总池 → 暂停; 临近 80% → 邮件提醒用户加购 |
| 每日 09:00 | 扫描 `subscription.expires_at - 7 days ≤ now`: 发"7 天后到期"邮件 |
| 每日 09:00 | 扫描 `resource_server.expires_at - 7 days ≤ now` 和 `resource_ip_pool.expires_at - 7 days ≤ now`: 写 `alert(RESOURCE_EXPIRING)` |

## 七、订阅 URL (聚合形式)

### 7.1 格式

单个订阅 URL 返回所有关联节点 (v2ray 标准 subscription format):

```
GET /api/sub/{token}

Response: Content-Type: text/plain
Body: Base64(节点 1 URL\n节点 2 URL\n节点 3 URL\n...)
```

每个节点 URL 是标准 `vmess://...` (含该 client 的 uuid / IP host / port / ws path / tls 等).

### 7.2 Token 设计

- `subscription` 表加 `sub_token` (32 char random hex, 防猜测)
- token 跟 subscription 是 1:1, 长期有效 (除非用户主动重置)
- 用户拿到的 URL 形如: `https://nook.example.com/api/sub/abc123def456...`

### 7.3 客户端体验

- v2rayN / Clash / Sing-box 等支持订阅: 用户填 URL → 自动 fetch + parse → 节点列表
- 用户切换不同 IP 时, 在客户端节点列表里点选, 不重新下载订阅
- 节点变更 (如自动切换换 IP) → 用户下次刷新订阅时看到新节点

## 八、告警系统

### 8.1 告警类型

| type | severity | 触发 | 处理建议 |
|---|---|---|---|
| `IP_POOL_LOW` | WARN | 某区域可用 IP < 套餐需求 | admin 补 IP |
| `SERVER_OVERLOAD` | WARN | server 客户数 / 流量水位 > 90% | 减少分配 / 升配 |
| `IP_TRAFFIC_HIGH` | WARN | IP 月流量超 90% | 准备替换或加额度 |
| `SERVER_TRAFFIC_HIGH` | WARN | server 月流量超 90% | 同上 |
| `RESOURCE_EXPIRING` | INFO | server / IP 到期 ≤ 7 天 | admin 续费 |
| `NODE_FAILOVER` | ERROR | 自动切换发生 | admin 排查故障节点 |

### 8.2 渠道

- **后台**: admin 首页"待处理告警"卡片, 列出 `WHERE resolved_at IS NULL ORDER BY severity DESC`
- **邮件**: 重要告警 (severity ≥ WARN) 触发时发邮件到 admin (邮件配置在 application.yml)

### 8.3 处理

- admin 点告警卡片 → 看详情 → 处理 (补 IP / 续费等) → 点"已处理" → 写 `resolved_at = now`
- 重复告警: 同 type + target 24h 内已存在未处理告警 → 不重复创建 (避免轰炸)

## 九、节点故障自动切换

### 9.1 故障检测

- 复用现有 reconciler (`CLIENT_RECONCILE` op): 每 10min 探 xray uptime + SSH 可达
- 探测失败累计 N 次 (默认 3 次, 30min) → 判定 server "故障"
- 触发 `nodeFailover(serverId)` 处理逻辑

### 9.2 切换流程

```java
nodeFailover(serverId) {
  // 1. 拉该 server 上所有活客户
  List<XrayClient> affected = clientMapper.findActiveByServer(serverId);
  
  for (XrayClient c : affected) {
    // 2. 找该 client 所属 subscription
    Subscription sub = subscriptionService.findByClientId(c.id);
    if (sub == null) continue;  // 孤儿不动
    
    // 3. 在该 subscription 的 region 池找一个新 IP (硬约束 + 评分)
    Candidate newIp = allocatorService.pickAlternative(
        sub.region, excludeServers=[serverId]);
    
    if (newIp == null) {
      // 该 region 没有备用 → 写告警, 等 admin 处理
      alertService.raise(NODE_FAILOVER, severity=ERROR, c.id, "无备用 IP");
      continue;
    }
    
    // 4. 在新 IP 上 provision 一个新 client (复用 subscription 的 uuid / email 不能变)
    //    实际是: 在新 server 上 add user (相同 uuid + email) + rule + outbound
    XrayClient newClient = clientService.provisionForFailover(
        newIp.serverId, newIp.ipId, c.memberId, c.clientUuid, c.clientEmail, ...);
    
    // 5. DB 切换: subscription_client 改指向新 client; 老 client 标 status=2 (revoked)
    transactionTemplate.execute(s -> {
      subscriptionClientMapper.updateClient(sub.id, c.id, newClient.id);
      clientMapper.markRevoked(c.id);
      ipPoolService.releaseToCooling(c.ipId);
      return null;
    });
    
    alertService.raise(NODE_FAILOVER, severity=ERROR,
        targetId=c.id, "切换至 " + newClient.id);
  }
}
```

### 9.3 用户感知

- xray_client.clientUuid 不变 → 用户老订阅 URL 里的 vmess 凭据继续可用
- 但**节点 host 变了** (从老 IP 换到新 IP) → 必须下次订阅刷新才能拿到新 host
- 这一点提示用户在客户端打开"自动刷新订阅" (大多数客户端默认开 24h 刷一次)

### 9.4 限制

- 跨 region 切换不做 (用户买 JP 不会自动换到 US, 改变商品本质)
- 区域内无备用 IP → 标告警等 admin 补货, 不强行降级

**目标**: 用户能注册登录 / 看订阅 / 拿订阅 URL / 看流量用量.

## 十四、风险

| 风险 | 描述 | 缓解 |
|---|---|---|
| 总池流量 enforce 滞后 | 5min 调度间隔, 用户最多多用 5 分钟 | 业务可接受; 5min 调度足够 |
| 单 IP 撞顶但总池有余 | 用户主动切节点解决 | UI 提示"流量已分配到 N 个节点, 单节点撞顶时切换其他" |
| IP 池区域不均 | 某热门区域售罄, 冷门区域闲置 | 告警 IP_POOL_LOW + 运营策略调整定价 / 上下架套餐 |
| 节点故障 + 区域无备用 | 用户被动断线, 自动切换失败 | 告警 NODE_FAILOVER + ERROR severity; admin 1 小时内响应 SLA |
| 订阅 URL token 泄露 | 第三方拿到 token 可下载凭据 | token 32 char 随机; 用户可在 portal "重置订阅 URL" |
| 退款纠纷 | 不退款政策可能被吐槽 | 服务条款明示 + 工单可走部分场景 (服务故障等) |
