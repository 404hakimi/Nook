# Nook 后端开发规范

> 适用范围：`com.nook` 下所有 Java 模块。
> 本文档只列**约定**，不重复 [模块设计.md](模块设计.md) 中的目录结构。
> 凡是与本规范冲突的代码，提交前必须修正。

---

## 一、命名

### 1.1 数据库

| 对象 | 规则 | 示例 |
|------|------|------|
| 库名 | 全小写 + 下划线 | `nook` |
| 表名 | `<模块前缀>_<业务名>`，全小写下划线，**前缀使用完整单词不缩写** | `system_user`、`member_subscription` |
| 模块前缀 | 见 [模块设计.md](模块设计.md) §七 | `system` `member` `business` `resource` `xray` `monitor` |
| 字段名 | 全小写 + 下划线 | `created_at`、`member_user_id` |
| 外键列 | `<引用表>_id`；若引用表带模块前缀，保留完整表名以避免歧义 | `plan_id`、`system_user_id`、`member_user_id` |
| 索引名 | `idx_<列1>[_列2]`、`uk_<列>`、`pk_<列>`(很少用) | `idx_member_status`、`uk_username` |
| 主键 | 一律 `id`，类型 `CHAR(32)` | — |
| 布尔语义 | 用 `TINYINT`，不允许 `BOOLEAN`/`BIT`；状态值在 COMMENT 中列出 | `status TINYINT COMMENT '1=正常 2=禁用'` |
| 金额 | `DECIMAL(M,2)`，不用 FLOAT/DOUBLE | `price DECIMAL(10,2)` |
| 时间 | `DATETIME`(秒级)，跨时区场景才用 `TIMESTAMP` | `expire_at DATETIME` |
| JSON | `JSON`，复杂结构才用，**不滥用**(无法走索引) | `detail JSON` |

### 1.2 Java 包结构

```
com.nook
├── common.*               # nook-common 模块
├── framework.*            # nook-framework 模块
├── biz.<sub>.*            # nook-biz-<sub> 各业务模块
│      ├── api             # 对外契约：跨模块调用的接口 + DTO（其他模块只允许引用此包）
│      │   └── dto
│      ├── controller
│      │   ├── admin       # 后台 /admin/<sub>/**
│      │   └── portal      # 前台 /portal/<sub>/**
│      ├── service
│      │   └── impl
│      ├── mapper
│      ├── entity
│      ├── dto             # 模块内部 DTO（不对外）
│      ├── vo              # Controller 出参
│      ├── constant
│      └── exception
└── (启动类放 nook-server 模块的 com.nook 根包)
```

#### 跨模块引用约束

- 其他 biz 模块**只能 import `com.nook.biz.<sub>.api.*`**，不允许碰对方的 controller / service / mapper / entity / dto / vo。
- `api` 包暴露 `XxxApi` 接口与必要的 DTO，**禁止**直接返回 entity。
- `XxxApi` 的实现放在本模块 `service/impl/XxxApiImpl.java`，加 `@Service`。
- 这是约定式边界，编译器不强制；review 时把关。

#### `package-info.java` 约定

- 仅作占位用途（保证空包被 git 跟踪、保证包路径在 IDE 中可见）。
- **不允许**写任何 Javadoc / 注释 / 注解。
- 仅一行：`package xxx.yyy.zzz;`，文件以单个换行结尾。
- 包级文档/约束写在 [模块设计.md](模块设计.md) 或 [后端开发规范.md](后端开发规范.md) 中，不散落到代码里。

| 类型 | 后缀 | 例 |
|------|------|------|
| 对外开放接口 | `Api` | `MemberApi`、`PlanApi` |
| Controller | `Controller` | `MemberAdminController`、`MemberPortalController` |
| Service 接口 | `Service` | `MemberService`（模块私有） |
| Service 实现 | `ServiceImpl` | `MemberServiceImpl` |
| Api 实现 | `ApiImpl` | `MemberApiImpl` |
| Mapper | `Mapper` | `MemberUserMapper` |
| 实体 | 表名驼峰 | `MemberUser`、`BusinessCdk`、`MonitorTrafficSnapshot` |
| 入参 DTO | `<动作>Request` 或 `<对象>DTO` | `LoginRequest`、`CreateCdkDTO` |
| 出参 VO | `<对象>VO` | `MemberVO`、`SubscriptionVO` |
| 枚举 | `<业务>Enum` | `CdkStatusEnum` |
| 常量 | `<业务>Constants` | `RedisKeyConstants` |
| 异常 | `<业务>Exception` | `CdkRedeemException` |

> **实体类名直接对应表名驼峰**。`member_user` → `MemberUser`，`xray_inbound` → `XrayInbound`。
> 不去掉模块前缀，避免不同模块出现重名实体。

### 1.3 字段命名（构造注入）

Spring 注入的字段（`@RequiredArgsConstructor` + `private final`）**用类型名小驼峰**，**不要起短别名**。原因：

1. 一眼看出注入的是哪个 Bean，跨文件搜索 `systemUserMapper` 即可定位所有引用
2. 避免同类型多次注入时命名冲突
3. 与 Spring 默认 Bean Name 一致，文档/日志报错可对照

| 类型 | ✅ 字段名 | ❌ 反例 |
|------|---------|--------|
| `SystemUserMapper` | `systemUserMapper` | `mapper` |
| `SystemUserService` | `systemUserService` | `userService` / `service` |
| `SystemAuthService` | `systemAuthService` | `authService` |
| `BCryptPasswordEncoder` | `bCryptPasswordEncoder` | `passwordEncoder` / `encoder` |
| `RedissonClient` | `redissonClient` | `redisson` / `client` |
| `RedisTemplate<String, Object>` | `redisTemplate` | `redis` / `template` |

**例外**：方法内的局部变量、方法参数允许使用语义化短名（如 `User user = mapper.selectById(id)`），不受此规则约束。仅 `@Component` / `@Service` / `@Repository` 等 **Spring 注入字段** 强制全名。

### 1.4 Redis Key

`<业务域>:<对象>:<id>[:<子项>]`，全小写、冒号分隔，**不允许**驼峰或下划线分段。

| 用途 | Key 模板 |
|------|---------|
| 会员令牌 | `token:member:{uuid}` |
| 后台令牌 | `token:system:{uuid}` |
| 会员令牌反查 | `member:tokens:{memberId}` (Set) |
| 会员设备 | `member:devices:{memberId}` (Hash) |
| 会员日流量缓存 | `traffic:member:{memberId}:today` |
| IP 实时流量 | `traffic:ip:{ipId}:current` |
| CDK 兑换锁 | `lock:cdk:{codeHash}` |
| 兑换频率 | `rate:redeem:{memberId}` |

所有 Key 必须在 `RedisKeyConstants` 中定义，禁止在业务代码里散落字符串拼接。

## 二、数据库字段约定

### 2.1 通用列

每张业务表至少包含：

```sql
`id`         CHAR(32)  NOT NULL,
`created_at` DATETIME  NOT NULL COMMENT '创建时间',
`updated_at` DATETIME  NOT NULL COMMENT '更新时间',
PRIMARY KEY (`id`)
```

- **不写** `DEFAULT CURRENT_TIMESTAMP` 和 `ON UPDATE CURRENT_TIMESTAMP`。所有时间由 MyBatis-Plus `MetaObjectHandler` 在 INSERT/UPDATE 时统一填充。
- 直接 SQL 写入（种子数据、运维 SQL、数据迁移脚本）必须手动 `NOW()`。
- 业务时间字段（`expire_at`、`last_login_at`、`snapshot_time` 等）由业务代码显式赋值，**不属于自动填充范围**。

### 2.2 逻辑删除

- `deleted TINYINT NOT NULL DEFAULT 0 COMMENT '0=未删 1=已删'`
- **按需添加**，仅"主数据表"用：用户类、套餐、服务器、IP 池、IP 类型。
- 日志、快照、明细类一律不加，需要清理走分区或归档。
- 加了 `deleted` 的表，**唯一索引必须把 `deleted` 列纳入**，否则软删后不能复用同名：
  ```sql
  UNIQUE KEY `uk_username` (`username`, `deleted`)
  ```

### 2.3 字段 COMMENT

- **必写**，描述字段业务含义，不写实现细节
  - ✅ `COMMENT '创建时间'`
  - ✅ `COMMENT '1=正常 2=禁用'`
  - ❌ `COMMENT '由 MyBatis-Plus 填充'`（这是实现细节，写在本规范里就够了）
  - ❌ `COMMENT 'created at'`（不写英文重复字段名）
- 枚举类字段必须列出所有取值含义。
- 外键列在 COMMENT 中标明引用表，便于 navicat 等工具直读：`COMMENT '关联 business_plan.id'`。

### 2.4 索引

> ⚠️ **现阶段（需求 TBD）**：DDL 中只保留 `PRIMARY KEY (id)`，不写任何二级索引、唯一索引、外键约束、分区。等业务稳定、查询模式明确、数据量上来后，按真实 EXPLAIN 结果通过 ALTER TABLE 增加。

后续补索引时遵循以下约定：

- 普通索引前缀 `idx_`，唯一索引前缀 `uk_`。
- 高频查询字段（外键列、status、created_at 排序）必须建索引。
- 联合索引按 ESR 原则排列：等值 → 排序 → 范围。
- 单表索引数 ≤ 6（含主键、唯一），超出说明设计有问题。
- 加 `@TableLogic` 的表，唯一索引必须把 `deleted` 列纳入，避免软删后无法复用。

## 三、MyBatis-Plus 使用规范

### 3.1 BaseEntity

所有实体继承 `com.nook.framework.mybatis.entity.BaseEntity`（位于 `nook-spring-boot-starter-mybatis`），统一以下字段，**不要在子类重复声明**：

```java
@TableId(value = "id", type = IdType.ASSIGN_UUID)
private String id;

@TableField(value = "created_at", fill = FieldFill.INSERT)
private LocalDateTime createdAt;

@TableField(value = "updated_at", fill = FieldFill.INSERT_UPDATE)
private LocalDateTime updatedAt;
```

逻辑删除字段 `deleted` 不在基类里，需要的实体自己加：

```java
@TableLogic
private Integer deleted;
```

### 3.2 注解使用

| 场景 | 注解 |
|------|------|
| 实体绑定表 | `@TableName("xxx")`，**强制写表名**，不依赖 `table-prefix` 配置兜底 |
| 主键 | `@TableId(type = IdType.ASSIGN_UUID)` |
| 字段映射不一致 | `@TableField("col_name")` |
| 不入库的虚拟字段 | `@TableField(exist = false)` |
| 自动填充 | `@TableField(fill = FieldFill.INSERT)` 或 `INSERT_UPDATE` |
| 逻辑删除 | `@TableLogic` |
| 乐观锁 | `@Version` |

### 3.3 写法约定

#### Mapper 层职责

**所有数据访问代码（`Wrappers` / SQL）必须收敛到 Mapper 接口或 XML 中**，Service 不直接构造 Wrapper。

- **简单查询/更新**：在 Mapper 接口里写 `default` 方法封装 LambdaWrapper：
  ```java
  @Mapper
  public interface SystemUserMapper extends BaseMapper<SystemUser> {

      default SystemUser selectByUsername(String username) {
          return selectOne(Wrappers.<SystemUser>lambdaQuery()
                  .eq(SystemUser::getUsername, username));
      }

      default boolean existsByEmailExcludingId(String email, String excludeId) {
          return exists(Wrappers.<SystemUser>lambdaQuery()
                  .eq(SystemUser::getEmail, email)
                  .ne(SystemUser::getId, excludeId));
      }

      default int updatePasswordHash(String id, String passwordHash) {
          return update(null, Wrappers.<SystemUser>lambdaUpdate()
                  .set(SystemUser::getPasswordHash, passwordHash)
                  .eq(SystemUser::getId, id));
      }
  }
  ```
- **复杂查询**（多表 JOIN / 动态拼接 30+ 行 / 报表统计）：走 `xml`，文件放 `nook-biz-<sub>/src/main/resources/mapper/<Entity>Mapper.xml`。
- 一律用 `LambdaQueryWrapper` / `LambdaUpdateWrapper` 引用字段，**禁止 String 拼字段名**：
  - ✅ `wrapper.eq(SystemUser::getStatus, 1)`
  - ❌ `wrapper.eq("status", 1)`
- 分页一律用 MyBatis-Plus `Page<T>`，禁止手工 `LIMIT`。

#### Service 层职责

Service 编排业务流程：参数校验、状态机、事务、调用其它 Service / Mapper。

- **禁止**直接出现 `Wrappers.lambdaQuery()` / `Wrappers.lambdaUpdate()`；如发现，立即抽到对应 Mapper 的 default 方法。
- 通过 Mapper 的业务化方法访问数据，例如 `mapper.selectByUsername(name)`、`mapper.existsByEmail(email)`、`mapper.updatePasswordHash(id, hash)`。
- `selectById` / `updateById` / `deleteById` 等 MyBatis-Plus 自带方法可以在 Service 直接调用（这些是按主键操作，不需要包 Wrapper）。
- 大批量更新走 `mapper.update(...)` 的封装方法或 `updateBatchById`，禁止 for 循环单条更新。

#### 反例 vs 正例

❌ **wrong** — Service 自己拼 Wrapper：

```java
@Service
public class SystemUserServiceImpl {
    public SystemUser findByUsername(String username) {
        return mapper.selectOne(Wrappers.<SystemUser>lambdaQuery()
                .eq(SystemUser::getUsername, username));   // Wrapper 不该出现在这里
    }
}
```

✅ **correct** — Service 调 Mapper 业务方法：

```java
@Service
public class SystemUserServiceImpl {
    public SystemUser findByUsername(String username) {
        return mapper.selectByUsername(username);
    }
}
```

## 四、统一返回与异常

实现位置：

| 包 | 类 |
|---|---|
| `com.nook.common.web.response` | `Result` |
| `com.nook.common.web.error` | `ErrorCode`(接口) / `CommonErrorCode`(枚举) |
| `com.nook.common.web.exception` | `BusinessException` |
| `com.nook.framework.web.exception` | `GlobalExceptionHandler` (位于 starter-web) |

### 4.1 返回结构

所有 Controller 返回 `com.nook.common.web.response.Result<T>`：

```java
{
  "code": 0,            // 0=成功，其它=错误码
  "message": "ok",
  "data": ...           // 成功时返回的数据；失败时通常为空
}
```

写法：

```java
return Result.ok();                                  // 成功无返回
return Result.ok(memberVO);                          // 成功带数据
return Result.fail(SystemErrorCode.USER_NOT_FOUND);  // 失败用错误码（推荐）
return Result.fail(1999, "兜底错误码");              // 不推荐，应优先用枚举
```

业务侧绝大多数场景**不显式构造 fail**，直接 `throw new BusinessException(errorCode)`，由全局处理器统一转换。

### 4.2 错误码

#### 设计

`com.nook.common.web.error.ErrorCode` 是接口：

```java
public interface ErrorCode {
    int getCode();
    String getMessage();
    default String format(Object... args) { ... }   // 支持 %s 占位
}
```

每个模块在自己的 `constant/<Sub>ErrorCode.java` 里定义枚举实现该接口。

#### 段位划分

| 段位 | 模块 |
|------|------|
| `0` | 成功 |
| `1xxx` | 通用错误（`com.nook.common.web.error.CommonErrorCode`，跨模块） |
| `2xxx` | system |
| `3xxx` | member |
| `4xxx` | business |
| `5xxx` | resource |
| `6xxx` | xray |
| `7xxx` | monitor（含统计 + 健康检查 + 告警） |

#### CommonErrorCode 已包含

`SUCCESS / INTERNAL_ERROR / PARAM_INVALID / UNAUTHORIZED / FORBIDDEN / NOT_FOUND / METHOD_NOT_ALLOWED / RATE_LIMITED / CONCURRENT_CONFLICT`，业务模块新建错误码前先确认是不是通用场景。

#### 模块错误码示例

```java
// nook-biz-system/src/main/java/com/nook/biz/system/constant/SystemErrorCode.java
@Getter
@RequiredArgsConstructor
public enum SystemErrorCode implements ErrorCode {
    USER_NOT_FOUND(2001, "用户 %s 不存在"),
    PASSWORD_INVALID(2002, "密码错误"),
    ACCOUNT_DISABLED(2003, "账号已禁用");

    private final int code;
    private final String message;
}
```

### 4.3 异常体系

`com.nook.common.web.exception.BusinessException` 是所有业务异常的基类，**默认抑制堆栈**（业务异常通常是预期错误，不需要打全堆栈）。

#### 用法

```java
// 错误码无参
throw new BusinessException(SystemErrorCode.PASSWORD_INVALID);

// 错误码 + 运行时参数（消息含 %s）
throw new BusinessException(SystemErrorCode.USER_NOT_FOUND, username);

// 包装下层异常
throw new BusinessException(CommonErrorCode.INTERNAL_ERROR, ioException);
```

#### 是否需要子异常

**优先用错误码区分场景**，不要轻易继承 BusinessException。仅当：
- 需要让上层 `catch` 走不同分支
- 需要携带额外结构化数据（不只是 code+message）

时才继承，例如：

```java
public class CdkRedeemException extends BusinessException {
    private final String cdkId;
    // ...
}
```

#### 全局处理

`com.nook.framework.web.exception.GlobalExceptionHandler` 已统一接管：

| 异常类型 | 处理 | 日志级别 |
|---------|------|---------|
| `BusinessException` | 按 code+message 原样返回 | WARN |
| `MethodArgumentNotValidException` / `BindException` | 拼接字段错误 → `PARAM_INVALID` | （不记） |
| `ConstraintViolationException` | 同上 | （不记） |
| `MissingServletRequestParameterException` | `PARAM_INVALID` 加参数名 | （不记） |
| `HttpMessageNotReadableException` | `PARAM_INVALID` "请求体格式错误" | （不记） |
| `HttpRequestMethodNotSupportedException` | `METHOD_NOT_ALLOWED` | （不记） |
| `NoHandlerFoundException` | `NOT_FOUND` | （不记） |
| 其它 `Throwable` | `INTERNAL_ERROR` | ERROR + 完整堆栈 |

#### 强制约束

- Controller 层**禁止**直接 `try-catch`（除非有明确补偿逻辑）。
- Service 层**不要**吞异常返回 null。要么处理，要么抛上去。
- 抛业务异常**必须用 ErrorCode 枚举**，禁止 `throw new BusinessException(9999, "硬编码")`。

## 五、Service 与事务

- 接口与实现分离，接口在 `service/`，实现在 `service/impl/`。
- `@Transactional` 默认 `rollbackFor = Exception.class`，避免只回滚 RuntimeException 的坑。
- 事务方法：
  - **不要** 在事务里调用外部 RPC / HTTP / Xray gRPC（除非业务允许失败重试）。
  - **不要** 在事务里循环调用其他事务方法，避免连接占用过长。
  - 事务方法之间的相互调用要么改写成同类方法、要么使用 `TransactionTemplate`。
- 兑换、扣款、状态机变更类操作必须显式 `@Transactional`。

## 六、Mapper / SQL 编写

- xml 文件路径：`nook-biz-<sub>/src/main/resources/mapper/<Entity>Mapper.xml`，文件名与 Mapper 接口同名。
- `<resultMap>` 显式声明列与字段映射，**不依赖** `mapUnderscoreToCamelCase` 兜底。
- `SELECT *` 仅在 `selectById` 等 MP 自带方法允许；自定义 SQL 必须列出所需字段。
- 跨表 JOIN ≤ 3 张；超过用应用层拼装。
- 所有 `WHERE` 条件必须命中索引，新增 SQL 必须 `EXPLAIN` 验证。

## 七、Redis / Redisson

### 7.1 客户端

统一用 **Redisson**（`redisson-spring-boot-starter`），不要再注入 `RedisTemplate` 或 `StringRedisTemplate` 写新代码。Redisson 的 API 类型化更强、分布式锁/限流/集合开箱即用。

注入方式：

```java
@Service
@RequiredArgsConstructor
public class CdkRedeemService {
    private final RedissonClient redisson;
    // ...
}
```

### 7.2 常用 API 速查

| 用途 | API |
|------|-----|
| 字符串 KV | `redisson.getBucket(key).set(value, Duration.ofSeconds(ttl))` / `.get()` |
| 哈希 | `redisson.getMap(key)` (`RMap<K,V>`) |
| 集合 | `redisson.getSet(key)` / `getList(key)` / `getSortedSet(key)` |
| 计数器 | `redisson.getAtomicLong(key).incrementAndGet()` |
| **分布式锁** | `RLock lock = redisson.getLock(key); lock.tryLock(wait, lease, TimeUnit.SECONDS);` |
| **限流** | `RRateLimiter limiter = redisson.getRateLimiter(key); limiter.trySetRate(...); limiter.tryAcquire();` |
| **延时队列** | `RDelayedQueue<T>` |

锁示例（CDK 兑换互斥）：

```java
RLock lock = redisson.getLock("lock:cdk:" + cdkHash);
boolean got = lock.tryLock(0, 30, TimeUnit.SECONDS);   // 不等待，最多持有 30s
if (!got) throw new BusinessException(CommonErrorCode.CONCURRENT_CONFLICT);
try {
    // 兑换核心逻辑
} finally {
    if (lock.isHeldByCurrentThread()) lock.unlock();
}
```

### 7.3 sa-token 与 Redisson

sa-token 通过 `sa-token-redisson-jackson` 直接复用项目的 `RedissonClient`，不要单独配置 sa-token 的 Redis 连接。token 的 key 前缀（`token:system:` / `token:member:`）由 [StpSystemUtil](../nook-framework/src/main/java/com/nook/framework/security/stp/StpSystemUtil.java) / [StpMemberUtil](../nook-framework/src/main/java/com/nook/framework/security/stp/StpMemberUtil.java) 决定。

### 7.4 Key 命名

仍按 §1.4 Redis Key 命名规范：`<业务域>:<对象>:<id>[:<子项>]`，全小写冒号分隔。所有 Key 模板必须在 `<sub>/constant/<Sub>RedisKey.java` 集中定义，禁止散落字符串拼接。

## 八、日志

- 框架：`SLF4J + Logback`，禁用 `System.out`、`e.printStackTrace()`。
- 日志级别：
  - `ERROR`：导致请求失败、需要人工介入的异常
  - `WARN`：业务降级、外部依赖异常但可继续
  - `INFO`：关键路径（登录、兑换、IP 分配、Xray 操作）
  - `DEBUG`：开发期排查，生产关闭
- 占位符：`log.info("xxx {} yyy {}", a, b)`，**禁止**字符串拼接。
- 异常：`log.error("CDK 兑换失败 cdkId={}", cdkId, ex)`，最后一个参数传 ex 才能打印堆栈。
- 业务日志（兑换、登录、Xray 操作）落 `*_log` 表，不只依赖文件日志。

## 九、注释

### 8.1 总原则

- **类注释和方法注释都是一句话简述**，说清"是什么 / 给谁用 / 什么时候用"，不重复签名、不放使用示例。
- **重要的"为什么"放行内注释**（违反直觉的设计决策、性能优化、临时绕过、坑点等），紧贴对应代码行。
- **使用示例、设计背景、对外契约这类长内容**写到 `docs/` 下的文档里，不进代码。

### 8.2 类注释

```java
/** 业务异常，由 GlobalExceptionHandler 统一转为 Result 返回。 */
public class BusinessException extends RuntimeException { ... }
```

- 一行 Javadoc，一句话讲清"是什么 / 谁来调用"。
- 禁止贴用法示例代码块（`<pre>...</pre>` / 多段 sample），那是文档的事。
- 实现类（`*ServiceImpl` / `*ApiImpl`）通常不需要类注释，接口上的注释已经说明了职责。

### 8.3 方法注释

#### 基础封装类（common / framework / api 接口）

一行 Javadoc，**不需要 `@param` / `@return` / `@throws`**：

```java
/** 抛指定错误码，并保留下层 cause 的堆栈。 */
public BusinessException(ErrorCode errorCode, Throwable cause) { ... }
```

签名已经表达了入参和返回，重复只增加噪音。

#### 业务方法（Service / Controller）

可以使用完整 Javadoc，含 `@param` / `@return` / `@throws`，但首行**仍是一句话简述**：

```java
/**
 * 校验 CDK 并兑换为指定会员的订阅。
 *
 * @param memberId 兑换发起人会员 ID
 * @param code     CDK 原文(明文)
 * @return 新生成的订阅 ID
 * @throws BusinessException CDK 不存在 / 已使用 / 已过期 / 会员状态异常
 */
public String redeem(String memberId, String code) { ... }
```

判断标准：**参数语义不能从命名推断、或方法有重要副作用/异常路径**时，才上完整 Javadoc。

### 8.4 行内注释

- 紧贴代码行（上方独立行 或 行尾 `//`）。
- 用来回答"**为什么**"，不解释"是什么"。代码本身能说明的，不要重复写在注释里。

```java
@Override
public synchronized Throwable fillInStackTrace() {
    // 业务异常承载的是预期错误(参数无效 / 状态冲突等)，无须付出生成堆栈的开销;
    // 真正未预期的系统异常走原生 RuntimeException 路径，堆栈仍正常生成。
    return this;
}
```

反例：

```java
// 设置用户名                                              // ❌ 重复签名
user.setUsername(name);

// 把 status 字段设为 1                                     // ❌ 翻译代码
status = 1;

// 这里有个 bug 待修，先这样                                 // ❌ 用 TODO 不用注释
return null;
```

正例（行内注释回答"为什么"）：

```java
// 用 SET NX 防止同一 CDK 被多并发线程同时兑换；锁 30s 兜底防死锁
String lockKey = "lock:cdk:" + cdkHash;
boolean locked = redisTemplate.opsForValue().setIfAbsent(lockKey, "1", Duration.ofSeconds(30));
```

### 8.5 TODO / FIXME

- `// TODO(@user, yyyy-mm-dd): 简述待办`，要带责任人和日期，禁止裸 TODO。
- `// FIXME(@user, yyyy-mm-dd): 简述问题`，用于已知缺陷需要后续修复。
- 提交前 grep 检查项目中 TODO / FIXME 数量，超过 30 个考虑专项清理。

### 8.6 `package-info.java`

- 仅占位用，**不允许**任何 Javadoc / 注释 / 注解（包级文档统一在 `docs/` 维护）。
- 文件单行：`package xxx.yyy.zzz;`

### 8.7 文件头版权 / 作者

不需要。`@author` 信息走 git blame，文件头版权信息没有意义。

## 十、提交与代码风格

- 缩进 4 空格，`UTF-8`，行尾 `LF`。
- Lombok：`@Data`、`@Slf4j`、`@RequiredArgsConstructor`(优先于 `@Autowired`)。
- 字段 `private final`，构造注入；**禁止** `@Autowired` 字段注入。
- 工具类一律 `public final` + `private` 构造器，所有方法 `static`。
- 类长度 ≤ 500 行，方法 ≤ 80 行，超出考虑拆分。

### 10.1 空 / 空白 / 集合判空

**统一用 Hutool 工具类**，不允许裸写 `x == null` / `s != null && !s.isEmpty()` 这类样板。

| 判断 | 用法 |
|------|------|
| 对象 是否为 null | `ObjectUtil.isNull(x)` / `ObjectUtil.isNotNull(x)` |
| 对象 是否相等 | `ObjectUtil.equal(a, b)`（null 安全） |
| 字符串 是否为空（含空白） | `StrUtil.isBlank(s)` / `StrUtil.isNotBlank(s)` |
| 字符串 是否为空（仅长度 0） | `StrUtil.isEmpty(s)` / `StrUtil.isNotEmpty(s)` |
| 字符串 是否相等 | `StrUtil.equals(a, b)`（null 安全） |
| 集合 是否为空 | `CollUtil.isEmpty(c)` / `CollUtil.isNotEmpty(c)` |
| 数组 是否为空 | `ArrayUtil.isEmpty(arr)` / `ArrayUtil.isNotEmpty(arr)` |
| Map 是否为空 | `MapUtil.isEmpty(m)` / `MapUtil.isNotEmpty(m)` |

**正反例**：

```java
// ❌ wrong
if (exist == null) throw new BusinessException(USER_NOT_FOUND);
if (dto.getEmail() != null && !dto.getEmail().isBlank()) { ... }
if (id.equals(currentLoginId)) { ... }   // id 为 null 会 NPE

// ✅ correct
if (ObjectUtil.isNull(exist)) throw new BusinessException(USER_NOT_FOUND);
if (StrUtil.isNotBlank(dto.getEmail())) { ... }
if (StrUtil.equals(id, currentLoginId)) { ... }   // null 安全
```

#### 防御性入口校验

参数入口的"必须非空"断言，仍然推荐 `Objects.requireNonNull(x, "msg")` 或 `Assert.notNull(x, ...)`，不要重复 `if (x == null) throw ...` 模板。这条与上面的"条件判断"互不冲突。

### Git Commit

格式：`<type>(<scope>): <subject>`

| type | 含义 |
|------|------|
| feat | 新功能 |
| fix | 修复缺陷 |
| refactor | 重构 |
| perf | 性能优化 |
| docs | 文档 |
| style | 格式（非业务） |
| test | 测试 |
| chore | 构建/依赖 |

scope 用模块名（system / member / biz / xray / ...）或全局（global）。

## 十一、安全

- **永远不**在日志里打印密码、token、CDK 原文、UUID 凭据。
- 入参校验用 `@Valid` + `jakarta.validation`，DTO 上加 `@NotBlank` `@Size` `@Pattern`。
- 跨模块传递 ID 用 `String`，**不要** 反序列化为 `long`，UUID 字符串混淆容易出 bug。
- 所有外部输入都要校验，禁止信任前端传来的 status / type / role 值。
- SQL 注入：用 `#{}` 参数绑定，禁用 `${}`（除非是动态表名/列名且已校验白名单）。

## 十二、文档

- 每个模块（`nook-biz-<sub>`）的根目录放 `README.md`，描述：模块职责、对外提供的 Service、依赖了哪些其他模块、关键配置项。
- 公共契约（错误码、Redis Key、事件名）变更必须同步本规范文档。
- 接口文档用 Knife4j 自动生成，Controller 上标注 `@Operation` `@Schema`。
