## 一、为什么要写这份文档

### 1.1 关键修正: 旧"方案 C" 的 gRPC 控制路径有误

[产品化与实现计划.md](产品化与实现计划.md) 第二节描述的"方案 C: gRPC"包含三连调用:

> nook 通过 gRPC (HandlerService.AlterInbound) 加 user
> nook 通过 gRPC (HandlerService.AddOutbound) 加 socks5 outbound
> **nook 通过 gRPC (RoutingService.AddRule) 按 email 分流**

**第三条是错的**: Xray 的 `RoutingService` gRPC API **只有** `SubscribeRoutingStats` 和 `TestRoute` 两个方法, **没有** `AddRule` / `RemoveRule` / `SetRules` 任何运行时改 rule 的能力。

详见 Xray 源码 `proto/app/router/command/command.proto`。社区 (sing-box / 各路 fork) 也证实此限制。

**直接后果**: 当前实现里"加客户必须重写 xray.json + restart"的根因, 就是 routing 不能动 → 只能改文件 → 文件改了必须重启才生效。每次操作 = 整台 server 上所有客户断连 1-2 秒。

### 1.2 真正能"0 客户感知操作"的路径: 1:1 + slot 预分配

绕过 RoutingService 限制的唯一可行方案:

- **routing rule 集合在 server 启动时一次性预置 50 条**, 之后**永不变更**
- 业务侧只通过 `HandlerService` 的 inbound / outbound API 改运行时
- 这要求路由不能再"按 email 分流" (因为 email 与客户绑定, 增删客户就要改 rule)
- 必须改成 **"按 inbound tag 静态映射到同名 outbound tag"**, 即每个客户独享一个 inbound

详见第三节阶段 1 方案。

### 1.3 长期愿景: 从"运营工具"演进为"成熟平台"

当前 nook 是**运营工具**: 运营手动选 server + IP 给客户开通, 节点宕机要人工切, 客户没有自助能力。

**成熟平台**应具备:
- 客户自助 (注册 / 下单 / 续费 / 重置密钥 / 换 IP, 不需联系运营)
- 资源自动调度 (按容量 / 健康度 / 地域评分挑节点)
- 故障自愈 (节点宕机自动迁移, IP 被封自动换源)
- 客户端无感迁移 (DNS 动态切换, 客户配置不用改)
- 可观测 (流量监控 / 健康告警 / 异常风控)

完整建成需要持续投入。本文档把演进拆成 4 个阶段, 每阶段独立产生价值, 任一阶段叫停不浪费已投入。

## 三、阶段 1 详细方案: Xray 1:1 + slot 改造

### 3.1 业务模型变化

#### 现状 (问题模型)

```
[客户 A] ──┐
[客户 B] ──┼──→ 1 个共享 vless inbound :443  ──→ routing 按 email 选 outbound ──→ socks5 落地
[客户 C] ──┤      (users[]: A, B, C, D)         (rules: A→out-A, B→out-B...)
[客户 D] ──┘
```

- 1 个共享 inbound, 多个 user
- N 个 outbound (随客户数变)
- N 条 routing rule (随客户数变)

每加一个客户 → routing 改 → **必须 restart xray** → 全断 1-2 秒。

#### 升级后 (1:1 + slot 模型)

```
[客户 A] ─→ inbound in_slot_01 :30001 ─→ 静态 rule (永不变) ─→ out_slot_01 (gRPC 灌入 socks5)
[客户 B] ─→ inbound in_slot_02 :30002 ─→ 静态 rule (永不变) ─→ out_slot_02 (gRPC 灌入 socks5)
[客户 C] ─→ inbound in_slot_03 :30003 ─→ 静态 rule (永不变) ─→ out_slot_03 (gRPC 灌入 socks5)
                                          ...
                                          in_slot_50 → out_slot_50 (50 条 rule 启动时定死)
```

- **N 个独立 inbound**, 每客户一个, 监听不同端口
- 每个 inbound **只有 1 个 user**
- **50 个 outbound 槽位** (启动时是占位 freedom, 业务用时 gRPC 替换为 socks5)
- **50 条 routing rule** (启动时静态预置, server 寿命内永不变)

每加一个客户 → 调 4 次 gRPC → **不重启 xray** → 其他客户 0 感知。

### 3.2 业务前提确认

| 前提 | 状态 | 说明 |
|-|-|-|
| 单 server 客户上限 | ≤20 (设计), slot 池 50 (含冗余) | 已确认 |
| 客户能容忍非 :443 端口 | ✅ 能 | 客户从家用宽带 / 4G / 海外环境为主, 校园 NAT 强限制场景较少 |
| 抗审查需求 | 一般 | 海外客群为主, 不需要 REALITY 单端口极致伪装 |

### 3.3 xray.json 预置结构

server 第一次部署时由 install 脚本生成, **server 寿命内不再被 nook 修改**。

```json
{
  "log": { "level": "warning", "loglevel": "warning" },
  "api": {
    "tag": "api",
    "services": ["HandlerService", "StatsService"]
  },
  "stats": {},
  "policy": {
    "levels": { "0": { "statsUserUplink": true, "statsUserDownlink": true } },
    "system": { "statsInboundUplink": true, "statsInboundDownlink": true }
  },
  "inbounds": [
    {
      "tag": "api",
      "listen": "127.0.0.1",
      "port": 8080,
      "protocol": "dokodemo-door",
      "settings": { "address": "127.0.0.1" }
    }
    // ⚠️ 没有客户 inbound, 全部由 nook 通过 gRPC 动态加
  ],
  "outbounds": [
    { "tag": "api", "protocol": "freedom" },
    { "tag": "direct", "protocol": "freedom" },
    // ⭐ 50 个 placeholder freedom outbound, 占住 tag 让 routing rule 启动时合法
    { "tag": "out_slot_01", "protocol": "freedom" },
    { "tag": "out_slot_02", "protocol": "freedom" },
    // ... out_slot_03 ~ out_slot_50
    { "tag": "out_slot_50", "protocol": "freedom" }
  ],
  "routing": {
    "rules": [
      { "type": "field", "inboundTag": ["api"], "outboundTag": "api" },
      // ⭐ 50 条预置 rule, server 寿命内永不变
      { "type": "field", "inboundTag": ["in_slot_01"], "outboundTag": "out_slot_01" },
      { "type": "field", "inboundTag": ["in_slot_02"], "outboundTag": "out_slot_02" },
      // ... in_slot_03 ~ in_slot_50
      { "type": "field", "inboundTag": ["in_slot_50"], "outboundTag": "out_slot_50" }
    ]
  }
}
```

**Slot 编号 → 端口映射**: `port = 30000 + slot_index` (slot 1 → 30001, slot 50 → 30050)。

### 3.4 DB schema 改动

```sql
-- 新增表: 每台 server 50 个 slot 槽位
CREATE TABLE xray_slot_pool (
    server_id     VARCHAR(32) NOT NULL,
    slot_index    INT NOT NULL,
    used          BOOLEAN NOT NULL DEFAULT FALSE,
    used_by       VARCHAR(32) NULL COMMENT '占用此 slot 的 xray_client.id',
    updated_at    DATETIME NOT NULL DEFAULT CURRENT_TIMESTAMP ON UPDATE CURRENT_TIMESTAMP,
    PRIMARY KEY (server_id, slot_index),
    INDEX idx_used (server_id, used)
);

-- 初始化: 每台新 server 部署完成后, 插入 50 行 slot_index=1..50, used=false
-- (在 nook server 注册流程里完成, 见 3.7)

-- xray_client 加字段
ALTER TABLE xray_client ADD COLUMN slot_index INT NOT NULL COMMENT '占用的 slot 编号 1-50';
ALTER TABLE xray_client ADD COLUMN listen_port INT NOT NULL COMMENT '客户独享监听端口 = 30000 + slot_index';

-- resource_server 加字段, 用于 replay 检测
ALTER TABLE resource_server ADD COLUMN last_replayed_at DATETIME NULL COMMENT 'nook 最近一次 replay 时间';
ALTER TABLE resource_server ADD COLUMN slot_pool_size INT NOT NULL DEFAULT 50 COMMENT '该 server 的 slot 池大小';
```

### 3.4.5 Install 脚本模块化设计

现有 `install-line-server.sh.tmpl` 把所有逻辑塞在一个 ~250 行的脚本里 (xray + UFW + BBR + 时区 + 用户创建 + config 生成 ...), 阶段 1 借势重构成**模块化**:

#### 拆分原则

- **xray 模块必装** (它是产品核心)
- **其他模块前端勾选**: UFW / BBR / swap / 时区 / fail2ban / unattended-upgrades 等
- **每个模块自洽** (独立的 `.sh.tmpl`, 单独 cat 即可读懂)
- **nook Java 端按勾选拼接**, 不引入新模板引擎依赖
- **xray 版本可指定** (前端默认填项目预设的稳定版, 高级用户可改)

#### 文件目录

```
scripts/templates/
├── modules/                              ← 各独立模块, nook 按需挑选
│   ├── 00-prepare-env.sh.tmpl            必装 (root 检查 / OS 检查 / apt update)
│   ├── 10-timezone.sh.tmpl               可选 (timedatectl set-timezone)
│   ├── 20-swap.sh.tmpl                   可选 (创建 swapfile + persist)
│   ├── 30-bbr.sh.tmpl                    可选 (启用 BBR 拥塞控制)
│   ├── 40-ufw.sh.tmpl                    可选 (UFW + allow 22 + allow 30000-30050)
│   ├── 50-xray.sh.tmpl                   必装 (xray 安装 + 预置 50 placeholder config + systemd)
│   └── 99-finalize.sh.tmpl               必装 (打印总结 / 公网 IP / 关键端口)
└── orchestrator.sh.tmpl                  (备选) 极简编排器, 仅在 nook 不想拼脚本时用
```

数字前缀决定执行顺序, **永远** 00 → 10 → ... → 99。

### 3.6 业务流程

#### provision (开通客户)

```
1. nook 收到 provision(memberUserId, ipId, serverId, protocol)
2. DB 事务开始
   ① selectByMemberAndIp 检查是否已开通
   ② 占 slot:
        UPDATE xray_slot_pool
        SET used=true, used_by=NULL  -- used_by 等 client.id 生成后回填
        WHERE server_id=? AND used=false
        ORDER BY slot_index LIMIT 1
        → 拿到 slot_index=N
   ③ 生成 newUuid + email = "member_{userId}_{ipId}"
3. gRPC AddInboundHandler:
        tag = "in_slot_N"
        port = 30000 + N
        protocol = vless (含 REALITY 配置)
        users = [{ name: email, uuid: newUuid }]
4. gRPC RemoveOutboundHandler(tag = "out_slot_N")
        → 移除占位 freedom
5. gRPC AddOutboundHandler:
        tag = "out_slot_N"
        protocol = "socks"
        servers = [{ address: ipPool.socks5Host, port, users }]
6. DB INSERT xray_client (server_id, ip_id, member_user_id, slot_index=N,
        listen_port=30000+N, client_uuid, client_email, status=1)
7. DB UPDATE xray_slot_pool SET used_by=client.id
8. DB COMMIT
9. 返回 ClientRespVO + 订阅链接 (vless://uuid@server:30000+N?...)
```

**0 重启**, **其他客户 0 感知**, 总耗时约 30-50ms。

#### revoke (取消客户)

```
1. findById → slot_index=N
2. gRPC RemoveInboundHandler(tag = "in_slot_N")
        → 该端口停止 listening, 该客户连接断
3. gRPC RemoveOutboundHandler(tag = "out_slot_N")
4. gRPC AddOutboundHandler:
        tag = "out_slot_N"
        protocol = "freedom"
        → ⚠️ 必须还原占位, 否则 routing rule 引用空 tag
5. DB 事务:
   ① UPDATE xray_client SET deleted=1 (软删 @TableLogic)
   ② UPDATE xray_slot_pool SET used=false, used_by=NULL WHERE slot_index=N
6. DB COMMIT
```

**只有被 revoke 的客户自己断**, 其他 19 个客户不受影响。

#### rotate (换密钥)

```
1. findById → slot_index=N, port=30000+N, uuid_old
2. newUuid = UUID.randomUUID()
3. gRPC RemoveInboundHandler(tag = "in_slot_N")
4. gRPC AddInboundHandler:
        tag = "in_slot_N"
        port = 30000 + N  (★ 同端口)
        users = [{ name: email, uuid: newUuid }]
5. DB UPDATE xray_client SET client_uuid=newUuid
```

**端口不变, 客户订阅链接只换 UUID**, 中间 ~50ms 客户重连即恢复。其他客户 0 感知。

#### rotateIp (换落地 IP)

```
1. findById → slot_index=N
2. ipPool 加载新 IP 凭据
3. gRPC RemoveOutboundHandler(tag = "out_slot_N")
4. gRPC AddOutboundHandler:
        tag = "out_slot_N"
        protocol = "socks"
        servers = [{ address: 新 IP, port, users }]
5. DB UPDATE xray_client SET ip_id=newIpId
```

**vless 入站连接保持** (inbound 没动), 客户感知"出网 IP 切了", 不需要改任何客户端配置。

### 3.7 启动 replay 机制

**为什么需要**: xray 进程重启后 (主机重启 / OOM / 升级二进制), 运行时 inbound 与真实 outbound 全没了, 只剩预置 placeholder。需要 nook 把 DB 的活跃客户重新通过 gRPC 灌进去。

```
@Scheduled(fixedDelayString = "${nook.xray.replay.interval:30000}")
public void replay() {
    for (server : DB.findActiveServers()) {
        // ① 通过 SSH probe 拿 xray uptime
        Instant xrayUptime = serverProbe.getUptime(server.id);

        // ② 比对上次 replay 时间
        if (xrayUptime > server.lastReplayedAt) {
            log.warn("[replay] server={} xray 重启过, 启动 replay", server.id);

            // ③ 扫 DB 全量活跃 client
            List<XrayClientDO> activeClients = mapper.selectByServerIdAndDeletedFalse(server.id);

            // ④ 逐个 replay (幂等: ALREADY_EXISTS 即忽略)
            for (XrayClientDO client : activeClients) {
                replayOne(server.id, client);
            }

            // ⑤ 标记 last_replayed_at = now
            DB.updateLastReplayedAt(server.id, Instant.now());
        }
    }
}

private void replayOne(String serverId, XrayClientDO client) {
    int N = client.getSlotIndex();
    try {
        // 替换 placeholder outbound
        outboundClient.removeOutbound(serverId, "out_slot_" + N);
    } catch (BusinessException e) {
        if (e.code != NOT_FOUND) throw e;  // 幂等
    }
    outboundClient.addOutbound(serverId, "out_slot_" + N, /* socks5 from ip_pool */);

    // 加 inbound
    try {
        inboundClient.addInbound(serverId, "in_slot_" + N,
                                 30000 + N,
                                 client.getClientEmail(),
                                 client.getClientUuid());
    } catch (BusinessException e) {
        if (e.code != ALREADY_EXISTS) throw e;
    }
}
```

**关键点**:
- 检测信号: `systemctl show xray -p ActiveEnterTimestamp` 与 `last_replayed_at` 比较
- 幂等: 重复 add 同 tag 收到 ALREADY_EXISTS 即忽略
- 30 秒检测间隔 → xray 重启后客户最多 30 秒不通即恢复
- nook 启动时立即跑一次 replay (不依赖第一次定时触发)

### 3.8 异常路径与回滚

provision 中任一步 gRPC 失败时, 必须把已产生的副作用回滚:

| 失败步骤 | 已产生副作用 | 回滚动作 |
|-|-|-|
| ② 占 slot | DB slot 已 used | DB 释放 slot |
| ③ AddInbound | inbound 已加 | gRPC RemoveInbound + 释放 slot |
| ④ RemoveOutbound 占位 | 占位 freedom 已删 | gRPC AddOutbound 还原占位 + 反向回滚步骤 ③ |
| ⑤ AddOutbound 真实 | 真实 outbound 已加 | gRPC RemoveOutbound + AddOutbound 还原占位 + 反向回滚步骤 ③ |
| ⑥ DB INSERT 失败 | inbound + outbound 都已加, slot 已 used | 全套反向 gRPC + 释放 slot |

**回滚本身可能失败** (例如远端 gRPC 网络抖动): 此时记录详细日志 + 把 server 加入 reconcile 队列, 由下次 replay 兜底。

代码组织建议: `XrayClientServiceImpl.provision` 用 try-catch 嵌套, 每层 catch 调对应的回滚方法。

### 3.9 风险与 spike 验证项

实施前必须用一台测试机跑通以下 spike (1-2 天):

| 风险 | 验证方式 | 备选方案 |
|-|-|-|
| **xray 启动时 routing rule 引用还不存在的 inboundTag 是否报错** | 部署预置 config (含 50 条 rule, 0 个客户 inbound) → 看 xray 是否成功启动 | 如果报错, 改成"启动时 routing.rules 为空, replay 时按需补 rule"。但 RoutingService.AddRule 不存在, 这条退路不通 → 真不行就要写 50 条 rule + 50 个最简 dummy inbound |
| **gRPC RemoveOutbound 后, 引用该 tag 的 routing rule 把流量送哪** | 删除 out_slot_15 → 在 in_slot_15 上发流量, tcpdump 看包流向 | 如果走默认 outbound (有可能漏到 freedom direct, 客户 IP 泄漏), 必须保证"删除 placeholder + 加 socks5"两步是原子的, 中间不能有流量。考虑加锁串行化同 slot 操作 |
| **同 inboundTag/outboundTag 的 add → remove → add 是否有竞态** | 并发 100 次 provision/revoke 同 server 不同 slot | 加 server-level ReentrantLock |
| **xray 进程 OOM kill 后 nook replay 的幂等性** | 灌入 5 个客户 → kill -9 xray 进程 → systemctl 自动拉起 → 等 30 秒 → 验证客户连得上 | 幂等代码已在 3.7 设计 |

**spike 验收标准**: 4 个风险点的真机验证全部通过, 且并发 100 次 provision/revoke 无 inbound 状态错乱。

### 3.10 阶段 1 验收标准

完成阶段 1 必须满足:

- [ ] 预置 config.json 部署后 xray 正常启动, 无错误日志
- [ ] provision 一个客户后, 客户能用 vless 连上专属端口, 出网 IP = 落地 socks5 IP
- [ ] **provision 第二个客户时, 第一个客户连接保持不断**
- [ ] revoke 第一个客户, 第二个客户连接保持不断
- [ ] rotate 第一个客户的密钥, 端口不变, 用新 UUID 立即连得上
- [ ] rotateIp 第一个客户, vless 连接保持, 出网 IP 已切到新落地
- [ ] xray 重启 (systemctl restart) → 30 秒内 nook replay 完成 → 所有客户重新可连
- [ ] kill -9 xray 进程 → systemd 拉起 → replay 后客户可连
- [ ] provision 失败 (远端 gRPC 抛错) → DB / xray 状态都干净, 没有残留
- [ ] 并发 20 次 provision 不同 slot, 全部成功且 slot 分配无重叠

## 七、决策点

### 决策点 1: 阶段 1 完成后 (1-2 周后)

**触发条件**: 阶段 1 的所有验收标准达成。

**决策**: 是否启动阶段 2?

- **启动**: 商业模式已初步验证, 已有意向客户名单或在 V2Board / Xboard 上跑过付费用户
- **不启动**: 没有客户验证, 先停下来用现有 nook-admin 跑半年商业, 验证后再投阶段 2

### 决策点 2: 阶段 2 完成后 (2-3 个月后)

**触发条件**: Portal 上线, 至少跑 1 个月, 拿到真实付费数据。

**决策**: 是否启动阶段 3?

- **启动**: 客户数稳定 200+, 月毛利 ≥ $3k, 故障人工处理已变成瓶颈
- **不启动**: 客户量未达 200, 自愈不是当前瓶颈, 先把精力放在拉新和反封锁

### 决策点 3: 任何时候

**触发条件**: 出现以下信号之一:

- **商业失败**: 6 个月内付费客户 < 50, 月入 < $1k → 暂停 nook 自研, 切 V2Board 或考虑放弃业务方向
- **技术信号**: Xray 上游有重大变更影响 1:1 模型 → 评估迁移路径
- **合规风险**: 受到法律 / 平台投诉 → 优先解决合规, 暂停功能迭代

## 九、术语表

| 术语 | 含义 |
|-|-|
| 线路机 / Server | 客户接入的中转节点, 跑 xray, 有自己的公网 IP |
| 落地 / Landing IP | 客户最终出网的 IP, 跑 socks5 服务 (3proxy / dante / gost) |
| 共享 inbound | 旧模型: 一个 vless inbound 容纳所有客户, 用 email 区分 |
| 1:1 inbound | 新模型: 每客户独享一个 inbound, 监听独立端口, 只有 1 个 user |
| Slot | 1:1 模型下的"槽位", server 启动时预置 50 个, 业务侧分配/回收 |
| Placeholder outbound | server 启动时预置的占位 freedom outbound, 让 routing rule 启动合法; 业务用时被 gRPC 替换 |
| Replay | nook 启动 / xray 重启后, 把 DB 活跃客户通过 gRPC 重新灌入 xray 进程的过程 |
