# 11-线路机故障切换设计-评审稿

> 版本: v1.0(决策已定) ｜ 日期: 2026-06-01 ｜ 定位: 把 [06-Roadmap §9 故障切换](./06-Roadmap-待建.md) 的设计意图,结合本轮已落地的能力收敛成**可实施设计**。§9 六项决策已拍板,可进入实施(由总开关 `nook.failover.enabled` 控制,默认关)。
>
> 本轮新增/变化的依赖:① 统一准入网关 `ResourceServerAdmission.filterAllocatable`(到顶/掉线/非LIVE 一处判定);② 增量计量 + `THROTTLED`(配额到顶 0.85);③ **`agent_task` 已移除,客户端态走 reconcile**(desired-state 拉取收敛)——所以 06 §9 里"派 task 改 outbound"的机制**作废**,本稿改用 reconcile。

---

## 1. 范围

- **只做线路机(frontline)故障切换**。落地机(landing)是用户独享出口 IP,**不自动换**(admin 手动,06 §9.3 不变)。
- 目标:**到顶 / 掉线**的 frontline 上的**存量订阅**,自动迁到**同区域**健康 frontline;用户客户端刷新订阅后恢复(**无 DNS,订阅接口动态返回新 host**)。

## 2. 触发信号(全部复用现状,不新增检测)

| 信号 | 来源(已实现) | 紧急度 | 迁移策略 |
|---|---|---|---|
| `THROTTLED`(配额到顶 ≥0.85) | `ResourceServerTrafficServiceImpl.markQuotaReached` | 低 | **渐进迁移**:把剩余 15% 余量当 drain runway,分批迁,迁完前服务不断 |
| `OFFLINE`(心跳 ≥5min) | `AgentHeartbeatTimeoutJob`(现仅记 `FAILOVER` 日志) | 高 | **紧急迁移**:尽快迁完(服务已中断) |

> 二者紧急度不同但目标选择逻辑相同,差别在"迁移速率"(§6)。

## 3. 核心原则

1. **换 frontline,保 landing**:出口 IP 不变,对客户只是中转点换了(frontline/landing 分离的意义)。
2. **迁移 = 改 `xray_client.server_id` → reconcile 自动收敛**:旧 frontline 的 `getDesiredClients` 不再含该 client → 其 reconcile 把它从 xray 摘除;新 frontline 的 `getDesiredClients` 新增它 → 其 reconcile 装上。**不走 task**(已移除),不在事务内直接调 agent。
3. **目标选择 = 复用 `pickFrontline`(同分配策略)**:同区域 + `filterAllocatable` 健康准入 + 带宽余量(committed+10%预留)+ 客户数上限 + 取余量最大。**故障源天然被 `filterAllocatable` 排除**(它 THROTTLED/OFFLINE),无需显式 exclude。
4. **逐个迁、每次重算余量**:每迁完一个就更新绑定 → 下一个 `pickFrontline` 看到的 committed 是新的 → **自分散**,不会把整机用户全堆到当前最空的一台。

## 4. 目标选择(复用 + 一处小改)

```
对故障机 X 上每个活跃订阅 s(逐个):
  Y = pickFrontline(s.plan.region, s.plan.bandwidth)   // 复用,X 已被 filterAllocatable 挡掉
  if Y != null: 迁移 s → Y(§5)
  else:         无目标兜底(§4.1)
```

- **手动排空(maintenance)**:源机健康、admin 主动排空时,`filterAllocatable` 不会排除它 → 需给 `pickFrontline` 加 **`exclude` 参数**(像 `pickLanding` 已有的那样)把源机踢出候选。**(唯一需要的代码小改)**

### 4.1 无健康目标兜底(failover 独有,新分配没有)

新分配没目标=拒绝下单即可;故障订阅已付费、服务中,**不能简单拒绝**。三档(同 06 §9.2 收敛):

| 池内情况 | 动作 |
|---|---|
| 有健康 frontline + 有余量 | 正常迁(§5) |
| 余量不足 **或** 无健康 frontline | **留在原机 + ERROR 告警**(`NODE_FAILOVER`),等故障机恢复 / admin 介入 |

> 决策(§9-2):**不硬塞超卖、不放宽预留、不自动跨区** —— 宁可留原机 + 告警,也不把别的机器拖垮或破坏地区语义。

## 5. 迁移单个订阅(动作 + 事务边界)

```
1. Y = pickFrontline(...)                       // 选目标
2. DB(单条事务):rebindFrontline(clientId, Y)   // UPDATE xray_client.server_id = Y(同 client/uuid,不动 ip_id/landing)
3. 发 SubscriptionMachineChangeEvent(FRONTLINE, from=X, to=Y, reason=FAILOVER)  // 落换机历史(已有机制)
4. 收敛(异步,非本事务):
   - X 的 frontline agent 下轮 reconcile → 从 X 的 xray 摘除该 client
   - Y 的 frontline agent 下轮 reconcile → 在 Y 的 xray 装上该 client
5. 用户客户端刷新订阅(sub_token URL 不变)→ 订阅接口返回新 host(=Y 的域名/IP)→ 恢复
```

- **事务边界**:第 2 步是单条 UPDATE(原子,可不显式事务);agent 落地靠 reconcile 异步收敛,**不在事务内调 agent**(符合规范 §10 / 03 §4.7 provision 模式)。
- **uuid/凭据保持不变**:re-point `server_id` 不换 client,客户端 uuid 不变,只是 host 变 → 客户端**刷新订阅**即得新 host(自动刷新的客户端无感;不刷新的需手动刷一次)。
- **需要的新接口**:node 侧一个 `rebindFrontline(clientId, newServerId)`(改 `xray_client.server_id`,显式 set updated_at)。其余全复用。

## 6. 触发与执行编排(不过度设计)

> 不上 op 队列 / 状态机引擎(memory:个位数 server 别上重器)。事件驱动检测 + 一个周期 sweep 足够。

- **一个周期 sweep**(扩 `AgentHeartbeatTimeoutJob` 或新建 `FrontlineFailoverJob`):每轮
  1. 找 `THROTTLED` 或 `OFFLINE` 的 frontline;
  2. 对其上每个活跃订阅,逐个 `pickFrontline` + 迁移(§5);
  3. **幂等**:迁移前再查绑定是否还在故障机,已迁走的跳过。
- **渐进速率**:`OFFLINE` 尽快迁完;`THROTTLED` 每轮限量迁(用 15% runway 平滑),避免一次性把整机用户抖到别处。

## 7. 幂等 / 并发 / 收敛

- **sweep 可重入**:每轮按"当前绑定"决策,迁过的天然不再处理。
- **与新分配并发**:`pickFrontline` 实时算 committed,迁移 + 新分配共用同一不超卖逻辑,不会双重过载。
- **心跳恢复不回迁**:`OFFLINE` 机恢复 `ONLINE` 后,已迁走的**不回迁**(避免抖动);它只是重新进 `filterAllocatable` 候选、可接新单。
- **THROTTLED 到重置日**:到点 `rollover` 把 throttle 复位 NORMAL → 自动重新可分配(已实现)。

## 8. 接线清单(复用为主)

| 复用(已有) | 新增(小) |
|---|---|
| `pickFrontline` / `ResourceServerAdmission.filterAllocatable` | `pickFrontline` 加 `exclude` 参数(手动排空用) |
| `SubscriptionMachineChangeEvent(FRONTLINE)` + 换机历史 | node 侧 `rebindFrontline(clientId, serverId)` |
| `committedBandwidthByFrontline`(不超卖) | `FrontlineFailoverJob`(或扩心跳 Job)|
| frontline reconcile(`getDesiredClients`)收敛 | 无目标兜底的 `force` 放宽预留开关 |
| 订阅接口动态返回 host(`renderSubscription`) | (可选)`NODE_FAILOVER` 告警(06 §8 已列) |

## 9. 决策(已定)

1. **THROTTLED(流量到顶)触发迁移**:✅ **是,渐进迁移** —— 趁剩 15% 余量分批把存量挪走,迁完前不断服务。
2. **无目标兜底**:✅ **留原机 + ERROR 告警**;**不**硬塞超卖、**不**放宽预留(宁可留原机告警,也不拖垮别的机器)。
3. **巡检节奏**(默认值,可调):每分钟巡一次;OFFLINE 尽快迁完,THROTTLED 每轮限量迁(渐进)。
4. **用户连接密钥(uuid)**:✅ **保持不变**,只换 host(新中转机),用户刷新一次订阅即恢复。
5. **跨区切换**:✅ **永不自动跨区**;跨区只 admin 手动。
6. **心跳恢复后回迁**:✅ **不回迁**(挪走就稳定待新机,避免来回抖动)。

> 总开关:整套故障切换由 `nook.failover.enabled` 控制,**默认关**;确认无误后再打开,避免测试期误迁。
