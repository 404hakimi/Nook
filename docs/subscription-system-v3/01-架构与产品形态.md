# 01-架构与产品形态

> v3 总览 — 产品形态 + 技术架构. 所有人必读.

---

## 1. 产品形态

Nook 卖**独享 IP 代理服务**, 运营模型类比独享公寓出租:

- **公寓 (落地 IP)** — 用户独享一个 IP, 14 天保留期
- **门禁 (线路机)** — 多套公寓共用一个入口, 用户的 vmess 流量从这里进
- **套餐 SKU** — 商品 = 一组门禁 + 一组公寓 + 流量限额 + 月租

**心智模型**: admin **基于现有资源** 决定上架哪些 SKU (自下而上). 公寓卖光 → SKU 自动售罄, 加货 → 自动上架.

| 维度 | 说明 |
|---|---|
| **客户** | 跨境电商 / LLM 开发者 / 远程办公; 对"住宅 IP / ISP IP / 干净 IP"敏感 |
| **商品** | admin 预定义并绑定具体资源; 不暴露用户自定义构建 |
| **价位带** | 闲鱼小作坊级 SaaS, 单 IP 套餐 ¥40-¥300/月 |
| **客户感知** | 每 sub 一个独立子域名 (e.g., `jp-alice-001.nook.com`), 写在 vmess URL 里 |
| | 用户级聚合订阅 URL (`member.sub_token`) 返回该用户所有活跃 sub 的 vmess 节点 |
| | 切换线路机走 DNS 改指向, 用户客户端 5-15min DNS 缓存过期后自动续上 |
| | 14 天 IP 保留期: 套餐过期后 IP 仍归用户 14 天, 续费能拿回原 IP |

### 2.2 SKU 与资源的关系

```
[SKU: JP 住宅 100GB]                              ❌ Sprint 1 待建
  ↓ plan_sku_resource 关联表 (详见 02-数据模型.md)
  ├─ 门禁池: [server-jp-01, server-jp-02, server-jp-03]
  └─ 公寓池: [ip-jp-1, ip-jp-2, ..., ip-jp-10]
       ↓
  容量 = 池里 LIVE + AVAILABLE 公寓数 (e.g., 10)
  已售 = 已被占用的公寓数 (status=OCCUPIED)
  剩余 = 容量 - 已售 → 显示在商品页

[卖出一个套餐]
  从公寓池选一个空闲 IP + 从门禁池选客户数最少的 server
  分配给用户, 公寓状态: AVAILABLE → OCCUPIED

[售罄]
  剩余 = 0 → 商品页"已售罄, 不可购买"
  admin 加 IP 进池 → 剩余 +N → 自动上架
```

### 2.3 业务流程概览

1. **admin 进货** ✅ — 上架 SKU 时绑定门禁池 + 公寓池 → SKU 容量自动按公寓池算 (Sprint 1)
2. **用户购买 SKU** ❌ — allocator 从池里挑 1 公寓 + 1 门禁 → backend 派任务给门禁 agent → agent 调本地 xray API 加 user → 同时调 CF API 创建独享子域名 (Sprint 1)
3. **用户使用** ❌ — 客户端连子域名 → DNS 解析到门禁 IP → xray 认证 → 路由到公寓 → 出站 (Sprint 1)
4. **故障 / 流量耗尽** ❌ — 池内调度选其他门禁 + 改子域名 DNS → 用户 5-15min 自动恢复 (Sprint 2)
5. **运行期** ✅ — 每台机器跑 agent → 主动 push 流量 / 心跳 / 状态 → backend 收数据 + 派任务

## 3. v3 关键决策

### 3.1 核心架构

- ✅ **VPS 实例模式**: 1 套餐 = 1 IP = 1 subscription (不共享池)
- ✅ **SKU 资源池**: `plan_sku_resource` 关联表显式绑定 SKU 到 server + IP 池, 容量 = 数 IP (砍 sold_traffic_gb 复杂度)
- ✅ **1 sub 1 server + DNS 切换**: 用户独享子域名, 切换走 DNS 改向 (5-15min 缓存恢复)
- ✅ **Agent 架构 (Go)**: 服务器跑轻量 agent 主动 push 数据 + pull 任务, backend 不再 SSH 拉
- ✅ **不限带宽**: xray / dante 都不限速, 用户跑满 IP 物理上限; 通过 `xray_node.touchdown_size` 控制单机用户密度间接管控带宽分摊

### 3.2 数据模型

- `plan_sku_resource` 关联表 (新增, 核心) — Sprint 1
- `sub_main` 加 `sub_domain` + `cf_record_id` (1:1 关系恢复) — Sprint 1
- `resource_server` 拆 6 张子表 (主表 + credential + billing + dns + capacity + runtime, 按更新频率分层) ✅
- `resource_server_capacity` 砍 `sold_traffic_gb` / `oversubscription_ratio` ✅
- `agent_task` + `agent_runtime_config` ✅
- 客户数上限走 `xray_node.touchdown_size` ✅

详见 [02-数据模型.md](./02-数据模型.md).

### 3.3 故障切换

- Agent 心跳分级: 1min WARN / 3min 暂时不健康 / 5min 真切换 (避免误判) ✅ 心跳 + temp_unhealthy 已实施; 真切换 ❌
- 池内全 THROTTLED 紧急 fallback (`force=true`) 救急 + ERROR 告警 — Sprint 2
- CF API 调用拆事务外 + ERROR 状态 (避免长事务 / 孤儿记录)

详见 [04-监控-流量-告警-切换.md §5](./04-监控-流量-告警-切换.md).

### 3.4 排除项 (起步不做)

| 项 | 范围 | 备注 |
|---|---|---|
| ❌ Mesh 多备份 | Sprint 5+ | 不预部署多 server, 用 DNS 缓存替代 |
| ❌ 跨实例配额聚合 enforce | — | 单 server xray totalBytes 准确, 不需要 |
| ❌ 流量包加购 | Sprint 5+ | 现状不做 |
| ❌ 用户自定义套餐 | — | admin 预定义 |
| ❌ CDN 套代理 (CF Proxied) | — | 直连 server IP, 避开 ToS |
| ❌ Agent 自动升级 | ✅ 已实施 | 走 task 派发, 不再 admin SSH |

完整决策表见 [07-决策](./07-决策-排除-风险-Checklist.md).
