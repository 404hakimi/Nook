## 1. 产品形态

Nook 卖**独享 IP 代理服务**, 运营模型类比独享公寓出租:

- **公寓 (落地 IP)** — 用户独享一个出网 IP, 14 天保留期
- **门禁 (线路机)** — 多套公寓共用一个入口, 用户的 vmess 流量从这里进
- **套餐 SKU** — 商品 = 一组门禁 + 一组公寓 + 流量限额 + 月租

**心智模型**: admin **基于现有资源** 决定上架哪些 SKU (自下而上)。公寓卖光 → SKU 自动售罄, 加货 → 自动上架。

| 维度 | 说明 |
|---|---|
| **客户** | 跨境电商 / LLM 开发者 / 远程办公; 对"住宅 IP / ISP IP / 干净 IP"敏感 |
| **商品** | admin 预定义并绑定具体资源; 不暴露用户自定义构建 |
| **价位带** | 闲鱼小作坊级 SaaS, 单 IP 套餐 ¥40-¥300/月 |
| **客户感知** | 用户级聚合订阅 URL (`https://nook.com/api/sub/{member.sub_token}`), 用户只存这一个, 可在 portal 重置 token |
| | 订阅接口动态返回该用户所有活跃 sub 的 vmess 节点 (host = 各 sub 当前所在线路机的固定域名) |
| | 切换线路机 = 后台改 DB + 订阅接口返回新节点; 用户刷新订阅后恢复 (不碰 DNS) |
| | 14 天 IP 保留期: 套餐过期后 IP 仍归用户 14 天, 续费能拿回原 IP |

> 产品形态是 v3 的目标态。当前代码已能管理资源 + 手动开客户端, 但"用户自助下单 → 自动分配 + 计费"链路待建 (见 [06-Roadmap](./06-Roadmap-待建.md))。

## 2. 技术总览

### 2.1 整体架构图

```
                    ┌────────────────────────────────────────┐
                    │       Nook 后台 (Java Backend)          │
                    │ ─ admin: 资源管理 / 上架 / 运维 (✅)    │
                    │ ─ portal: 用户购买 / 续费 (❌ 待建)     │
                    │ ─ 调度: 收数据 / 派任务 / 告警 (🟡)     │
                    │ ─ op 编排: 危险操作 FIFO 串行 (✅)      │
                    └────────────────────────────────────────┘
                       ▲ POST 数据 (1/5 min)    ▼ GET 任务 (轮询 60s)
                       │                         │
        ┌──────────────┴─────────────────────────┴──────────┐
        ▼                                                    ▼
   ┌────────────────────┐                       ┌────────────────────┐
   │ 线路机 (门禁)       │ ── vmess+ws+TLS ──→  │ 落地机 (公寓)       │
   │ resource_server    │     per-user routing  │ resource_server    │
   │  type=frontline    │      ↓ socks5 出站    │  type=landing      │
   │ xray + nook-agent  │ ────────────────────→ │ dante + nook-agent │
   └────────────────────┘                       └────────────────────┘
        ▲                                                    ▲
        │ vmess (加密) 入口                          socks5 出口 到互联网
        │                                                    │
   ┌────┴─────────┐                              ┌─────────────────┐
   │ 用户 v2rayN  │                              │   Internet      │
   └──────────────┘                              └─────────────────┘

                 ▲▲▲ 仅首次装机用 SSH ▲▲▲
                 (装好 agent 后, backend 不再 SSH 主动出站)
```

> **关键**: 线路机和落地机在数据模型上都是 `resource_server`, 靠 `server_type` 区分。两者都跑 nook-agent (frontline 跑 xray executor, landing 当前 socks5 executor 空壳)。

### 2.2 SKU 与资源的关系 ❌ (待建, 见 [06-Roadmap §1.2](./06-Roadmap-待建.md))

```
[SKU: JP 住宅 100GB]
  ↓ trade_plan_resource 关联表
  ├─ 门禁池: [frontline-jp-01, frontline-jp-02]   (resource_server, type=frontline)
  └─ 公寓池: [landing-jp-1, ..., landing-jp-10]    (resource_server, type=landing)
       ↓
  容量 = 池里 LIVE + AVAILABLE 公寓数 (e.g., 10)
  已售 = OCCUPIED 公寓数; 剩余 = AVAILABLE → 显示在商品页
```

### 2.3 业务流程概览

| 步 | 流程 | 状态 |
|---|---|---|
| 1 | **admin 进货**: 录入线路机 / 落地机 + SSH 装机 (xray / dante / agent) + lifecycle 流转 | ✅ |
| 2 | **admin 上架 SKU**: 绑定门禁池 + 公寓池, 容量按公寓池算 | ❌ |
| 3 | **admin 手动开客户端**: 指定落地机 + 会员 → op 框架 provision (远端 adu/adr/ado) | ✅ |
| 4 | **用户自助购买**: allocator 自动选 1 公寓 + 1 门禁 → 复用 provision (无 CF DNS) | ❌ |
| 5 | **运行期**: agent 主动 push 流量 / 心跳 / 状态, backend 收数据 | ✅ |
| 6 | **故障 / 流量耗尽**: 同 region 池内切到另一台线路机 + 订阅刷新恢复 | ❌ |

> 当前 admin 可以走 ③ 手动开客户端验证整条数据通路; ②④⑥ 的自动化是 roadmap。

## 3. v3 关键架构决策

### 3.1 已落地

- ✅ **落地机统一为 resource_server**: server_type=landing, 跟线路机共用主表 + credential/billing/capacity/runtime 子表, 各挂角色专属子表 (frontline → DNS, landing → socks5/dante)
- ✅ **Agent 架构 (Go)**: 主动 push 数据 + pull 任务, backend 不再维护 SSH 连接池 (仅装机用)
- ✅ **op 编排框架**: xray client provision/revoke/rotate/sync 走 `OpOrchestrator` — 同 server FIFO 串行 + watchdog 超时 + WebSocket 进度。避免并发危险操作互相踩
- ✅ **流量两层采集**: 用户层 xray statsquery (契约"100GB 用户感知流量") + 机房层 vnstat (机房账单), 互不推算

### 3.2 待建 (设计已定)

- 🟡 **VPS 实例模式**: 1 套餐 = 1 落地 IP = 1 subscription
- 🟡 **SKU 资源池**: `trade_plan_resource` 显式绑定 SKU 到 frontline + landing 池, 容量 = 数 landing
- 🟡 **1 sub 1 server + 订阅刷新切换**: 订阅 URL 动态返回当前节点, 切换改 DB + 客户端刷订阅恢复 (不碰 DNS, vmess host 用线路机级固定域名)
- 🟡 **dante + tc 双层限速 + 链路守恒**: 落地机 dante per-IP + 线路机 tc 兜底; admin 进货强校验 `Σ落地机带宽 ≤ 线路机带宽 × 0.9` (预留率默认 10%)

详见 [06-Roadmap](./06-Roadmap-待建.md)。

### 3.3 排除项 (起步不做)

| 项 | 备注 |
|---|---|
| ❌ Mesh 多备份 | 用 1 sub 1 server + 订阅刷新切换替代 |
| ❌ per-sub 子域名 + CF DNS 切换 | 砍掉; 订阅 URL 做间接层, 切换改 DB |
| ❌ 跨实例配额聚合 enforce | 单 server xray totalBytes 准确 |
| ❌ 流量包加购 | Sprint 5+ |
| ❌ 用户自定义套餐 | admin 预定义 |
| ❌ CDN 套代理 (CF Proxied) | 直连 server IP 避开 ToS |
| ✅ Agent 自动升级 | 已实施, 走 task 派发 |

完整决策表见 [07-决策](./07-决策-排除-风险-Checklist.md)。

## 4. v3 跟 v1 / v2 的演进
