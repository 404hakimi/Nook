# 10 · 线路机/落地机 流量计量与重置设计(评审稿)

> 版本: v0.1 评审稿 ｜ 日期: 2026-05-31 ｜ 范围: `resource_server_capacity` NIC 流量计量 + 周期重置 + 限流/切换衔接
>
> 本稿目标: 在"厂商重置时间不可控"的现实下,给出一套**自定义、保守、容错**的流量计量与重置方案,供评审后落地。

---

## 1. 背景与现状问题

`resource_server_capacity` 存的是 **agent 上报的服务器网卡(NIC)流量**,用于"线路机/落地机当周期已用多少、是否接近配额、是否限流"。

### 1.1 当前实现(现状)

- 表 `resource_server_capacity` 跟服务器 **1:1**(`server_id` 主键)→ **每台一行,反复覆盖**,无 period 列、无历史。
- agent(`nic.go`)只取 vnstat **当前自然月**累计,`periodStart` 恒为月 1 号。
- backend `receiveNicTraffic` → `applyNicTraffic` 做的是 **整行覆盖** `SET rx=?, tx=?, used=rx+tx`;上报的 `periodStart` **只打日志、不存不用**。
- `quota_reset_policy`(`BILLING_CYCLE 按账单日重置` / `FIXED 永不重置`)与账单日 **均未生效**,实际只有"自然月翻篇"。

### 1.2 问题清单

| # | 问题 | 影响 |
|---|---|---|
| P1 | 单行覆盖、无历史、无 period 列 | 看不到上月/历史用量,无法审计/对账 |
| P2 | `quota_reset_policy` + 账单日是死配置 | "按账单日重置"承诺了但不生效 |
| P3 | 90% 限流阈值按自然月算 | 若本意按业务周期配额,重置日错 |
| P4 | 覆盖写依赖 agent 绝对值 | agent 重装 / vnstat 库被清,用量会"变小",翻篇丢上月终值 |

### 1.3 运营现实(不可控因素)

- 多数 VPS 厂商**不公开**确切的流量重置时刻;只能设个**大概日期**。
- 即便日期对,重置可能在 **0 点 / 上午 8 点**,且厂商走 **UTC 或各自时区**。
- 服务器是宿主机虚拟化出来的,可能**整机故障并完全重置**;厂商也可能**把网卡计数归 0**。

> 结论: 重置时刻**对不准也无法对准**。设计不应追求"跟厂商精确对齐",而应**容错**。

---

## 2. 设计原则

1. **平台自定义重置时间** —— 不追厂商。我方定一个"近似重置日 + 统一时区",对不准没关系。
2. **配额留安全余量** —— `monthlyTrafficGb` 设成厂商实际上限的 ~80–90%。时区/日期的偏差被余量吸收。
3. **容错优先于精确** —— 计量要抗"计数器归零 / 整机重置 / agent 重装";账存在后端 DB,跟服务器解耦。
4. **核心目的是"容量守护"而非"对账"** —— 用量接近配额 → 限流、停止分新用户、(可选)迁移;真超了被厂商掐 → 当故障 → 现有 failover 自动切换兜底。

> 一句话: **自定义近似重置日 + 安全余量 + 增量累加 + failover 兜底**。精度不重要,稳和不超载才重要。

---

## 3. 时区策略(评审点 A)

- **显式钉一个平台时区**,写进配置 `nook.traffic.reset-zone`(默认 `Asia/Shanghai`,即当前后端 +08)。
- **不要隐式依赖** 后端 OS/JVM 默认时区 —— 否则后端迁移/换机/跑在 UTC 容器里时,**全平台重置时刻会悄悄整体平移**,难排查。
- **全平台统一一个时区即可**,不做 per-机器对厂商时区(过度设计;差几小时被配额余量盖掉)。
- 所有"重置日 00:00"的边界判定,一律按 `reset-zone` 计算 `ZonedDateTime.now(zone)`。

> **已定: `reset-zone = Asia/Shanghai`**(跟当前后端 +08 一致)。

---

## 4. 数据模型

### 4.1 `resource_server_capacity` 改造(主表,仍 1:1)

保留业务阈值,新增"游标 + 周期起点",`rx/tx/used` 语义改为**当周期累加值**(到我方重置日清零)。

| 列 | 状态 | 含义 |
|---|---|---|
| `server_id` | 不变 | 主键 |
| `monthly_traffic_gb` | 不变 | 当周期配额(= 我方预算 = 厂商上限 × 安全系数);0/null=不限 |
| `bandwidth_limit_mbps` / `client_max_count` | 不变 | 带宽限速 / 客户数上限 |
| `quota_reset_policy` | 语义调整 | `MONTHLY`(按我方重置日,每月)/ `FIXED`(永不重置:买断 / 不限流量 / 按量付费 机型) |
| `reset_day` | **复用** `billing_cycle_day`→ 或主表加列 | 我方近似重置日(1–28) |
| `rx_bytes` / `tx_bytes` | 语义改 | **当周期累加**下/上行字节 |
| `used_traffic_bytes` | 语义改 | 当周期累加 = rx+tx;限流/展示用 |
| `last_cum_rx_bytes` / `last_cum_tx_bytes` | **新增** | 上次上报的 vnstat **累计游标**(算增量用) |
| `period_start` | **新增** | 当前周期起点(我方重置日 00:00 @ reset-zone) |
| `throttle_state` | 不变 | NORMAL / THROTTLED |

> `reset_day`: 落地机/线路机现已有 `resource_server_billing.billing_cycle_day`(1–28)。建议**直接复用它当"我方重置日"**(含义改成"我们自己定的近似重置日",不再叫账单日),避免新字段;`quota_reset_policy=FIXED` 时忽略。

### 4.2 `resource_server_traffic` 历史表(新增,解决 P1)

每个已结束周期归档一行,可审计/看历史。

| 列 | 含义 |
|---|---|
| `id` | 主键(UUID) |
| `server_id` | 服务器 |
| `period_start` / `period_end` | 周期起止(@ reset-zone) |
| `rx_bytes` / `tx_bytes` / `used_bytes` | 该周期终值 |
| `closed_at` | 归档时间 |

主键/唯一: `(server_id, period_start)`。

## 5. 核心算法:增量累加(delta accumulation)

**思路: agent 报 vnstat "累计总量(`total`,跨月不清零)";backend 只累加"两次上报的增量",周期到点把累加值清零。** 不再"存累计减基线",避免计数器一归零就算错。

### 5.1 agent 侧改动

- `nic.go` 从 `sampleCurrentMonth()`(当前自然月)改为读 vnstat **累计 total**(`interfaces[].traffic.total.{rx,tx}`)。
- 上报体不变(`rxBytes/txBytes`),`periodStart` 可保留也可去掉(backend 不再依赖它)。

### 5.2 backend `receiveNicTraffic(serverId, cumRx, cumTx)` 伪代码

```text
cap = load(serverId)
zone = config.reset-zone
now  = ZonedDateTime.now(zone)

# 1) 周期判定: 是否跨过"我方重置日"
periodStart' = computePeriodStart(cap.quotaResetPolicy, cap.resetDay, now, zone)
    # FIXED        → 恒返回首个周期起点(永不推进 → 永不重置)
    # MONTHLY      → ≤now 的最近一个 reset_day 00:00:
    #                若 now.day >= reset_day → 本月 reset_day; 否则上月 reset_day (clamp 1..28)

if cap.periodStart == null OR cap.periodStart < periodStart':
    archiveHistory(cap, periodEnd = periodStart')   # 旧周期写历史表 (P1)
    cap.rxBytes = 0; cap.txBytes = 0; cap.usedTrafficBytes = 0
    cap.periodStart = periodStart'
    cap.throttleState = NORMAL                       # 周期重置 → 解除限流 (P3)

# 2) 增量累加 (抗计数器归零 / 整机重置 / 首次上报)
if cap.lastCumRx == null:                            # 首次上报 → 只建游标, 不计本段
    dRx = 0; dTx = 0
else:
    dRx = (cumRx >= cap.lastCumRx) ? cumRx - cap.lastCumRx : cumRx   # 变小=被归零 → 从0续
    dTx = (cumTx >= cap.lastCumTx) ? cumTx - cap.lastCumTx : cumTx

cap.rxBytes += dRx; cap.txBytes += dTx
cap.usedTrafficBytes = cap.rxBytes + cap.txBytes
cap.lastCumRx = cumRx; cap.lastCumTx = cumTx

# 3) 配额到顶判定 (容量守护 → 触发同地区切换, 见 §7)
if cap.monthlyTrafficGb > 0 AND cap.usedTrafficBytes >= cap.monthlyTrafficGb * GiB:
    cap.throttleState = THROTTLED                    # 配额到顶: 停止分新用户 + 触发同地区切换

save(cap)
```

- `GiB = 1024^3`。配额 `monthlyTrafficGb` 已 = 厂商上限 × 0.85(留 15% 余量),**到顶即触发即可**,不再叠第二层比例。
- (可选)想更早主动迁,可把单机配额再调小;无需额外阈值参数。

### 5.3 重置触发: 惰性 on-report(无需定时任务)

- 每次上报顺手判定是否跨周期,跨了就地清零。上报频繁(分钟级),**重置最多滞后一个上报间隔**,运营无感。
- **掉线的机器**不上报 → 不会触发重置;但掉线机器本就不产流量,且由 failover 接管,**无需**为它单独 cron。
- (可选)加一个**每日兜底 cron** 扫"`period_start` 已过期但长时间没上报"的机器顺手归档,纯锦上添花,非必须。

## 6. 容错矩阵(对应运营现实)

| 场景 | 现象 | 本方案如何兜住 |
|---|---|---|
| 厂商把网卡计数归 0 | `cumRx` 变小 | `cum < lastCum` → 当"从 0 续",累加不丢不负 |
| vnstat 库被清 / agent 重装 | `cum` 重新从小涨 | 同上,自动续 |
| 服务器整机故障 + 完全重置 | agent 重来、`cum` 归 0 | 同上续算;**`used` 存后端 DB,服务器没了我方账不丢**;掉线由 failover 切换 |
| 首次上报 | `cum` 是历史全量,会过量 | 首次仅建游标(`d=0`),不把历史计入 |
| 重置时刻跨报间隔 | 新周期含极小一段旧流量 | 可接受(报间隔短);**已声明的近似误差** |
| 我方重置日 ≠ 厂商重置日 | 两边账错开 | **安全余量吸收**;真超被掐 → 当故障 → failover |

### 6.1 agent 能否胜任流量上报?

**结论: 不追求"完美", 但 vnstat + 增量累加 + 后端存账 + 安全余量 + failover 足够稳。**

- **要不要改 agent**: 要, 但很小 —— 只把 `sampleCurrentMonth()` 改成读 vnstat **累计 total**, 上报协议不变。
- **vnstat 是不是最好的源**: 是本地最优。它**跨重启持久**(落盘), 优于 `/proc/net/dev`、`ip -s link`(重启即清零)。厂商一般不开放流量 API, 故 vnstat 是务实选择。
- **不"完美"的点(都被设计兜住)**:
  - vnstat 须在**装机时装好**并盯**正确网卡**(厂商计量的那块, 一般 eth0/公网口) → 落实到 provision 流程 + agent iface 配置。配错则数偏, 但**超了由 failover 接管**。
  - agent 宕机期间不上报 → 该段**延迟统计**非丢失: 恢复后累计已涨, 增量一次补回。
  - vnstat 库被清 / 整机重置 → 累计归零 → 增量逻辑"从 0 续", 不丢不负。
- **为什么不需要完美**: 我方是"容量守护 + 近似", 有 **0.85 余量 + failover 兜底**。即便 agent 少报/错报导致超了我方预算, 最坏就是被厂商掐 → 当故障 → 同地区切换。**系统对 agent 不完美是有韧性的。**
- **职责是否要拆**: 不用。流量上报本就是 agent 现有 NIC 职责, 加"报累计"不增负担。

## 7. 配额到顶 / 故障 → 统一走"同地区切换"

> 澄清: `throttle_state` **不是带宽限速**, 是"配额到顶 / 需撤离"的状态标志。它的动作跟"真故障"一致 —— 都是**线路机同地区切换**。

两条触发, 同一个动作(同地区 failover):

| 触发 | 性质 | 动作 |
|---|---|---|
| 我方配额到顶 (`used ≥ 配额`) | **主动**(赶在厂商掐之前) | 标 `THROTTLED` → 停止分新用户 + 触发**同地区切换**迁走在用用户 |
| 真故障 (心跳掉 / 被厂商掐 / 整机重置 / 重置日没对上被迫停机) | **被动**(兜底) | 现有 failover → **同地区切换** |

- 主动这条给了"优雅迁移窗口": 配额已留 15% 余量(0.85), 到顶时离厂商真实上限还有空间, 有时间把用户迁到同地区另一台再被掐。
- 没赶上(迁移没完成 / 厂商提前掐)→ 掉线 → 被动 failover 接住。**两条最终都是同地区切换, 用户不超载、能正常用。**
- 周期重置时 `THROTTLED → NORMAL`, 自动恢复可分配。
- **命名建议**: `ResourceServerThrottleStateEnum.THROTTLED` 易误解成"限速", 建议改名 `QUOTA_REACHED` / `EXHAUSTED`(可选小改)。
- **依赖**: 主动迁移 + 故障切换的"同地区 failover"机制本身是**另一块(规划中)**; 本设计只提供**触发信号**(配额到顶标志)+ 停止分配, 迁移动作复用未来的换机/切换逻辑。

| 配置 | 默认 | 说明 |
|---|---|---|

| 每机 `reset_day` | 复用 `billing_cycle_day`(1–28) | 我方近似重置日;`FIXED` 时忽略 |

## 9. 边界与已声明接受的误差

- 我方重置时刻与厂商**不对齐**(可能差几天/几小时)——靠**安全余量 + failover** 兜底,**接受**。
- 周期切换瞬间可能混入极小一段相邻周期流量——报间隔短,**接受**。
- 首次上报不计历史全量——**接受**(避免虚高)。
- vnstat 精度/采样间隔带来的轻微误差——**接受**。

> 核心 SLA: **只要还有配额,用户正常用;接近配额停止超载;真超当故障切换。** 精确对账非目标。

## 10. 数据流链路(目标态)

```
agent: vnstat → 累计 total (cumRx, cumTx)
   │  POST /api/agent/nic-traffic {rxBytes=cumRx, txBytes=cumTx}
   ▼
backend receiveNicTraffic(serverId, cumRx, cumTx):
   ① 按 reset-zone + reset_day 判周期 → 跨了就归档历史 + used 清零 + 解限流
   ② 增量累加: d = cum>=last ? cum-last : cum;  used += d;  last = cum
   ③ used ≥ 配额×ratio → THROTTLED
   ▼
resource_server_capacity (当周期累加 + 游标 + period_start)
resource_server_traffic  (历史归档, 每周期一行)
   ▼
分配器: THROTTLED 不分新用户 + (可选)迁移   ;  超限被掐 → failover 切换
```

## 11. 实施步骤(评审通过后)

1. **DDL(走 MCP 跟用户确认后执行,不留迁移脚本)**: `resource_server_capacity` 加 `last_cum_rx_bytes / last_cum_tx_bytes / period_start`;新建 `resource_server_traffic`。
2. **枚举**: `ResourceServerQuotaResetPolicyEnum` 调整为 `MONTHLY / FIXED`(去掉误导的 `BILLING_CYCLE` 命名,或保留值改 label)。
3. **agent**: `nic.go` 改报 vnstat 累计 total。
4. **backend**: `receiveNicTraffic`/`ResourceServerCapacityMapper` 实现 §5 算法;取 `reset_day`(billing 子表)+ `reset-zone` 配置;归档历史。
5. **限流/分配器衔接**: 确认 `THROTTLED` 已被分配器消费(停止分新用户);迁移为可选。
6. **前端**: 容量展示改读"当周期累加 used";区域/详情可加"历史用量"(读历史表)。

## 12. 评审决策(已定)

| # | 决策 | 结论 |
|---|---|---|
| 1 | 时区 | **`Asia/Shanghai`** |
| 2 | 安全系数 | **0.85**(配额 = 厂商上限 × 0.85) |
| 3 | 重置日 | **复用每机 `billing_cycle_day`**(含义改"我方近似重置日") |
| 4 | 历史表 | **建** `resource_server_traffic` |
| 5 | 配额到顶动作 | **配额到顶 / 故障 → 统一同地区切换**;主动这条先"停止分配 + 触发迁移"(见 §7) |
| 6 | `FIXED` 策略 | **保留**(买断 / 不限 / 按量付费 机型永不重置) |
| 7 | agent | **小改**(报 vnstat 累计 total);不追求完美,靠余量 + failover 兜底(见 §6.1) |

### 仍可继续讨论(非阻塞)
- `THROTTLED` 是否改名 `QUOTA_REACHED`(更准,可选)。
- "同地区 failover / 迁移"机制本身的设计(本设计只给触发信号,机制是另一块,规划中)。
- pay-as-you-go(`FIXED`)机型: `monthly_traffic_gb` 留空 = 纯计量不拦,还是设个自我成本上限、到顶也迁?
