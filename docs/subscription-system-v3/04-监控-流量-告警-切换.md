# 04-监控与流量 (现状)

> **本文只描述已实现的监控**: 流量统计 (用户层 xray + 机房层 NIC) + 心跳监控 + 配额 enforce + 服务状态查询。
> 告警系统 / throttle 状态机 / 周期切换 / 节点故障切换 → 全部未实现, 见 [06-Roadmap-待建.md](./06-Roadmap-待建.md) §7-§10。

---

## 1. 流量统计 (两层, 都由 Agent 上报)

| 层 | 数据源 | 用途 | 状态 |
|---|---|---|---|
| **用户层** | agent 调本地 xray statsquery → POST | 用户配额 enforce + portal 流量 + 对账 | ✅ |
| **机房层** | agent 调本地 vnstat → POST | 机房账单对账 + server 容量水位 | ✅ 接收已实现; 告警评估未实现 |

**关键认知**:
- 业务**永远走 xray** (跟用户契约: "100GB 用户感知流量")
- 机房**永远走 NIC** (跟机房账单契约: "NIC 跑了多少字节")
- 两者**不相互推算**, 各自独立采集
- 数据从 server **主动 push** 到 backend, 不再 SSH 拉

### 1.1 用户层 (xray statsquery)

**Agent** (每 5min): 调 `xray api statsquery` 拿每 user 的 up/down → POST `/api/agent/xray-traffic` (`[{email, up_bytes, down_bytes}]`)。

**Backend** (`AgentReportService.applyXrayTrafficSample`): 算增量 `delta = current - last_counter` → 累加到 `xray_client_traffic.uplink/downlink_bytes`。跨 xray 重启 (counter 归零) delta 为负时不累加 (基线重置)。

### 1.2 机房层 (vnstat)

**Agent** (每 5min): 调 `vnstat -i eth0 --json` → POST `/api/agent/nic-traffic` (`{rx_bytes, tx_bytes}`)。

**Backend** (`AgentReportService.applyNicTraffic`): 覆盖写 `resource_server_capacity.rx/tx/used_traffic_bytes` (vnstat 报绝对值, 非增量)。

> 落地机也是 resource_server, NIC 流量同样落 capacity 表。

### 1.3 流量展示约定

**核心**: 存储**永远分开**上下行, 计费 / 配额**永远用双向求和** (rx+tx 或 up+down)。

| 表 | 上行 | 下行 |
|---|---|---|
| `xray_client_traffic` (用户层) | `uplink_bytes` | `downlink_bytes` |
| `resource_server_capacity` (机房层) | `tx_bytes` | `rx_bytes` |

展示拆明细, 求和用双向。例: `↑ 上传 12.4 GB / ↓ 下载 67.6 GB / 共 80 GB / 配额 100 GB`。

> 周期切换 (跨 period 归零 + 历史归档) 未实现, 当前 capacity 一直累加 vnstat 值。见 [06-Roadmap §7.2](./06-Roadmap-待建.md)。

## 2. 配额 enforce (xray 原生) ✅

VPS 模式每客户独立配额, 由 xray totalBytes 在连接时 enforce (本地, 不依赖 backend):
- provision → xray user 带 `totalBytes / expiry / limitIp` (op 框架下发, 见 [03 §3](./03-业务核心-算法与流程.md))
- 用尽 → xray 自动拒连

**无跨 client 池化**, 无 5min "用尽暂停"调度 — xray 原生 enforce 已处理。

> **带宽 (Mbps) 限速 ≠ 流量配额**。带宽限速 (dante per-IP / tc) 完全未实现, 见 [06-Roadmap §6](./06-Roadmap-待建.md)。

## 3. 心跳监控 ✅

### 3.1 心跳上报

agent 每 1min POST `/api/agent/heartbeat` (+ agent_version) → `resource_server_runtime.last_heartbeat_at` 等。

### 3.2 在线状态分级 (`AgentOnlineState.classify`, 前端展示派生)

| elapsed | 状态 |
|---|---|
| < 60s | ONLINE |
| 60-180s | WARN |
| 180-300s 或 temp_unhealthy=1 | TEMP_UNHEALTHY |
| ≥ 300s | OFFLINE |
| last_heartbeat=null | NEVER |

### 3.3 `AgentHeartbeatTimeoutJob` (每 1min) ✅

扫所有 LIVE server (JOIN runtime), 分级:
- `consecutive_miss` 累加, 满 3 次 (≈3min) → 标 `temp_unhealthy=1`

**现状**:
- ✅ 心跳分级 + temp_unhealthy 自动标
- ❌ ≥5min 真故障切换 (`onServerFault`) 未实现
- ❌ AGENT_OFFLINE 告警未实现 (依赖 alert_record)
- ❌ 心跳恢复自动 resolve temp_unhealthy 未实现

故障切换完整设计见 [06-Roadmap §9](./06-Roadmap-待建.md)。

## 4. 服务状态查询 (通用 systemd 接口) ✅

线路机 xray / 落地机 dante 的运行状态走**同一套通用接口** (`ResourceServerOpsController` @ `/admin/resource/server`):

| 接口 | 用途 |
|---|---|
| `GET /get-system-info` | 主机信息 (hostname / OS / 内核 / uptime / 负载 / 内存 / 磁盘) |
| `GET /get-ufw-status` | UFW 防火墙状态 (ufw status verbose 原文) |
| `GET /get-systemd-status?unit=xxx` | **任意 systemd unit 运行状态** (active / uptimeFrom / enabled); xray 传 `unit=xray`, dante 传 `unit=danted` |
| `GET /get-service-log?unit=xxx` | 任意 unit 的 journalctl 日志 (行数 / 级别 / 关键词过滤) |
| `POST /test-connectivity` | SSH 探活 |
| `GET /list-network-interface` | 列网卡 |
| `POST /enable-swap` / `/enable-bbr` (流式) | OS 调优 |

> 这套接口按 unit 名参数化, xray / dante / nook-agent 各自的 systemd 状态 + 日志都走它。版本信息 (xray_version / dante_version) 从各自 DO 取, 不在 systemd 接口里。

agent 自身日志: `/admin/resource/server/get-service-log?unit=nook-agent` (agent 通过 journald 输出)。
