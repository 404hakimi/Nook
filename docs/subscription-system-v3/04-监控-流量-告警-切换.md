# 04-监控: 流量, 告警, 节点切换

> v3 运行期监控 — 流量统计 / 告警系统 / 节点故障切换 + 容量规划. 后端 + 运维必读.

---

## 五、流量统计

### 5.1 两层职责 (各管各的, 都由 Agent 上报)

| 层 | 数据源 | 用途 |
|---|---|---|
| **用户层** | Agent 调本地 xray statsquery → POST 上报 | 用户配额 enforce + portal 流量进度 + 用户对账 |
| **机房层** | Agent 调本地 vnstat → POST 上报 | 机房账单对账 + Server 容量水位监控 + 90% 告警 |

**关键认知**:
- 业务**永远走 xray** — 跟用户的契约是"100GB 用户感知流量"
- 机房**永远走 NIC** — 跟机房的账单契约是"NIC 跑了多少字节"
- 两者**不相互推算**, 各自有独立采集源
- **数据从 Server 主动 push 到 Backend**, 不再 SSH 拉 (见 [05-Agent架构](./05-Agent架构.md))

### 5.2 用户层 (Agent → xray statsquery)

**Agent 端** (每 5min):
- 调本地 `xray api statsquery -pattern=user>>>email>>>traffic`
- 拿到所有 user 的 up_bytes / down_bytes
- POST 到 backend `/api/agent/xray-traffic`, body 含 `[{email, up_bytes, down_bytes}, ...]`

**Backend 端**:
- 接收 → 写 `xray_client_traffic` 表 (per client, 字段 `up_bytes` / `down_bytes` 分开存)
- `client_email` 对应 sub_main.xray_client_id 关联的那个 client

**用途**:
- 用户配额 enforce: xray 自己按 `up + down` 双向求和判断是否撞 totalBytes (本地 enforce, 不依赖 backend)
- portal "我的流量": 分开展示上传 / 下载, 总和单独显示
- subscription 月度统计: 双向求和

### 5.3 机房层 (Agent → vnstat)

**Agent 端** (每 5min):
- 调本地 `vnstat -i eth0 --json` → 拿到 day / month / total 全粒度数据
- 按 server.capacity.quota_reset_policy 算 "当前周期" 的 rx/tx 累计
- POST 到 backend `/api/agent/nic-traffic`, body 含 `{ rx_bytes_period, tx_bytes_period, sampled_at, ... }`

**Server 装机时**:
- 装 vnstat 2.x (轻量 ~1MB)
- 配置 `vnstat.conf` 加 `DailyDays 70` (保留 70 天日明细, 覆盖 BILLING_CYCLE 跨月窗口计算)

**按月聚合存储** (一台机器每周期一行, 用 UPSERT 更新):

```sql
CREATE TABLE resource_server_traffic (
  id              CHAR(32)  PRIMARY KEY,
  server_id       CHAR(32)  NOT NULL,
  period_start    DATE      NOT NULL,         -- 周期起点 (CALENDAR_MONTH=本月1号, BILLING_CYCLE=上次账单日, FIXED=安装日)
  rx_bytes        BIGINT    NOT NULL,         -- 当周期累计入站
  tx_bytes        BIGINT    NOT NULL,         -- 当周期累计出站
  total_bytes     BIGINT    NOT NULL,         -- = rx + tx (双向求和, vnstat 直接给, 不再 × 2)
  last_sampled_at DATETIME  NOT NULL,         -- 本行最近一次更新时间
  created_at      DATETIME  NOT NULL,
  updated_at      DATETIME  NOT NULL,
  UNIQUE KEY uk_server_period (server_id, period_start)
);
-- resource_ip_pool_traffic 结构同上 (server_id → ip_pool_id)
```

**Backend 端** 接收 agent 上报后处理 (单一职责: 写表, 不算 throttle):

```java
// AgentTrafficController — 接收 agent 主动 push 的 NIC 数据
@PostMapping("/api/agent/nic-traffic")
ResponseEntity<Void> receiveNicTraffic(@RequestBody NicTrafficReport report) {
  // 1. 校验 agent token
  String serverId = agentAuthService.verifyAndGetServerId(report.token);
  
  // 2. UPSERT 周期行 (server_id + period_start 唯一)
  trafficMapper.upsert(serverId, report.periodStart, report.rxBytes, report.txBytes, 
                       report.rxBytes + report.txBytes, now());
  
  // 3. 同步 capacity.used_traffic_gb (这是 SoT 副本)
  capacityMapper.updateUsedTrafficGb(serverId, (report.rxBytes + report.txBytes) / (1024L * 1024 * 1024));
  
  // 4. 完事. throttle / 告警 由别的 Job 处理 (见 §5.6 本文 / [§6.3 调度任务 03-业务核心](./03-业务核心-算法与流程.md))
  return ResponseEntity.ok().build();
}
```

> **采集逻辑搬到 agent 端**: 调 vnstat + 按 policy 算周期 → push 到 backend. Backend 只做接收 + 落库. SSH 拉模式废弃 (见 [05-Agent架构](./05-Agent架构.md)).

**周期切换自然发生**:
- 跨过 `period_start` (e.g., 6 月 1 号 0:00), `computePeriodStart()` 返回新日期 → UPSERT 命中新行 (INSERT) → used 自然归零
- 上周期的行保留作历史
- vnstat 内部累计永不重置, 我们只是滑动窗口

### 5.4 派生统计 (用户层 SQL 聚合)

```sql
-- 某 server 本月已用 (维护 resource_server.used_traffic_gb 缓存)
SELECT SUM(up_bytes + down_bytes) / 1024 / 1024 / 1024 AS gb
FROM xray_client_traffic 
WHERE server_id = ? AND sampled_at >= DATE_FORMAT(CURDATE(), '%Y-%m-01');

-- 某 IP 本月已用
SELECT SUM(up_bytes + down_bytes) / 1024 / 1024 / 1024 AS gb
FROM xray_client_traffic xct JOIN xray_client xc ON xc.id = xct.xray_client_id
WHERE xc.ip_id = ? AND xct.sampled_at >= DATE_FORMAT(CURDATE(), '%Y-%m-01');

-- 某 sub_main 本月已用 (单 client 直查, VPS 模式下没有跨 client 聚合)
SELECT SUM(up_bytes + down_bytes) / 1024 / 1024 / 1024 AS gb
FROM xray_client_traffic xct
WHERE xct.xray_client_id = (SELECT xray_client_id FROM sub_main WHERE id = ?)
  AND xct.sampled_at >= sub.started_at;
```

### 5.5 流量展示统一约定 (全系统遵守)

**核心原则**: 存储**永远分开**上下行, 计费 / 配额 / 告警**永远用双向求和** (rx + tx 或 up + down).

**存储字段** (已落实):

| 表 | 上行字段 | 下行字段 | 双向求和 |
|---|---|---|---|
| `xray_client_traffic` (用户层) | `up_bytes` | `down_bytes` | 查询时算 `up + down` |
| `resource_server_traffic` (机房层) | `tx_bytes` | `rx_bytes` | 表自带 `total_bytes = rx + tx` |
| `resource_ip_pool_traffic` (机房层) | `tx_bytes` | `rx_bytes` | 同上 |

**展示规范**:

| 场景 | 展示内容 |
|---|---|
| 用户 portal "我的实例" | `↑ 上传 12.4 GB / ↓ 下载 67.6 GB / 共 80 GB 已用 / 100 GB 配额` |
| 用户邮件 "流量将耗尽" | `节点 「🇯🇵 东京」 已用 80GB / 100GB (上 12 / 下 68), 建议尽快购买新套餐` |
| admin 资源面板 | `线路机 JP-1 当月: 入 580 GB / 出 620 GB / 共 1200 GB / 配额 2000 GB (60%)` |
| admin 告警卡片 | `SERVER_TRAFFIC_HIGH: JP-1 已用 1800/2000 GB (90%)` (告警按双向求和判定, 详情可展开看上下) |

**计费 / 告警判定** (一律用双向):
- xray totalBytes enforce: `up + down ≥ totalBytes` → 拒连
- server `throttle_state` 转 THROTTLED: `(rx + tx) / monthly_traffic_gb ≥ 0.90`
- allocator sold 约束: `(sold + new) × 2 ≤ monthly_traffic_gb` (这里的 ×2 是事前估算, 见 [§4.2 03-业务核心](./03-业务核心-算法与流程.md))

### 5.6 容量状态机 + 告警评估 (单独 Job, 不和采集混)

**throttle_state 只有两种状态** (NORMAL / THROTTLED), 100% 烧爆**不另起状态**, 用告警 severity 升级表达紧迫度.

**状态转换**:

| 当前状态 | 触发条件 (单次评估) | 新状态 | 告警动作 |
|---|---|---|---|
| NORMAL | `used / quota ≥ 0.90` 且 < 1.00 | THROTTLED | 写 `SERVER_TRAFFIC_HIGH` **WARN** |
| NORMAL | `used / quota ≥ 1.00` | THROTTLED | 写 `SERVER_TRAFFIC_HIGH` **ERROR** |
| THROTTLED | `used / quota < 0.90` | NORMAL (自动恢复, 通常发生在周期重置后) | 已有告警 `resolved_at = now`, 不另发 |
| THROTTLED | `used / quota ≥ 1.00` 且当前告警 severity 仍是 WARN | THROTTLED (不变) | **升级**当前告警 severity = ERROR |

**Job 伪代码** (单一职责: 只读 capacity, 算状态, 触发告警):

```java
// ServerCapacityStateJob — 每 5min 在 NicSampleService 之后跑
@Scheduled(fixedDelay = 5 * 60 * 1000)
void evaluateAllServers() {
  for (ServerCapacity cap : capacityMapper.findAllLive()) {
    if (cap.monthlyTrafficGb == null || cap.monthlyTrafficGb == 0) continue;  // 未配额机器跳过
    double ratio = cap.usedTrafficGb * 1.0 / cap.monthlyTrafficGb;
    
    Severity targetSeverity = ratio >= 1.00 ? ERROR : ratio >= 0.90 ? WARN : null;
    ThrottleState targetState = (targetSeverity != null) ? THROTTLED : NORMAL;
    
    // 状态转换 (只在变化时写 DB)
    if (cap.throttleState != targetState) {
      capacityMapper.updateThrottleState(cap.serverId, targetState);
    }
    
    // 告警评估 (raise / upgrade / resolve)
    Alert existing = alertMapper.findActive(SERVER_TRAFFIC_HIGH, cap.serverId);
    if (targetSeverity != null) {
      if (existing == null) {
        alertService.raise(SERVER_TRAFFIC_HIGH, cap.serverId, targetSeverity, ratio);
      } else if (existing.severity.lt(targetSeverity)) {
        alertService.upgradeSeverity(existing.id, targetSeverity);  // WARN → ERROR
      }
      // 同 severity 已存在 → 不重复 (24h 防轰炸已在 §8.3)
    } else if (existing != null) {
      alertService.resolve(existing.id);  // 自动恢复, 标记已处理
    }
  }
}
```

**关键性质**:
- 状态机**幂等**: 任意时刻调一次都得到正确结果
- 跨周期自动 NORMAL: NicSampleService 写完新周期行后, used 已归零, 这个 Job 下次跑就把 throttle 调回去
- WARN → ERROR 平滑升级, 不需要先 resolve 再新建

### 5.7 配额 enforce (无"总池", xray 原生)

VPS 模式下每 subscription 独立配额, 直接由 xray totalBytes 在用户连接时 enforce:
- 新购 → `xray_client.totalBytes = plan.traffic_gb × 1GB`
- 加购 → `xray api adu` 累加
- 续费 → `xray api adu` 重置为 `plan.traffic_gb × 1GB`
- 用尽 → xray 自动拒连

**没有跨 client 池化**, 也没有 5min "用尽暂停"调度 — xray 原生 enforce 已处理. 5min 调度只用于其他场景 (server 配额水位 / 临近到期提醒).

## 八、告警系统

### 8.1 告警类型

| type | severity | 触发 | 处理建议 |
|---|---|---|---|
| `SKU_OUT_OF_STOCK` | WARN | 某 SKU 关联 IP 池 AVAILABLE=0, 用户购买被拒 | admin 加 IP 进该 SKU 池 / 接受售罄 |
| `SERVER_TRAFFIC_HIGH` | WARN | 线路机 `used / quota ≥ 90%` (单阈值) → 自动 `throttle_state=THROTTLED`, allocator 跳过此 server | admin 决策: 上架新机 / 手动池内迁移已挂客户 / 等月底重置 |
| `RESOURCE_EXPIRING` | INFO | server / IP `expires_at - 7 days ≤ now` | admin 续费机房 |
| `NODE_FAILOVER` | ERROR | Agent 心跳 3 次没来 (>3min) 或 admin 手动触发, 池内自动切换该 server 上 sub | admin 排查故障 server |
| `AGENT_OFFLINE` | WARN | Agent 心跳超时但还没触发故障切换 (e.g., 单次 1min 没来) | admin 检查 agent 进程 |

起步阶段告警类型保持 4 种, 简化处理. 等出现"告警太杂分不清紧急程度"或"误告警太多"时再拆细 (Sprint 5+).

### 8.2 渠道

- **后台**: admin 首页"待处理告警"卡片, `WHERE resolved_at IS NULL ORDER BY severity DESC`
- **邮件**: severity ≥ WARN 触发时发邮件 admin

### 8.3 处理

- admin 点告警 → 看详情 → 处理 → 点"已处理" → 写 `resolved_at`
- 重复告警: 同 type + target 24h 内已存在未处理 → 不重复 (防轰炸)

## 九、节点切换 (v3 SKU 池内简化)

**v3 核心改变**: 所有切换发生在 **SKU 池内**, 不需要全 region 扫描. 池子小 = 决策简单.

### 9.1 故障检测 (Agent 心跳 — 分级处理)

Agent 每 1min POST 心跳到 backend, `last_heartbeat_at` 字段刷新.

`AgentHeartbeatTimeoutJob` 每 1min 扫所有 LIVE server, **分级判断**:

| 心跳超时 | 系统动作 | 用户影响 |
|---|---|---|
| 1-3min | 写 `AGENT_OFFLINE` WARN 告警 → admin 邮件 | 无 (该 server 已挂客户可能感知卡顿, 但还能连) |
| 3-5min | 标 server `temp_unhealthy=true` (不接新订阅, 不切走老客户) | allocator 跳过此 server |
| **≥ 5min** | 判定真故障, 触发池内切换 (§9.2) | 用户在 5-15min DNS 缓存过期后自动连其他 server |

**为什么分级**:
- 网络抖动 / agent 进程偶尔崩重启 → 1-3min 内会自动恢复, 不触发切换
- 给 admin 5min 介入窗口: 收到 WARN 后能登服务器排查, 救得回就救
- 真故障 (机房挂 / 永久断网) → 5min 后才切换, 避免误判搬家

**额外兜底**: 切换前再做一次主动验证 (调 agent task, 若返回成功 = agent 复活 = 取消切换)

```java
// AgentHeartbeatTimeoutJob 伪代码
void check() {
  for (ResourceServer srv : serverMapper.findLive()) {
    long elapsed = (now() - srv.lastHeartbeatAt).seconds();
    
    if (elapsed >= 300) {  // ≥ 5min, 真故障
      // 切换前最后一次验证 (避免误判)
      boolean alive = agentTaskService.pingSync(srv.id, timeout=5s);
      if (alive) {
        log.warn("Server {} 心跳超时但 ping 通, 跳过切换", srv.id);
        continue;
      }
      onServerFault(srv.id);  // §9.2 池内切换
    } else if (elapsed >= 180) {  // 3-5min, 暂时不健康
      serverMapper.markTempUnhealthy(srv.id, true);
      // allocator 自动跳过 (硬约束加 temp_unhealthy=false)
    } else if (elapsed >= 60) {  // 1-3min, 告警观察
      alertService.raiseIfAbsent(AGENT_OFFLINE, srv.id, WARN,
        "Agent 心跳超时 " + elapsed + " 秒, 观察中");
    }
    
    // 心跳恢复, resolve 告警 + 取消 temp_unhealthy
    if (elapsed < 60 && srv.tempUnhealthy) {
      serverMapper.markTempUnhealthy(srv.id, false);
      alertService.resolveIfActive(AGENT_OFFLINE, srv.id);
    }
  }
}
```

**配套字段**:

```sql
ALTER TABLE resource_server
  ADD last_heartbeat_at  DATETIME COMMENT 'agent 上次心跳时间',
  ADD temp_unhealthy     TINYINT NOT NULL DEFAULT 0 COMMENT '3-5min 心跳缺失暂时性不健康, allocator 跳过';
```

> 比之前 SSH 探活更准 (1min 心跳 vs 5min 调度), 但加了分级缓冲避免抖动误判.

### 9.2 池内故障切换流程

**核心**: 该故障 server 上的每个 sub, 在该 sub 的 SKU 池内挑一个其他 server, 切过去.

```java
void onServerFault(serverId) {
  // 1. 拉该 server 上所有活跃 sub
  List<Subscription> affectedSubs = subscriptionMapper.findActiveByServer(serverId);
  
  for (Subscription sub : affectedSubs) {
    PlanSku sku = planSkuMapper.selectById(sub.planId);
    XrayClient oldClient = xrayClientMapper.selectById(sub.xrayClientId);
    
    // 2. 在该 sub 的 SKU 池内找新 server (排除故障 server)
    ResourceServer newServer = serverMapper.findBestInSkuPoolExcluding(
        sub.planId, serverId, sku.periodDays, /* allowThrottled */ false);
    
    if (newServer == null) {
      // 池内全部 NORMAL server 都没空 (要么不够要么 THROTTLED)
      // 紧急 fallback: 允许 THROTTLED server (宁愿超 quota 也要保用户连通)
      newServer = serverMapper.findBestInSkuPoolExcluding(
          sub.planId, serverId, sku.periodDays, /* allowThrottled */ true);
      
      if (newServer == null) {
        // 真没有 server 了 (e.g., SKU 池里只有 1 个 server 且就是它挂的)
        alertService.raise(NODE_FAILOVER, ERROR, sub.id, "SKU 池无备用 server, 用户断网");
        continue;  // 用户继续断网, 必须 admin 介入扩池
      }
      
      // 用 THROTTLED server 救急, 写严重告警
      alertService.raise(NODE_FAILOVER, ERROR, sub.id, 
        "池内全 THROTTLED, 紧急 fallback 到 " + newServer.id + ", 该 server 可能撞机房 quota");
    }
    
    // 3. 算剩余流量 (xray_client_traffic 累计 / 配额)
    long remainingBytes = sku.trafficGb * 1_073_741_824L 
        - trafficMapper.sumByClient(oldClient.id, sub.startedAt);
    
    // 4. 派任务给新 server 的 agent: add user (uuid/email 不变, 配额=剩余)
    agentTaskService.enqueueSync(newServer.id, "xray_provision_user", Map.of(
      "uuid", oldClient.clientUuid, "email", oldClient.clientEmail,
      "totalBytes", remainingBytes,
      "expiryMillis", sub.expiresAt.toEpochMilli(),
      "outboundTag", "out_" + oldClient.id,
      "danteEndpoint", ipPoolMapper.selectById(oldClient.ipId).publicIp + ":1080"
    ));
    
    // 5. 调 Cloudflare API: UPDATE 该 sub 的 DNS record 指向新 server
    cloudflareService.updateARecord(sub.cfRecordId, newServer.publicIp);
    
    // 6. UPDATE DB
    transactionTemplate.executeWithoutResult(s -> {
      xrayClientMapper.updateServerId(oldClient.id, newServer.id);
      // 注意: sub_main 不动, xray_client_id 还是同一个 (只是这个 client 现在跑在新 server)
    });
    
    // 7. 派任务给老 server agent: rmu user (best effort, 失败也无所谓)
    agentTaskService.enqueueAsync(serverId, "xray_remove_user", Map.of(
      "email", oldClient.clientEmail
    ));
    
    emailService.notifyFailover(sub.memberUserId, "您的节点已自动切换");
  }
}
```

### 9.3 用户感知 (故障切换) + 缓解措施

**默认时间线**:
- T+0:    节点连不上 (server 挂了)
- T+5min: 后端心跳分级判定真故障 (1min 告警 → 3min 暂时不健康 → 5min 切换, §9.1)
- T+5min: 后端切换完成 (新 server 已部署 + Cloudflare DNS 已改)
- T+10-20min: DNS 缓存过期 → 客户端连新 IP → 恢复

**vmess 凭据 / sub_domain / 落地 IP 全部不变**, 用户客户端不需要任何操作.

**缓解 1: 缩短 Cloudflare DNS TTL (Sprint 1 落地)**

- A 记录默认 TTL = **60s** (CF Free 计划最低 60, Pro 可 30s)
- 故障切换流程加一步: 切换前 5min 把 TTL 临时调到 30s, 让客户端更频繁刷新
- 实际效果: 遵守 TTL 的客户端 1-5min 恢复, 不遵守的仍 5-15min

**缓解 2: portal 主动提示 + 邮件双通道 (Sprint 2 落地)**

```
[用户 portal "我的实例"]
  系统检测到该用户某 sub 所在 server 故障
  实例卡片显示警告:
    ⚠️ 该节点正在切换, 预计 5-15 分钟自动恢复
       若急需使用: 客户端右键节点 → "测试连接" / "重新解析 DNS"

[邮件同步发送]
  标题: 节点切换提醒
  正文: 您的 「🇯🇵 东京」 节点正在自动切换, 预计 10 分钟内恢复. 无需操作.
```

**缓解 3: 自研客户端 (Sprint 5+)**

跟你之前讨论的 "基于开源客户端重构 UI" 配套:
- 节点连失败 → 立即重新解析 DNS (绕过本地缓存)
- 节点连失败 → 自动拉取最新订阅 URL
- 节点连失败 → UI 弹窗 + 系统通知
- 故障恢复: 秒级 (客户端自己处理)

> Sprint 1-2 用缓解 1+2, 接受 5-15min 用户感知断网. Sprint 5+ 通过自研客户端把故障恢复提速到秒级.

### 9.4 流量耗尽切换 (跟故障切换共用流程)

**触发**: `SERVER_TRAFFIC_HIGH` 告警 + admin 手动决策 (Sprint 1 不自动化)

```
[admin 后台]
  收到告警 "server-jp-03 已用 90%"
  点 "迁移此 server 上的所有 sub 到 SKU 池内其他 server"
  
  后端: 遍历该 server 上的 sub, 走跟 §9.2 一样的切换流程
```

### 9.5 IP 故障 / 被墙 (admin 手动换 IP)

**核心**: 落地 IP 是用户独享, **系统不自动换**. admin 主动决策.

```
[场景] Alice 投诉 "ip-jp-1 连不上 Twitter"
[admin 后台]
  ① 找到 Alice 的 sub
  ② 点 "换 IP" → 选 SKU 池里 AVAILABLE 的同类型 IP (如 ip-jp-7)
  
[后端]
  ① 派任务给 Alice 当前 server 的 agent: 
     "修改 Alice 的 outbound, danteEndpoint 改为 ip-jp-7 的 IP:1080"
  ② agent 调 xray api 修改 outbound (秒级)
  ③ UPDATE: xray_client.ip_id 改为 ip-jp-7
  ④ ip-jp-1 状态: OCCUPIED → COOLING (7 天冷却, 防 IP 信誉污染传播)
  ⑤ ip-jp-7 状态: AVAILABLE → OCCUPIED (alice)

[Alice 感知]
  客户端继续连 jp-alice-001.nook.com (host 没变)
  出口 IP 从 ip-jp-1 → ip-jp-7 (用户去 ip.sb 检测才看得到)
  几秒内生效, 无需用户操作
```

### 9.6 SKU 容量管理 (砍掉 sold, 只看 IP)

**进货阶段** (admin 录入):
| 字段 | 含义 |
|---|---|
| `resource_server.monthly_traffic_gb` | 机房账单月配额 (NIC 视角) |
| `resource_server.lifecycle_state` | INSTALLING → READY → LIVE → RETIRED |
| `resource_server_capacity.quota_reset_policy` | CALENDAR_MONTH / BILLING_CYCLE / FIXED |

**售卖时**:
- 直接查 SKU 关联的 IP 池有几个 AVAILABLE → 这就是剩余
- 卖一份 → IP 状态从 AVAILABLE → OCCUPIED
- 全卖光 → SKU "已售罄"

**运营监控**:
- 每 5min 算 server `used / quota` (used 由 Agent push 来的 NIC 流量算)
- ≥ 90% → throttle_state=THROTTLED + 告警 admin
- admin 看告警决策: 临近月底就等, 否则点"迁移此 server 上 sub"

### 9.7 Server 退役流程

```
[admin 后台] server-jp-03 → 点 "下架退役"
弹窗: "该 server 上还有 N 个活跃 sub, 必须先迁移. 一键迁移?"
admin 确认

[后端]
  ① 对该 server 上每个 sub: 走 §9.2 切换流程, 迁到 SKU 池内其他 server
  ② 派任务给 server-jp-03 agent: "停止 xray + 清理"
  ③ UPDATE: lifecycle_state = RETIRED
  ④ DELETE FROM plan_sku_resource WHERE resource_id = serverId (从所有 SKU 池移除)

[admin 处理]
  ① 后台看到该 server 已 RETIRED
  ② SSH 进去关机或归还机房
```

### 9.8 关键认知 — 跟 v2 mesh 的对比

| 维度 | v2 mesh (N=3) | v3 池内 (1 sub 1 server) |
|---|---|---|
| 故障恢复时间 | 秒级 (客户端 fallback) | 5-15min (DNS 缓存) |
| 跨实例 enforce | 必须 | 不需要 (单 server xray totalBytes 准确) |
| MeshTopologyJob | 必须 | 不需要 |
| 客户端节点数 | 3 个/套餐 | 1 个/套餐 |
| 实施复杂度 | 高 | 中等 |
| 切换范围 | 全 region | SKU 池 |

v3 用 **DNS 切换 + 池内调度** 换 mesh 的复杂度, **5-15min 故障恢复** 是已知的代价.
