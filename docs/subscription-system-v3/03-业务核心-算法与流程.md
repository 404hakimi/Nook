# 03-业务核心: 算法与流程 (现状)

> **本文只描述已实现的业务核心**: xray client 运维 (op 编排框架 + SSH CLI 三段式) + 落地机 SOCKS5 装机 + lifecycle 流转 + 资源上下架。

## 1. 危险操作编排框架 (OpOrchestrator) ✅

Nook 把 xray client provision/revoke/rotate/sync 等**有远端副作用的危险操作**统一走 `OpOrchestrator`, 不裸调 service。

### 1.1 为什么要编排层

- **同 server 串行**: 同一台 server 上的 op 严格 FIFO 互斥 (每 server 一个队列 + `AtomicBoolean workerActive`), 避免并发 xray API 调用互相踩
- **跨 server 并行**: 不同 server 的 op 并行 (受 `workerPool` 容量约束)
- **去重**: 同 `(serverId, opType, targetId)` 活跃期最多 1 条 (`op_log.active_key` 唯一约束), 重复入队抛 `DUPLICATE_OP`
- **超时兜底**: watchdog 把超 `exec_timeout_seconds` 的执行切 `TIMED_OUT`
- **进度可视**: `OpProgressHub` (WebSocket) 实时广播进度给前端

### 1.2 调用链

```
Controller / Service
  → OpOrchestrator.submitAndWait(req, waitTimeout)   // 阻塞等终态
    或 enqueue(req)                                   // fire-and-forget
  → 入队 op_log (status=QUEUED, active_key=server|type|target)
  → 每 server FIFO worker 取出 → CAS QUEUED→RUNNING
  → OpHandler.handle(ctx)  (按 OpType 分派)
    → ClientOpExecutor.doXxx(...)   // 实际执行 (防循环依赖, 不重入 Service)
  → CAS RUNNING → DONE / FAILED; watchdog 超时 → TIMED_OUT
```

### 1.3 OpType + Handler 映射

| OpType | Handler | Executor 方法 | 语义 |
|---|---|---|---|
| `XRAY_RESTART` | RestartXrayHandler | ServerOpExecutor.doRestart | 重启 xray daemon |
| `SERVER_AUTOSTART` | SetAutostartHandler | ServerOpExecutor.doSetAutostart | 切 xray 开机自启 |

每个 OpType 在 `op_config` 有一行配置 (exec/wait 超时, enabled), admin 可在线调。

> **现状: 客户端数据操作 (开通 / 吊销 / 轮换 UUID) 一律 DB-only, 不经 op 框架、后端不直连 xray。** `XrayClientService.provision/revoke/rotateXrayClient` 直接调 `ClientOpExecutor.doProvision/doRevoke/doRotate` 只改 DB; agent 定时拉 `/api/agent/reconcile/desired` 后本地把 inbound 用户收敛到期望态 (装新 UUID 用户 / 摘旧)。op 框架 + SSH 仅保留 xray 守护进程级**命令式**操作 (重启 / 自启) —— 它们无法用声明式 reconcile 表达, 故仍由后端 SSH 下发。

## 2. xray client 三段式模型 ✅

### 2.1 共享 inbound + per-client 路由

Nook 的 xray 不是"每客户一个 inbound", 而是**一个共享 inbound + 每客户一条路由规则 + 每客户一个落地出口**:

```
                          ┌─ rule(email_A → out_A) → outbound out_A → 落地机 A socks5
共享 inbound (vmess+ws)   │
  user_A (email+uuid) ────┼─ rule(email_B → out_B) → outbound out_B → 落地机 B socks5
  user_B (email+uuid) ────┘
  ...                       (SHARED_INBOUND_TAG, 监听 shared_inbound_port=443)
```

每个 `xray_client` = 共享 inbound 里加一个 user + 一条 routing rule + 一个 socks outbound:
- **user**: `email = member_{memberId}_{ipId}` + uuid + (totalBytes / expiry / limitIp 配额)
- **rule**: `ruleTag`, 把该 email 的流量路由到 `outboundTag`
- **outbound**: `outboundTag`, socks5 指向该客户独享落地机的 `ip:socks5_port` (+ socks 用户名/密码)

> 这就是"1 客户独享 1 落地 IP"的实现: 共享入口 + 按 email 分流到各自落地出口。

### 2.2 CLI 封装

远端调用走 4 个 CLI 封装 (`xray api` 命令, SSH 执行):
- `XrayInboundCli` — addUser / removeUser
- `XrayRoutingCli` — addRule / removeRule
- `XrayOutboundCli` — addSocksOutbound / removeOutbound
- `XrayStatsCli` — readUserTraffic (流量采样)

## 3. provision (开通客户端) ✅

`ClientOpExecutor.doProvision` (`@Transactional` + 远端 saga 补偿):

```
1. 校验入参 + IP 未被占用 (clientValidator)
2. 加载 server + xray_config + capacity; 校验 client_max_count 未超
3. 占用落地节点 (landingService.occupyById CAS; 同事务回滚自动归还)
   - 校验落地机 socks5 凭据齐全 (ip + port)
4. 生成凭据: clientUuid + email=member_{memberId}_{ipId} + outboundTag + ruleTag
5. INSERT xray_client (DB; ip_id 重复 → CLIENT_IP_ALREADY_USED)
6. SSH 三段下发 (非事务, 旗标记已成功步骤):
   ① inboundCli.addUser     注册客户到共享 inbound
   ② routingCli.addRule     email → outboundTag 路由规则
   ③ outboundCli.addSocksOutbound  落地出口 → 落地机 socks5
7. 任一段失败 → 按相反顺序手动撤销 (out → rule → user) + 抛错回滚 DB
```

**事务边界**: DB (占用 + insert) 在 `@Transactional` 内; 远端 SSH 三段在事务内但用旗标做手动补偿 (SSH 副作用不可回滚, 失败时反向清理)。

**入参** (`XrayClientProvisionReqVO`): serverId (线路机) + ipId (落地机) + memberUserId + totalBytes / expiryEpochMillis / limitIp (xray 配额, 0=不限)。

> 落地机当前由 admin 在 dialog **手动选** (从 SKU 池自动选是 roadmap 的 allocator)。

## 4. revoke (吊销客户端) ✅

`ClientOpExecutor.doRevoke`:

```
1. 加载 client
2. 事务内 (executeWithoutResult):
   - 删 xray_client + 删 xray_client_traffic
   - landingService.releaseToCoolingForRevoke(ipId)  落地机 OCCUPIED → COOLING
3. 事务外 SSH 清理远端 (失败仅 warn, 留孤儿由对账兜底):
   - inboundCli.removeUser (user 已不存在则忽略)
   - routingCli.removeRule
   - outboundCli.removeOutbound
```

**设计**: DB 先提交 (业务以 DB 为准), 远端清理 best-effort。SSH 失败留下的孤儿 user/rule/outbound 由 `CLIENT_ALL_SYNC` 对账修复。落地机进 COOLING 防 IP 信誉污染 (按 `system_ip_type.cooling_minutes`)。

## 5. rotate / sync / replay ✅

### 5.1 rotate (轮换 UUID)

换新 UUID → 事务内 `updateClientUuid` + 远端 removeUser + addUser (新 uuid)。CLI 失败则回滚 DB + 标 `status=3` (待同步) 等对账。

### 5.2 sync (单客户端补推)

`doSyncOne`: 幂等重建该 client 的三段 (removeUser→addUser, removeRule→addRule, removeOutbound→addOutbound)。任一段失败标 `status=3`。用于修复单个漂移的 client。

### 5.3 replay (全量重放, CLIENT_ALL_SYNC)

`doReplayServer`: 拉该 server 所有非停 client, 逐个走 sync 路径幂等重推。批量预拉落地机凭据 (`getLandingMap`)。用于 xray 重装后整机恢复, 或对账修复。返回 `XrayClientReplayReportRespVO` (total / success / failed 列表)。

### 5.4 reset-traffic (流量清零)

`doResetTraffic`: 读远端当前累计作为新基线, 写 `xray_client_traffic.resetWithBaseline`, 后续采样从此起算。

## 6. 落地机 SOCKS5 装机 ✅

`ResourceServerLandingSocksOpsServiceImpl.installSocks5Stream` (流式 SSE):

```
1. SSH 连接落地机
2. 装机参数写回 resource_server_landing 子表 (socks5 端口/账密/路径/开关)
3. scriptCatalog.run(SOCKS5_INSTALL, vars, lineSink)  流式跑 dante 装机脚本
   - apt 装 dante-server + 写 sockd.conf + PAM + systemd drop-in + UFW
4. 探测 dante 版本 + systemd uptime
5. 回写 landing.installedAt / danteVersion / lastDanteUptime
6. 主表 lifecycle → LIVE (finalizeInstall, 用 TransactionTemplate)
```

> 落地机装机收尾会自动把 lifecycle 切到 LIVE (跟线路机不同, 线路机 xray 装机不自动切)。这块对称化 + 自动推进在 [06-Roadmap §11](./06-Roadmap-待建.md)。

dante 配置详见装机脚本 `socks5-dante.sh.tmpl`; SOCKS5 服务状态 / 日志走通用 systemd 接口 (见 [04-监控](./04-监控-流量-告警-切换.md))。

## 7. lifecycle 流转 (现状手动) ✅

`POST /admin/resource/server/transition-lifecycle?id=&state=` → `ResourceServerServiceImpl.transitionLifecycle`:

**状态机** (`ResourceServerLifecycleEnum`): `INSTALLING → READY → LIVE → RETIRED`。

**允许转移** (`ALLOWED_LIFECYCLE_TRANSITIONS`):
```
INSTALLING ↔ READY
READY      ↔ LIVE
READY → RETIRED / LIVE → RETIRED
RETIRED → LIVE (复活)
```

**校验顺序**: server 存在 → 目标 state 合法 → 同状态短路 → 在白名单 → (切 LIVE 前置) frontline.domain 非空 → `updateLifecycleState`。

> ⚠️ **现状全手动**。xray 装机完成不会自动 INSTALLING → READY; agent 心跳上来也不会自动 → LIVE。落地机 SOCKS5 装机是唯一自动切 LIVE 的路径。自动推进设计见 [06-Roadmap §11](./06-Roadmap-待建.md)。

## 8. 资源上下架 (admin) ✅ / 🟡

### 8.1 新增线路机 (frontline) ✅

```
1. admin 填: name / region / SSH 凭据 / 账面 / DNS(frontline 子表)
   → resource_server (server_type=frontline, lifecycle=INSTALLING)
2. SSH 装 agent (/admin/agent/install-agent 流式)
3. SSH 装 xray (/admin/xray/server/install-xray 流式)
   - 装机时填 client_max_count (写 resource_server_capacity)
4. 手动切 lifecycle → READY → LIVE (LIVE 前置: frontline.domain 必填)
```

### 8.2 新增落地机 (landing) ✅

```
1. admin 填: name / region / ip_type / SSH / 账面
   → resource_server (server_type=landing) + resource_server_landing (lifecycle=INSTALLING)
2. SSH 装 agent
3. self_deploy 路径: 装 dante (§6); 装机收尾 lifecycle → LIVE
4. (第三方现成 socks5: provision_mode=2, 只录入不装机)
```

### 8.3 下架 🟡

`lifecycle_state → RETIRED`。**已挂客户的自动迁移 / 退役切换未实现** (见 [06-Roadmap §9](./06-Roadmap-待建.md)), 当前下架不会自动迁走在跑客户。
