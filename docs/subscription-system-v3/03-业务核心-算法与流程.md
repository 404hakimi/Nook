## 1. 资源分配器 (Allocator)

> 状态: **❌ 整体未实施** — 依赖 `plan_sku` / `plan_sku_resource` / `sub_main` (Sprint 1+). 现状仅有底层 `XrayClientServiceImpl.provisionXrayClient` (走 op orchestrator), 缺 SKU 池绑定层. 候选 SQL 跟现状字段已对齐.

### 1.1 触发场景

| 场景 | 行为 | IP 状态变化 |
|---|---|---|
| 新购 | `allocate()`: 从 SKU 池选 1 IP + 1 server + provision + 建 sub + 创 Cloudflare DNS | AVAILABLE → OCCUPIED |
| 续费 ACTIVE 期 | 延长 `expires_at` + 派任务 agent `xray api adu` 重填 totalBytes | 不变 |
| 续费 EXPIRED 期 (14 天内) | 派任务 agent 重新部署 client + 延长 expires_at | OCCUPIED (一直保留) |
| EXPIRED → 14 天 → CANCELLED | 派任务 agent revoke client + 删 Cloudflare DNS | OCCUPIED → COOLING |
| 池内 server 切换 (故障 / 流量耗尽) | 派任务: 新 server agent add user, 旧 server agent rmu + Cloudflare API 改 DNS | 不变 (IP 用户独享, 跨 server 但 IP 不动) |
| admin 手动换 IP (§9.5 04-监控) | 派任务: agent 改 outbound 指向新 IP | 老 IP OCCUPIED → COOLING; 新 IP AVAILABLE → OCCUPIED |

### 1.2 候选过滤 (SKU 池范围)

候选从 **plan_sku_resource 关联表**里出, 不再扫全 region.

**IP 候选过滤**:
```sql
SELECT ip.id FROM resource_ip_pool ip
JOIN plan_sku_resource psr ON psr.resource_id = ip.id
JOIN resource_ip_type it ON it.id = ip.ip_type_id
WHERE psr.plan_sku_id = ?
  AND psr.resource_type = 'IP_POOL'
  AND psr.enabled = 1
  AND ip.lifecycle_state = 'LIVE'
  AND ip.status = 'AVAILABLE'
  AND ip.expires_at >= DATE_ADD(NOW(), INTERVAL ${plan.periodDays} DAY)
LIMIT 1
FOR UPDATE;  -- 行锁防并发抢占
```

**Server 候选过滤**:
```sql
SELECT srv.id FROM resource_server srv
JOIN plan_sku_resource psr ON psr.resource_id = srv.id
JOIN resource_server_capacity cap ON cap.server_id = srv.id
JOIN resource_server_billing bil ON bil.server_id = srv.id
JOIN xray_node xn ON xn.server_id = srv.id
WHERE psr.plan_sku_id = ?
  AND psr.resource_type = 'SERVER'
  AND psr.enabled = 1
  AND srv.lifecycle_state = 'LIVE'
  AND cap.throttle_state = 'NORMAL'                                     -- ❌ throttle_state 状态机未实施
  AND bil.expires_at >= DATE_ADD(NOW(), INTERVAL ${plan.periodDays} DAY)
  AND xn.touchdown_size > (SELECT COUNT(*) FROM xray_client WHERE server_id = srv.id AND status = 1)
ORDER BY (SELECT COUNT(*) FROM xray_client WHERE server_id = srv.id AND status = 1) ASC
LIMIT 1;
```

> v3 砍掉 `sold_traffic_gb × 2 ≤ monthly_traffic_gb` 约束. 容量由 IP 池决定. NIC used 监控仍保留 (机房账单视角, 见 [04-监控 §1](./04-监控-流量-告警-切换.md)).

> 代理流量过 NIC 系数 2 仍存在 (NIC used = client up+down × 2), 但只用于事后监控, 不参与事前 allocate.

### 1.3 选择策略

**起步规则**:
1. **IP 选择**: 从 SKU 池里选第一个 AVAILABLE (按 id 升序, 简单稳定)
2. **Server 选择**: 选当前客户数最少的 LIVE + NORMAL server

**为什么这么简单**:
- SKU 池本身就是 admin 精心搭配, 不需要再加复杂评分
- 客户数最少 = 流量分布天然均衡 (新订阅自动分散)
- 简单 = 容易理解 / 调试 / 测试

**未来扩展** (Sprint 5+, 真出现性能问题再加):
- 综合评分: 客户数 + NIC 流量水位 + 带宽实时占用
- 智能调度: 流量耗尽时自动迁移
- 现在不预先设计

### 1.4 购物车批量分配

```java
List<Subscription> batchAllocate(memberId, List<OrderItem> items) {
  return transactionTemplate.execute(status -> {
    List<Subscription> result = new ArrayList<>();
    for (OrderItem item : items) {
      for (int i = 0; i < item.quantity; i++) {
```

### 1.7 算法伪代码 (Sprint 1 单步分配)

CF API 调用拆事务外, 避免长事务 + 孤儿记录:

```java
Subscription allocate(memberId, planSkuId) {
  PlanSku sku = planSkuMapper.selectById(planSkuId);
  if (!sku.enabled) throw new BusinessException(SKU_DISABLED);

  // ━━━ 第 1 阶段: DB 事务 (短事务, 只 DB) ━━━
  AllocateResult ar = transactionTemplate.execute(status -> {
    // 1.1 选 IP (FOR UPDATE 行锁)
    ResourceIpPoolDO ip = ipPoolMapper.findFirstAvailableInSkuPool(planSkuId, sku.periodDays);
    if (ip == null) throw new BusinessException(SKU_OUT_OF_STOCK);

    // 1.2 选 Server (客户数最少的 LIVE + NORMAL)
    ResourceServerDO srv = serverMapper.findBestInSkuPool(planSkuId, sku.periodDays);
    if (srv == null) throw new BusinessException(NO_AVAILABLE_SERVER);

    // 1.3 占 IP (CAS)
    int updated = ipPoolMapper.tryOccupy(ip.id, "AVAILABLE", memberId);
    if (updated == 0) throw new BusinessException(IP_RACE_LOST);  // 上层重试

    // 1.4 生成身份 (sub_domain 仅 region + 随机 short id, 不含 PII)
    String uuid = UUID.randomUUID().toString();
    String subShortId = generateShortId();  // e.g., "abc123"
    String email = "sub-" + subShortId + "@nook";
    String subDomain = String.format("%s-%s.nook.com",
        sku.regionCode.toLowerCase(), subShortId);  // e.g., jp-tyo-abc123.nook.com

    // 1.5 INSERT xray_client (DB only, 远端推送由 agent task)
    XrayClient client = XrayClient.builder()
        .serverId(srv.id).ipId(ip.id).memberUserId(memberId)
        .clientUuid(uuid).clientEmail(email).status(1).build();
    xrayClientMapper.insert(client);

    // 1.6 INSERT sub_main: 业务 ACTIVE + 流程 PROVISIONING
    //     DB 已锁, 远端未就绪 → 暂不算"可用"
    //     真正可用 = status=ACTIVE AND provision_state=PROVISIONED
    Subscription sub = Subscription.builder()
        .memberUserId(memberId).planId(sku.id).xrayClientId(client.id)
        .subDomain(subDomain).cfRecordId(null)  // 待 CF 创建后回填
        .startedAt(now()).expiresAt(now() + sku.periodDays * 86400_000L)
        .status(ACTIVE).provisionState(PROVISIONING).build();
    subscriptionMapper.insert(sub);

    return new AllocateResult(sub, srv, ip);
  });

  // ━━━ 第 2 阶段: 外部调用 (事务外, 失败可回滚) ━━━
  try {
    // 2.1 调 Cloudflare 创建 DNS A 记录
    String cfRecordId = cloudflareService.createARecord(ar.sub.subDomain, ar.srv.publicIp);

    // 2.2 派任务 agent 远端 provision (异步, agent 拉到才执行)
    long totalBytes = sku.trafficGb * 1024L * 1024L * 1024L;
    agentTaskService.enqueue(ar.srv.id, "xray_provision_user", Map.of(
      "uuid", ar.sub.clientUuid, "email", ar.sub.clientEmail,
      "totalBytes", totalBytes, "expiryMillis", ar.sub.expiresAt.toEpochMilli(),
      "outboundTag", "out_" + ar.sub.xrayClientId,
      "danteEndpoint", ar.ip.ipAddress + ":" + ar.ip.socks5Port,
      "danteUser", ar.ip.socks5Username, "dantePass", ar.ip.socks5Password
    ));

    // 2.3 流程 PROVISIONING → PROVISIONED, 回填 cf_record_id
    subscriptionMapper.markProvisionedWithCfRecord(ar.sub.id, cfRecordId);
    return subscriptionMapper.selectById(ar.sub.id);

  } catch (Exception e) {
    // ━━━ 失败处理: 流程 ERROR, 业务 status 保持 ACTIVE, 等 admin ━━━
    subscriptionMapper.markProvisionError(ar.sub.id, e.getMessage());
    alertService.raise(NODE_FAILOVER, ERROR, ar.sub.id, "Provision 失败: " + e.getMessage());
    throw new BusinessException(PROVISION_FAILED, ar.sub.id);
  }
}
```

**关键设计**:
1. **CF API 调用拆事务外** — 事务只做 DB, 短事务行锁很快释放
2. **双状态字段配合** — 业务 `status=ACTIVE` 一旦付费即定; 流程 `provision_state` 走 `PROVISIONING → PROVISIONED` 或 `→ ERROR`; 真正可用 = `status=ACTIVE AND provision_state=PROVISIONED`
3. **失败标 ERROR 不自动回滚** — 避免 CF 半成品状态 + DB 回滚导致不一致, 让 admin 看到 ERROR 介入
4. **派任务给 agent 是异步** — 即使 agent 暂离线, 任务在队列等; 客户端等 DNS 生效 + agent 执行完才能用 (~30s-1min)

**Admin 处理异常 sub** (provision_state IN PROVISIONING/ERROR):
- 后台"异常订阅"列表显示, 含 provision_error / provision_retry
- admin 选择: 重试 (重新调 CF + 派任务) / 退订 (释放 IP + status → CANCELLED + 通知用户退款)

## 2. 订阅生命周期

> 状态: **❌ 未实施** — 依赖 `sub_main` 表 (Sprint 1).

### 2.1 状态机

**无宽限期**: 套餐到期当天直接 EXPIRED (立即 revoke client, IP 保留 14 天等续费).

```
[新购] → PENDING_PAYMENT
       │   (订单付款)
       ▼
       ACTIVE  ←──────────────────────┐
       │                               │  (续费 NEW 模式 — 重新部署 client)
       │   (expires_at 到)             │
       ▼                               │
       EXPIRED  ────────────────────────┘  (到期立即 revoke client; IP 保留 14 天)
       │
       │   (14 天没续 ×)
       ▼
       CANCELLED  (彻底吊销; IP status: OCCUPIED → COOLING → AVAILABLE)
```

**关键差异**:
- ACTIVE 期续费: 仅延长 `expires_at` + xray adu 重填 totalBytes (轻量, 配置都在)
- EXPIRED 期续费: **重新部署 client** (因为已 revoke) + 延长 `expires_at` + 恢复 IP 占用
- 14 天宽限给 IP 保留, 不是给服务可用性: EXPIRED 期间用户**连不上**, 但 IP 还属于他, 14 天回来续费仍能拿回原 IP

### 2.2 周期计算

- **新购**: `started_at = now`, `expires_at = now + period_days`
- **月中购买**: 15 号买 30 天 → 下月 14 号过期 (按购买日 + period_days)
- **ACTIVE 期续费**: `expires_at = max(expires_at, now) + period_days`; xray adu 重填 totalBytes
- **EXPIRED 期续费** (14 天内): 视为重启, `started_at = now`, `expires_at = now + period_days`; **必须重新部署 xray client**
- **流量包加购** (Sprint 5+): 起步不做; 后续做时单独建 `sub_addon` 表存累加历史, 不改 sub_main
- **每个 subscription 独立周期**: 同一用户 N 个套餐 N 个独立到期日; portal "实例列表" 按到期日升序

### 2.3 调度任务 (按职责分组, 单一职责)

每个 Job 只做一件事, 独立可调试 / 失败重试 / 启停.

#### 🔵 流量数据接收 (Agent push, ✅ 已实施)

| 端点 | 触发 | 职责 |
|---|---|---|
| `AgentController.applyNicTraffic` ✅ | Agent POST (5min) | 接收 NIC 流量, 覆盖写 `resource_server_capacity.rx/tx/used_traffic_bytes` |
| `AgentController.heartbeat` ✅ | Agent POST (1min) | 接收心跳, 更新 `resource_server_runtime.last_heartbeat_at` 等 |
| `AgentController.xrayTrafficSample` ✅ | Agent POST (5min) | 接收 xray stats 累计, 写 `xray_client_traffic` |
| `AgentController.tasks` (GET) ✅ | Agent 轮询 (60s) | 拉 PENDING 任务 CAS markPicked |
| `AgentController.taskResult` ✅ | Agent POST | 接收任务结果 SUCCESS/FAILED |

> 数据流方向反转完成. 后端无任何调度 Job 主动 SSH 拉数据.

#### 🟠 状态维护类 (只算状态, 不发告警)

| Job | Cron | 状态 | 职责 |
|---|---|---|---|
| `AgentHeartbeatTimeoutJob` | 每 1min | ✅ | 扫 runtime 心跳分级处理: 1min WARN / 3min `temp_unhealthy=1` / ≥5min 真触发池内切换 |
| `ServerCapacityStateJob` | 每 5min | ❌ | 读 capacity.used/quota → 算 throttle_state (NORMAL↔THROTTLED). 见 [04-监控 §3](./04-监控-流量-告警-切换.md) |
| `SubscriptionExpireJob` | 每日 00:30 | ❌ | `status=ACTIVE AND expires_at < now` → 标 EXPIRED + 派任务 revoke |
| `SubscriptionCancelJob` | 每日 02:00 | ❌ | `EXPIRED AND expires_at + 14d < now` → CANCELLED + 删 CF DNS + IP `OCCUPIED → COOLING` |
| `IpCoolingReleaseJob` | 每 30min | ❌ | `status=COOLING AND cooling_until < now` → `AVAILABLE` |
| `FixedQuotaRetireJob` | 每日 03:00 | ❌ | `quota_reset_policy=FIXED` 且 used/quota ≥ 0.95 → 标 lifecycle=RETIRED |
| `OrderTimeoutCleanupJob` | 每 5min | ❌ | order.status=PENDING AND reserve_expires_at < now → EXPIRED + 释放 (Sprint 3) |

#### 🟢 告警评估类 (只发告警, 不动状态)

| Job | Cron | 状态 | 职责 |
|---|---|---|---|
| `ServerTrafficAlertJob` | 每 5min (在 StateJob 之后) | ❌ | 触发/升级/resolve `SERVER_TRAFFIC_HIGH` 告警 |
| `ResourceExpiringAlertJob` | 每日 09:00 | ❌ | server/IP `expires_at - 7d ≤ now` → `RESOURCE_EXPIRING` 告警 |

#### 🟣 用户通知类 (只发邮件)

| Job | Cron | 状态 | 职责 |
|---|---|---|---|
| `UserQuotaNearLimitNotifyJob` | 每 5min | ❌ | 活订阅 client(up+down) ≥ 80% × totalBytes → 邮件"流量将耗尽" |
| `SubscriptionRenewalReminderJob` | 每日 09:00 | ❌ | `expires_at - 7d ≤ now` → 邮件"7 天后到期" |
| `SubscriptionExpiredNotifyJob` | 每日 01:00 | ❌ | 当天刚 EXPIRED 的 sub → 邮件"已到期, 14 天内续费可保留原 IP" |

#### 执行顺序

```
[5min 链路]  Agent push → AgentController.applyNicTraffic 写 capacity.used
    → ServerCapacityStateJob (:01)  算 throttle_state
      → ServerTrafficAlertJob (:02) 触发/升级/resolve 告警

[每日链路]
  SubscriptionExpireJob (00:30)
    → SubscriptionExpiredNotifyJob (01:00)
      → SubscriptionCancelJob (02:00, 处理上轮 14 天前的 EXPIRED)
        → FixedQuotaRetireJob (03:00)

[1min 链路]
  Agent push 心跳 → 写 resource_server_runtime
  AgentHeartbeatTimeoutJob (每 1min) 扫超时, 分级 1/3/5min
```

> 顺序通过 cron 偏移保证, 不上分布式锁. 单实例部署起步够用; 多实例 Sprint 5+ 加 Quartz 集群锁.

## 3. 订阅 URL (用户级聚合 + 每 sub 独享子域名)

> 状态: **❌ 未实施** — portal sub URL 端点未建.

### 3.1 两层 URL 结构

```
[用户层] member 聚合订阅 URL — 用户在客户端只导入这一个
  GET /api/sub/{member.sub_token}
  → 返回 Base64(该用户所有活跃 sub 的 vmess URL 列表)

[节点层] 每个 sub 一个独享子域名 — 写在 vmess URL 里
  vmess://...@jp-tyo-abc123.nook.com:443/...
                ↑
  指向 sub 当前所在 server 的 IP (Cloudflare DNS)
  切换 server 时只改 DNS, vmess URL 内容不变
```

### 3.2 Member Token (聚合 URL)

- `member_user.sub_token`: 32 char random hex, 注册时生成 ✅ DDL 已有
- 永久有效 (除非用户 portal "重置订阅 URL")
- 用户 URL: `https://nook.example.com/api/sub/abc123def456...`

### 3.3 Sub 独享子域名 (核心切换机制)

- `sub_main.sub_domain`: 创建时生成 (region 前缀 + sub_id short hash)
- Cloudflare API 自动建 DNS A 记录: `sub_domain → server.public_ip`
- 切换 server 时 UPDATE 该 record (PUT `/zones/{zone}/dns_records/{cf_record_id}`)
- TTL 60s (CF Free 最低; ISP/OS/客户端缓存让实际过期 5-15min)
- DNS 缓存过期后客户端自动连新 server, **无需用户刷订阅**

**为什么每 sub 一个子域名 (而不是每 server 一个)**:
- 切换灵活: 单 sub 切 server 不影响其他 sub
- DNS 数量可控: 1000 sub = 1000 子域名 (CF 单 zone 上限远超)
- 用户视角简单: vmess URL 永远稳定

### 3.4 服务端实现

```java
@GetMapping("/api/sub/{token}")
public ResponseEntity<String> getSubscription(@PathVariable String token) {
  MemberUser member = memberMapper.findByToken(token);
  if (member == null) return ResponseEntity.notFound().build();

  // 只返回真正可用的 sub: status=ACTIVE AND provision_state=PROVISIONED
  List<Subscription> subs = subscriptionMapper.findUsableByMember(member.id);
  List<String> nodes = subs.stream().map(this::buildVmessUrl).toList();

  String body = Base64.encode(String.join("\n", nodes).getBytes());
  return ResponseEntity.ok()
      .header("Content-Type", "text/plain")
      .header("Subscription-Userinfo", buildUserinfoHeader(member))
      .body(body);
}

private String buildVmessUrl(Subscription sub) {
  XrayClient client = xrayClientMapper.selectById(sub.xrayClientId);
  XrayNode node = xrayNodeMapper.selectByServerId(client.serverId);
  // host 用 sub_domain (而非 server 自身 domain), 这样 server 切换时 vmess URL 不变
  return vmess({
    host: sub.subDomain,           // jp-tyo-abc123.nook.com
    port: node.sharedInboundPort,  // 默认 443
    uuid: client.clientUuid,
    path: node.wsPath,             // /ws
    tls: true,
    ps: buildNodeRemark(sub)       // §3.6
  });
}
```

### 3.5 切换流程 (核心 v3 操作)

**场景**: sub_main `sub-001` 当前在 server A, 切到 server B.

```
[后端事务 — 步骤 1-5]
  ① 派任务给 server B agent: "add user (uuid+email 不变, totalBytes=剩余流量)"
  ② 等 agent 上报"成功"
  ③ 调 Cloudflare API: PUT /dns_records/{sub.cf_record_id}
       content = server B 的 public_ip
  ④ 派任务给 server A agent: "rmu user" (失败也无所谓, 客户端马上不连 A 了)
  ⑤ UPDATE xray_client SET server_id = B (sub_domain / cf_record_id 不变)

[用户感知]
  T+0:    客户端连 jp-tyo-abc123.nook.com → DNS 缓存还指向 A → 连 A 失败 (rmu 了)
  T+5-15min: DNS 缓存过期 → 重新解析 → 拿 B 的 IP → 连上 ✓

[用户不需要] 不需要刷订阅 / 改 vmess / 任何操作
```

### 3.6 节点备注规范

vmess URL 的 `ps` 字段:
```
"{flag} {region_name} | {ip_type} | {traffic_gb}GB | 到期 {MM-DD}"
例: "🇯🇵 东京 | 住宅 | 100GB | 到期 06-15"
```

### 3.7 客户端体验

- v2rayN / Clash / Sing-box 支持订阅: 填一次 member sub URL → 自动 fetch + parse
- 用户买新套餐 → 客户端 24h 自动刷新 → 节点列表多一个
- 用户某 sub 过期 → 节点自动从列表消失
- Server 切换 (含故障 / 池内迁移): vmess URL 不变, DNS 自动指向新 IP

## 4. 业务流程 (新购 / 续费 / 切换 / 退订)

### 4.1 新购 (Sprint 3 目标态: 三段式锁库存 + 支付 + 激活)

> Sprint 1: §1.7 单步分配; Sprint 3 接入 Stripe 升级为下面的三段式.

```
[用户 portal]
  浏览商品 → 加购物车 (前端 LocalStorage) → 结算

[结算 — 第 1 段: 锁库存]
  POST /api/orders { items: [...] }
  后端事务:
    a. quantity>1 的 NEW item 拆成 quantity 条独立 item
    b. 对每条 NEW item 调 reserveForOrder(planId):
       - 候选 + 评分排序
       - 抢 server 容量 + IP RESERVED
       - 失败重试下个候选; 全失败 INSUFFICIENT_RESOURCES
       - 成功 → item.reserved_server_id / reserved_ip_id 填入
    c. INSERT order_main (status=PENDING, reserve_expires_at = now + 30min)
    d. INSERT order_item × N (含 reserved 字段, CHECK 约束兜底)
  任一条失败 → 整单回滚 + 已 reserve 全释放
  返回 order_id + Stripe checkout URL

[支付 — 第 2 段]
  用户跳 Stripe → 付款 (30min 内付完都安全)
  Stripe webhook → POST /api/orders/webhook
  后端事务:
    UPDATE order_main SET status=PAID, paid_at=now
    遍历 order_item:
      对每条 NEW item 调 activateForItem():
        - IP RESERVED → OCCUPIED
        - provision xray_client + 建 sub_main (status=ACTIVE, provision_state=PROVISIONING)
        - 派任务 agent + 调 CF DNS
        - 成功后 provision_state → PROVISIONED

[通知 — 第 3 段]
  邮件: "订单已激活, 新增 N 个节点; 订阅 URL: ..."
```

**异常路径**:

| 情况 | 处理 |
|---|---|
| 结算时某 item 抢不到 | 整单失败, 已 reserve 全释放, "已售罄" |
| 进了支付页但 30min 没付 | `OrderTimeoutCleanupJob` 标 EXPIRED + 释放 |
| 付完, 激活异常 (xray 推送失败) | sub provision_state=ERROR; agent_task 队列 retry; 仍失败 admin 介入 |
| 付完, Stripe webhook 丢了 | 30min 超时释放; 实付走人工对账退款 |

### 4.2 续费 (按 sub 状态分两条路径)

#### 4.2.1 ACTIVE 期续费 — 轻量路径

```
[用户 portal]
  实例列表里 sub "3 天后到期" (status=ACTIVE)
  点"续费", 默认续 plan.period_days
  后端校验 plan_sku.enabled=1; 已下架则拒绝
  用 plan_sku.price_cny 算续费价
  生成 order(N=1 item, type=RENEW)
[支付完成]
  sub_main.expires_at = max(expires_at, now) + plan.period_days
  派任务 xray api adu 重填 client.totalBytes = plan_sku.traffic_gb × 1GB
  流量统计基线重置
[通知]
  邮件: "节点 「🇯🇵 东京 | 住宅 | 100GB」 已续期至 YYYY-MM-DD"
```

#### 4.2.2 EXPIRED 期续费 — 重新部署路径

```
[用户 portal]
  sub 已 EXPIRED (xray client 已 revoke; IP 保留 14 天)
  点"续费"
  后端校验 14 天内 (now < expires_at + 14d), 否则 sub 已 CANCELLED, 引导买新套餐
  校验 plan_sku.enabled=1
  生成 order(N=1 item, type=RENEW), 标记 "redeploy=true"
[支付完成 — 重新部署 client]
  sub_main.started_at = now, expires_at = now + period_days, status=ACTIVE
  调 clientService.provision() 完整推一遍 (adu+adr+ado 三段)
  xray_client_traffic 新统计周期开始
[通知]
  邮件: "节点 「...」 已恢复服务, 有效期至 YYYY-MM-DD"
```

> **核心区别**: ACTIVE 续费仅 `adu` 重填 totalBytes; EXPIRED 续费要把 user/rule/outbound 三段都重新部署. 续费价格两条路径都取当前 plan_sku.price_cny.

> 批量续费: 勾选 N 个 sub 一次结算; 后端按各 sub 当时状态分别走 ACTIVE / EXPIRED 路径.

### 4.5 资源上下架 (admin) ✅

**目前已实施 lifecycle 流转 + ssh 装机:**

```
[admin 新增线路机]  ✅
  1. admin UI 填: name / region / SSH (credential 子表) / billing (子表) / DNS (子表)
  2. resource_server.lifecycle_state = INSTALLING (default)
  3. admin 点 SSH 装机: ResourceServerOpsController.installXray 走流式部署 (~5-10min)
     - 装 xray + nook-agent
     - xray 装机时 admin 填 xray_node.touchdown_size (必填):
        建议值 1C1G=50-100, 2C2G=200, 4C4G=500, 8C8G=1000
  4. 等 agent 心跳到达 → admin 手动切 lifecycle → READY
  5. admin 关联到目标 SKU 资源池 (❌ plan_sku_resource 表未建)
  6. admin 切 → LIVE
     LIVE 前置: dns.domain 必填 (ResourceServerServiceImpl.transitionLifecycle 校验)

[admin 新增落地 IP]  ✅
  1. admin UI 填: ip_address / region / ip_type / SSH (provision_mode=1 才填) / 账面字段
  2. resource_ip_pool.lifecycle_state = INSTALLING / LIVE
  3. self_deploy 路径: ResourceIpSocksService 装 dante
  4. admin 切 lifecycle → LIVE

[admin 下架]
  1. server 下架: lifecycle_state → RETIRED. 已挂客户走 §4 节点切换  ❌
  2. IP 下架: lifecycle_state → RETIRED. 已挂客户走手动换 IP  ❌
```
