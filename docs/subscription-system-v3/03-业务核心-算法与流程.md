## 四、核心算法: 资源分配器

### 4.1 触发场景

**v3 起步极简**: 单步分配, 容量数 IP. 不再有 sold 概念.

| 场景 | 行为 | IP 状态变化 |
|---|---|---|
| 新购 | `allocate()`: 从 SKU 池选 1 IP + 1 server + provision + 创建 sub + 创 Cloudflare DNS | AVAILABLE → OCCUPIED |
| 续费 ACTIVE 期 | 延长 `expires_at` + 派任务 agent `xray api adu` 重填 totalBytes | 不变 |
| 续费 EXPIRED 期 (14 天内) | 派任务 agent 重新部署 client + 延长 expires_at | OCCUPIED (一直保留) |
| EXPIRED → 14 天 → CANCELLED | 派任务 agent revoke client + 删 Cloudflare DNS | OCCUPIED → AVAILABLE |
| 池内 server 切换 (故障 / 流量耗尽) | 派任务: 新 server agent add user, 旧 server agent rmu + Cloudflare API 改 DNS 指向 | 不变 (IP 是用户独享, 跨 server 但 IP 不动) |
| admin 手动换 IP (见 [§9.5 04-监控](./04-监控-流量-告警-切换.md)) | 派任务: agent 改 outbound 指向新 IP | 老 IP OCCUPIED → COOLING; 新 IP AVAILABLE → OCCUPIED |

### 4.2 候选过滤 (SKU 池范围)

候选从 **plan_sku_resource 关联表**里出, 不再扫全 region.

**IP 候选过滤**:

```sql
SELECT ip.id FROM resource_ip_pool ip
JOIN plan_sku_resource psr ON psr.resource_id = ip.id
WHERE psr.plan_sku_id = ?
  AND psr.resource_type = 'IP_POOL'
  AND psr.enabled = 1
  AND ip.lifecycle_state = 'LIVE'
  AND ip.status = 'AVAILABLE'
  AND ip.expires_at >= DATE_ADD(NOW(), INTERVAL ${plan.periodDays} DAY)
LIMIT 1
FOR UPDATE;  -- 行锁防并发抢占
```

**Server 候选过滤**:

```sql
SELECT srv.id FROM resource_server srv
JOIN plan_sku_resource psr ON psr.resource_id = srv.id
JOIN resource_server_capacity cap ON cap.server_id = srv.id
JOIN xray_node xn ON xn.server_id = srv.id
WHERE psr.plan_sku_id = ?
  AND psr.resource_type = 'SERVER'
  AND psr.enabled = 1
  AND srv.lifecycle_state = 'LIVE'
  AND cap.throttle_state = 'NORMAL'
  AND srv.expires_at >= DATE_ADD(NOW(), INTERVAL ${plan.periodDays} DAY)
  AND xn.touchdown_size > (SELECT COUNT(*) FROM xray_client WHERE server_id = srv.id AND status = 'ACTIVE')
ORDER BY (SELECT COUNT(*) FROM xray_client WHERE server_id = srv.id AND status = 'ACTIVE') ASC
LIMIT 1;
```

> **v3 砍掉**: `sold_traffic_gb × 2 ≤ monthly_traffic_gb` 约束. 容量由 IP 池决定, 不再需要事前预留. used 监控仍然保留 (机房账单视角, NIC 实采).

> **代理流量过 NIC 系数 2** 仍然存在 (NIC used = client up+down × 2), 但只用于 **事后监控** (NIC used vs quota), 不参与事前 allocate 决策.

### 4.3 选择策略 (SKU 池内挑)

**起步规则**:
1. **IP 选择**: 从 SKU 关联的 IP 池里, 选第一个 AVAILABLE 的 (按 id 升序, 简单稳定)
2. **Server 选择**: 从 SKU 关联的 Server 池里, 选当前客户数最少的 LIVE + NORMAL server

**为什么这么简单**:
- SKU 池子本身就是 admin 精心搭配的"合理资源组", 不需要再加复杂评分
- 客户数最少 = 流量分布天然均衡 (新订阅自动分散)
- 简单 = 容易理解 + 容易调试 + 容易测试

**未来扩展** (Sprint 5+, 真出现性能问题再加):
- 综合评分: 客户数 + NIC 流量水位 + 带宽实时占用
- 智能调度: 流量耗尽时自动迁移
- 现在不预先设计.

### 4.4 购物车批量分配

```java
List<Subscription> batchAllocate(memberId, List<OrderItem> items) {
  return transactionTemplate.execute(status -> {
    List<Subscription> result = new ArrayList<>();
    for (OrderItem item : items) {
      for (int i = 0; i < item.quantity; i++) {
        result.add(allocate(memberId, item.planId));
      }
    }
    return result;
    // 任一失败 → 整单回滚: 已占 IP / 已建 client / 已 reserve 容量全撤
  });
}
```

> 远端 xray API (rmu/adu) 不进事务 — 通过 `agent_task` 队列异步派发到 agent 执行, 失败由队列 retry 机制兜底 (见 [05-Agent架构 §15.7](./05-Agent架构.md)).

### 4.5 资源不足处理

候选 = 0 → **硬拒** (购物车整单退):
- 抛 `BusinessException(INSUFFICIENT_RESOURCES, region, planId)`
- 写 `alert_record(type=SKU_OUT_OF_STOCK, target=planSkuId)` (统一一种 type, message 里区分原因: 无 IP / 无可用 server)
- 邮件通知 admin
- 整单回滚, 用户钱不扣 (支付前) 或全额退款 (支付后异步)

不做软降级 (不自动改换其他区域分配).

### 4.6 定价 (无公式, 直接读 plan)

| 场景 | 定价依据 |
|---|---|
| 新购 | `unit_price_cny = plan_sku.price_cny`, `amount = unit_price × quantity` |
| 续费 | `unit_price_cny = plan_sku.price_cny` (plan 价格永不变, 等同快照); **续费前校验 `plan_sku.enabled=1`**, 已下架 SKU 拒绝续费 → 用户改买新 SKU |

无折扣 / 无周期系数 (不同周期就是不同 SKU, 年付 SKU 直接定低价). 加购流量包 Sprint 5+ 再做.

### 4.7 算法伪代码

**v3 单步分配** — 从 SKU 池选, 数 IP 判容量, **CF 调用拆事务外避免长事务 + 孤儿记录**.

```java
Subscription allocate(memberId, planSkuId) {
  PlanSku sku = planSkuMapper.selectById(planSkuId);
  if (!sku.enabled) throw new BusinessException(SKU_DISABLED);
  
  // ━━━ 第 1 阶段: DB 事务 (短事务, 只做 DB 操作) ━━━
  AllocateResult ar = transactionTemplate.execute(status -> {
    // 1.1 选 IP (FOR UPDATE 行锁防并发抢占)
    ResourceIpPool ip = ipPoolMapper.findFirstAvailableInSkuPool(planSkuId, sku.periodDays);
    if (ip == null) throw new BusinessException(SKU_OUT_OF_STOCK);
    
    // 1.2 选 Server (客户数最少的 LIVE + NORMAL)
    ResourceServer srv = serverMapper.findBestInSkuPool(planSkuId, sku.periodDays);
    if (srv == null) throw new BusinessException(NO_AVAILABLE_SERVER);
    
    // 1.3 占 IP (CAS 状态变更)
    int updated = ipPoolMapper.tryOccupy(ip.id, AVAILABLE, memberId);
    if (updated == 0) throw new BusinessException(IP_RACE_LOST);  // 上层重试
    
    // 1.4 生成身份 (sub_domain 不含用户名/PII, 仅 region + 随机 short id)
    String uuid = UUID.randomUUID().toString();
    String subShortId = generateShortId();  // e.g., "abc123"
    String email = "sub-" + subShortId + "@nook";
    String subDomain = String.format("%s-%s.nook.com",
        sku.regionCode.toLowerCase(), subShortId);  // e.g., jp-tyo-abc123.nook.com
    
    // 1.5 INSERT xray_client (DB only)
    long totalBytes = sku.trafficGb * 1024L * 1024L * 1024L;
    long expiryMillis = now() + sku.periodDays * 86400_000L;
    XrayClient client = XrayClient.builder()
        .serverId(srv.id).ipId(ip.id).clientUuid(uuid).clientEmail(email)
        .totalBytes(totalBytes).expiryMillis(expiryMillis).status(ACTIVE).build();
    xrayClientMapper.insert(client);
    
    // 1.6 INSERT sub_main: 业务 ACTIVE + 流程 PROVISIONING (双状态, v3 优化)
    //     DB 资源已锁, 但 DNS / xray 远端还没就绪 → 暂不算"可用"
    //     真正可用 = status=ACTIVE AND provision_state=PROVISIONED
    Subscription sub = Subscription.builder()
        .memberUserId(memberId).planId(sku.id).xrayClientId(client.id)
        .subDomain(subDomain).cfRecordId(null)  // 待 CF 创建后回填
        .startedAt(now()).expiresAt(expiryMillis)
        .status(ACTIVE).provisionState(PROVISIONING).build();
    subscriptionMapper.insert(sub);
    
    return new AllocateResult(sub, srv, ip);
  });
  
  // ━━━ 第 2 阶段: 外部调用 (事务外, 失败可回滚) ━━━
  try {
    // 2.1 调 Cloudflare API 创建 DNS A 记录
    String cfRecordId = cloudflareService.createARecord(ar.sub.subDomain, ar.srv.publicIp);
    
    // 2.2 派任务给 server agent (异步, agent 拉到才执行)
    agentTaskService.enqueue(ar.srv.id, "xray_provision_user", Map.of(
      "uuid", ar.sub.clientUuid, "email", ar.sub.clientEmail,
      "totalBytes", ar.sub.totalBytes, "expiryMillis", ar.sub.expiresAt.toEpochMilli(),
      "outboundTag", "out_" + ar.sub.xrayClientId,
      "danteEndpoint", ar.ip.publicIp + ":1080",
      "danteUser", ar.ip.danteUser, "dantePass", ar.ip.dantePass
    ));
    
    // 2.3 流程 PROVISIONING → PROVISIONED, 回填 cf_record_id (业务 status 已是 ACTIVE)
    subscriptionMapper.markProvisionedWithCfRecord(ar.sub.id, cfRecordId);
    return subscriptionMapper.selectById(ar.sub.id);
    
  } catch (Exception e) {
    // ━━━ 失败处理: 流程 PROVISIONING → ERROR, 业务 status 保持 ACTIVE, 等 admin 介入 ━━━
    // 不自动回滚 DB (因为 CF 可能已部分创建, 自动回滚反而引发不一致)
    // admin 在"异常订阅"列表 (provision_state IN PROVISIONING,ERROR) 看到, 手动决定: 释放资源退款 / 重试外部调用
    subscriptionMapper.markProvisionError(ar.sub.id, e.getMessage());
    alertService.raise(NODE_FAILOVER, ERROR, ar.sub.id,
      "Provision 失败: " + e.getMessage());  // 复用 NODE_FAILOVER, message 区分场景
    throw new BusinessException(PROVISION_FAILED, ar.sub.id);
  }
}
```

**关键变化** (相对之前):
1. **CF API 调用拆到事务外**: 事务只做 DB 操作 (短事务, 行锁很快释放)
2. **双状态字段配合 (v3 优化)**: 业务 `status=ACTIVE` 一旦付费即定; 流程 `provision_state` 走 `PROVISIONING → PROVISIONED` (正常) 或 `PROVISIONING → ERROR` (失败). 真正可用 = `status=ACTIVE AND provision_state=PROVISIONED`
3. **失败标 ERROR 不自动回滚**: 避免 CF 半成品状态 + DB 回滚导致不一致, 让 admin 看到 ERROR 介入
4. **派任务给 agent 是异步**: 即使 agent 暂时不在线, 任务在队列里等; 客户端等 DNS 生效后还要等 agent 执行完才能用 (~30s-1min)

**Admin 处理异常 sub** (provision_state IN PROVISIONING/ERROR):
- 后台 "异常订阅" 列表显示, 含 provision_error 错误原因 + provision_retry 已重试次数
- admin 选择:
  - 重试: 后端重新调 CF API + 派任务, 成功转 PROVISIONED
  - 退订: 释放 IP + status → CANCELLED + 通知用户退款

**关键 SQL**:


**第 1 步: 创建 order 时锁库存** (在 `OrderService.checkoutCart` 内为每个 NEW item 调一次)

```java
ReservedSlot reserveForOrder(planId) {
  Plan plan = planMapper.selectById(planId);
  
  // 1. 拉候选 (硬约束过滤)
  List<Candidate> candidates = candidateMapper.findEligible(
      plan.regionCode, plan.ipType, plan.trafficGb, plan.periodDays);
  if (candidates.isEmpty()) {
    alertService.raiseAllocateFailure(plan);
    throw new BusinessException(INSUFFICIENT_RESOURCES, plan.regionCode, plan.ipType);
  }
  
  // 2. 评分排序
  List<Candidate> sorted = candidates.stream()
      .sorted(comparing(c -> scoreOf(c)).reversed()).toList();
  
  // 3. 依次尝试抢占容量 (race 失败换下一个候选)
  for (Candidate c : sorted) {
    if (serverMapper.reserveCapacity(c.serverId, plan.trafficGb) > 0) {
      // 同时抢占 IP (软标 reserved, 跟 server 容量一起释放)
      if (ipPoolService.tryReserve(c.ipId, memberId)) {
        return new ReservedSlot(c.serverId, c.ipId);  // 库存到手, 把 (serverId, ipId) 存进 order_item
      } else {
        // IP 被并发占了, 撤掉刚才抢的 server 容量, 换下个候选
        serverMapper.releaseCapacity(c.serverId, plan.trafficGb);
      }
    }
  }
  alertService.raiseAllocateFailure(plan);
  throw new BusinessException(INSUFFICIENT_RESOURCES, plan.regionCode, plan.ipType);
}
```

**第 2 步: Stripe PAID 回调后真激活** (库存已抢, 这步只 provision)

```java
Subscription activateForItem(memberId, OrderItem item) {
  Plan plan = planMapper.selectById(item.planId);
  long totalBytes = plan.trafficGb * 1024L * 1024L * 1024L;
  long expiryMillis = now() + plan.periodDays * 86400_000L;
  
  return transactionTemplate.execute(status -> {
    // IP 从 reserved 转为 occupied (item 里存的 reservedIpId)
    ipPoolService.confirmReserve(item.reservedIpId, memberId);
    
    XrayClient client = clientService.provision(
        item.reservedServerId, item.reservedIpId, memberId,
        totalBytes, expiryMillis);  // 不限设备 (xray policy.levels 不设 limitIp)
    
    Subscription sub = Subscription.builder()
        .memberUserId(memberId)
        .planId(plan.id)
        .xrayClientId(client.id)
        .startedAt(now())
        .expiresAt(expiryMillis)
        .status(ACTIVE)
        .build();
    subscriptionMapper.insert(sub);
    return sub;
  });
}
```

**第 3 步: 30min 超时释放** (`OrderTimeoutCleanupScheduler`, 每 5min)

```java
@Scheduled(fixedDelay = 5 * 60 * 1000)
void releaseExpiredOrders() {
  List<Order> expired = orderMapper.findExpiredPending();  
  // WHERE status='PENDING' AND reserve_expires_at < now()
  
  for (Order order : expired) {
    transactionTemplate.executeWithoutResult(status -> {
      for (OrderItem item : order.items) {
        if (item.type == NEW && item.reservedServerId != null) {
          Plan plan = planMapper.selectById(item.planId);
          serverMapper.releaseCapacity(item.reservedServerId, plan.trafficGb);
          ipPoolService.releaseReserve(item.reservedIpId);
        }
        // RENEW/ADDON 没占 server 容量 (RENEW 不动 sold; ADDON 锁的 traffic_gb 也要释放)
      }
      orderMapper.updateStatus(order.id, EXPIRED);
    });
  }
}
```

## 六、订阅生命周期

### 6.1 状态机

**无宽限期**: 套餐到期当天直接 EXPIRED (立即 revoke client, IP 不动等用户回来续费).

```
[新购] → PENDING_PAYMENT
       │   (订单付款)
       ▼
       ACTIVE  ←──────────────────────┐
       │                               │  (续费 NEW 模式 — 重新部署 client)
       │   (expires_at 到)             │
       ▼                               │
       EXPIRED  ────────────────────────┘  (到期立即 revoke client; IP 保留 14 天等续费)
       │
       │   (14 天没续 ×)
       ▼
       CANCELLED  (彻底吊销; IP status: OCCUPIED → COOLING 7 天 → AVAILABLE)
```

**关键差异**:
- ACTIVE 期续费: 仅延长 `expires_at` + xray adu 重填 totalBytes (轻量, 配置都在)
- EXPIRED 期续费: **重新部署 client** (因为已 revoke) + 延长 `expires_at` + 恢复 IP 占用
- 14 天宽限给 IP 保留, 不是给服务可用性: EXPIRED 期间用户**连不上**, 但 IP 还属于他, 这 14 天回来续费仍能拿回原 IP

### 6.2 周期计算 (滚动周期, 每订阅独立)

- **新购**: `started_at = now`, `expires_at = now + period_days`
- **月中购买**: 15 号买 30 天套餐 → 下月 14 号过期 (按购买日 + period_days)
- **ACTIVE 期续费**: `expires_at = max(expires_at, now) + period_days`; xray adu 重填 totalBytes
- **EXPIRED 期续费** (14 天内): 视为重启, `started_at = now`, `expires_at = now + period_days`; **必须重新部署 xray client** (见 §10.2)
- **流量包加购** (Sprint 5+): 起步不做; 后续做时单独建 `sub_addon` 表存累加历史, 不改 sub_main
- **每个 subscription 独立周期**: 同一用户买 N 个套餐有 N 个独立到期日; portal "实例列表"按到期日升序

### 6.3 调度任务 (按职责分组, 单一职责)

每个 Job 只做一件事, 独立可调试 / 失败重试 / 启停, 互不影响.

#### 🔵 流量数据接收类 (Agent push, backend 不调度采集)

| 类型 | 触发 | 职责 |
|---|---|---|
| `AgentTrafficController` | Agent POST | 接收 agent 上报的 NIC + xray 流量数据, 写表 |
| `AgentHeartbeatController` | Agent POST (1min) | 接收心跳, 更新 server.last_heartbeat_at |

> 不再有调度 Job 去 SSH 拉数据. 数据流方向反转 (server → backend).

#### 🟠 状态维护类 (只算状态, 不发告警)

| Job | Cron | 职责 |
|---|---|---|
| `ServerCapacityStateJob` | 每 5min | 读 capacity.used / quota → 算 throttle_state (NORMAL ↔ THROTTLED), 见 [§5.6 04-监控](./04-监控-流量-告警-切换.md) |
| `AgentHeartbeatTimeoutJob` | 每 1min | 扫 `resource_server_runtime.last_heartbeat_at` 分级处理: 1min WARN / 3min `temp_unhealthy=1` / **5min** 真触发池内切换 (详见 [§9.1 04-监控](./04-监控-流量-告警-切换.md)) |
| `SubscriptionExpireJob` | 每日 00:30 | `status=ACTIVE AND expires_at < now` → 标 `status=EXPIRED, expired_at=now` + 派任务 agent revoke client. 邮件交给通知 Job |
| `SubscriptionCancelJob` | 每日 02:00 | `status=EXPIRED AND expires_at + 14 days < now` → 标 `status=CANCELLED, cancelled_at=now` + 派任务 agent revoke (兜底) + 删 Cloudflare DNS + IP `status: OCCUPIED → COOLING` (7 天冷却后由 IP 自身调度回 AVAILABLE) |
| `FixedQuotaRetireJob` | 每日 03:00 | 对 `quota_reset_policy=FIXED` 的机器, 若 `used / quota ≥ 0.95` → 标 `lifecycle_state=RETIRED` |
| `OrderTimeoutCleanupJob` | 每 5min | `order_main.status=PENDING AND reserve_expires_at < now` → 标 EXPIRED, 释放 IP (Sprint 3 接支付时生效) |

#### 🟢 告警评估类 (只发告警, 不动状态)

| Job | Cron | 职责 |
|---|---|---|
| `ServerTrafficAlertJob` | 每 5min (StateJob 之后) | 根据 capacity 数据触发 / 升级 / resolve `SERVER_TRAFFIC_HIGH` 告警 (90% WARN / 100% ERROR). 见 [§5.6 04-监控](./04-监控-流量-告警-切换.md) |
| `ResourceExpiringAlertJob` | 每日 09:00 | server / IP `expires_at - 7 days ≤ now` → 写 `RESOURCE_EXPIRING` 告警 |

#### 🟣 用户通知类 (只发邮件)

| Job | Cron | 职责 |
|---|---|---|
| `UserQuotaNearLimitNotifyJob` | 每 5min | 扫活订阅: client(up+down) ≥ 80% × totalBytes → 邮件"流量将耗尽" (含上下行明细, 见 [§5.5 04-监控](./04-监控-流量-告警-切换.md)) |
| `SubscriptionRenewalReminderJob` | 每日 09:00 | `sub_main.expires_at - 7 days ≤ now` → 邮件"7 天后到期" |
| `SubscriptionExpiredNotifyJob` | 每日 01:00 (在 ExpireJob 之后) | 当天刚 EXPIRED 的 sub → 邮件"已到期, 14 天内续费可保留原 IP" |

#### 执行顺序

```
[5min 链路] (NIC 数据由 Agent 持续 push, 不依赖 backend 调度采集)
  Agent push → AgentTrafficController 写 capacity.used_traffic_gb
    → ServerCapacityStateJob (:01)  算 throttle_state
      → ServerTrafficAlertJob (:02) 触发 / 升级 / resolve 告警

[每日链路]
  SubscriptionExpireJob (00:30)
    → SubscriptionExpiredNotifyJob (01:00)
      → SubscriptionCancelJob (02:00, 处理上轮 14 天前的 EXPIRED)
        → FixedQuotaRetireJob (03:00)

[1min 链路]
  Agent push 心跳 → 写 resource_server_runtime
  AgentHeartbeatTimeoutJob (每 1min) 扫超时, 分级处理 1/3/5min
```

> **顺序通过 cron 偏移保证**, 不上分布式锁. 单实例部署起步够用; 多实例后续 Sprint 5+ 加 Quartz 集群锁.

## 七、订阅 URL (用户级聚合 + 每 sub 独享子域名)

### 7.1 两层 URL 结构 (v3 关键设计)

```
[用户层] member 聚合订阅 URL — 用户在客户端只导入这一个
  GET /api/sub/{member.sub_token}
  → 返回 Base64(该用户所有活跃 sub 的 vmess URL 列表)

[节点层] 每个 sub 一个独享子域名 — 写在 vmess URL 里
  vmess://...@jp-tyo-abc123.nook.com:443/...
                ↑
  这个子域名指向 sub 当前所在 server 的 IP (Cloudflare DNS)
  切换 server 时只改 DNS, vmess URL 内容不变
```

### 7.2 Member Token (聚合 URL)

- `member_user.sub_token`: 32 char random hex, 注册时生成
- 永久有效 (除非用户 portal "重置订阅 URL")
- 用户 URL: `https://nook.example.com/api/sub/abc123def456...`

### 7.3 Sub 独享子域名 (核心切换机制)

- `sub_main.sub_domain`: 创建时生成, e.g., `jp-tyo-abc123.nook.com` (region 前缀 + sub_id short hash)
- 通过 Cloudflare API 自动创建 DNS A 记录: `sub_domain → server.public_ip`
- 切换 server 时, 调 CF API UPDATE 该 record (PUT /zones/{zone}/dns_records/{cf_record_id})
- TTL 60s (允许设置但实际 ISP / OS / 客户端缓存控制力有限, 实际过期 5-15min)
- DNS 缓存过期后客户端自动连新 server, **无需用户刷订阅**

**为什么每 sub 一个子域名 (而不是每 server 一个)**:
- 切换灵活: 单个 sub 切 server 不影响其他 sub
- DNS 数量可控: 1000 sub = 1000 子域名 (CF 单 zone 上限远超此数)
- 用户视角简单: vmess URL 永远稳定, 不需要刷新

### 7.4 服务端实现

```java
@GetMapping("/api/sub/{token}")
public ResponseEntity<String> getSubscription(@PathVariable String token) {
  MemberUser member = memberMapper.findByToken(token);
  if (member == null) return ResponseEntity.notFound().build();
  
  // 只返回真正可用的 sub: status=ACTIVE AND provision_state=PROVISIONED
  // (EXPIRED 已 revoke 不返回; ACTIVE 但 provision_state=PROVISIONING/ERROR 远端未就绪也不返回)
  List<Subscription> subs = subscriptionMapper.findUsableByMember(member.id);
  List<String> nodes = subs.stream()
      .map(this::buildVmessUrl)
      .toList();
  
  String body = Base64.encode(String.join("\n", nodes).getBytes());
  return ResponseEntity.ok()
      .header("Content-Type", "text/plain")
      .header("Subscription-Userinfo", buildUserinfoHeader(member))  // 流量信息, 客户端可识别
      .body(body);
}

private String buildVmessUrl(Subscription sub) {
  XrayClient client = xrayClientMapper.selectById(sub.xrayClientId);
  // host 用 sub_domain (而非 server.domain), 这样 server 切换时 vmess URL 不变
  return vmess({
    host: sub.subDomain,           // jp-tyo-abc123.nook.com
    port: 443,
    uuid: client.clientUuid,
    path: "/ws",
    tls: true,
    ps: buildNodeRemark(sub)       // 备注 (§7.6)
  });
}
```

### 7.5 切换流程 (核心 v3 操作)

**场景**: sub_main `sub-001` 当前在 server A, 要切换到 server B (流量耗尽 / 故障 / admin 操作).

```
[后端事务]
  ① 派任务给 server B 的 agent: "add user (uuid+email 不变, totalBytes = 剩余流量)"
  ② 等 agent 上报"成功"
  ③ 调 Cloudflare API: PUT /dns_records/{sub.cf_record_id}
       content = server B 的 public_ip
       (DNS 记录 ID 已存在 sub_main.cf_record_id, 直接 UPDATE)
  ④ 派任务给 server A 的 agent: "rmu user"
       (失败也无所谓, 反正客户端马上就不连 A 了)
  ⑤ UPDATE xray_client SET server_id = B (sub_domain / cf_record_id 不变)

[用户感知]
  T+0:    客户端连 jp-tyo-abc123.nook.com → DNS 缓存还指向 A → 连 A 失败 (因 rmu 了)
  T+5-15min: DNS 缓存过期 → 重新解析 → 拿到 B 的 IP → 连上 ✓
  
[用户不需要]
  - 不需要刷订阅
  - 不需要改 vmess 配置
  - 不需要任何操作
```

### 7.6 节点备注规范

vmess URL 的 `ps` 字段:
```
"{flag} {region_name} | {ip_type} | {traffic_gb}GB | 到期 {MM-DD}"
例: "🇯🇵 东京 | 住宅 | 100GB | 到期 06-15"
```

用户在客户端能直接区分多个套餐对应的节点 / 到期日.

### 7.7 客户端体验

- v2rayN / Clash / Sing-box 支持订阅: 填一次 member sub URL → 自动 fetch + parse
- 用户买新套餐 → 客户端 24h 自动刷新 → 节点列表多一个
- 用户某 sub 过期 → 节点自动从列表消失
- **Server 切换 (含故障 / 池内迁移)**: vmess URL 不变, DNS 自动指向新 IP, 5-15min DNS 缓存过期后客户端自动恢复

## 十、业务流程 (新购 / 续费 / 切换 / 退订)

### 10.1 新购 (Sprint 3 目标态: 锁库存 + 支付 + 激活 三段式)

> **Sprint 阶段说明**: Sprint 1 用 §4.7 的**单步分配** (用户买了立即 provision, 简单粗暴, 用户少没并发问题). 接入 Stripe 支付后 (Sprint 3) 才升级为下面的**三段式** (预占 → 付款 → 激活). 本节描述的是 Sprint 3+ 的目标态.

```
[用户 portal]
  1. 浏览商品列表, 筛选 region / ip_type / 周期
  2. 加购物车 (前端 LocalStorage, 后端无状态):
     - 1× JP-TYO 住宅 100GB (¥80)
     - 2× JP-TYO 住宅 500GB (¥320 × 2 = ¥640)
     - 1× US-LAX ISP 200GB (¥120)
     共 4 IP, 总价 ¥840

[结算 — 第 1 段: 锁库存]
  3. 点"结算" → POST /api/orders { items: [...] }
  4. 后端事务:
     a. 把 quantity > 1 的 NEW item 拆成多条 item, 共 4 条 NEW item
     b. 对每条 NEW item 调 reserveForOrder(planId):
        - 拉候选 + 评分排序
        - 依次抢 server 容量 + IP 软预占 (RESERVED 状态)
        - 失败重试下一个候选; 全失败抛 INSUFFICIENT_RESOURCES
        - 成功 → item.reserved_server_id / reserved_ip_id 填入
     c. INSERT order_main (status=PENDING, reserve_expires_at = now + 30min)
     d. INSERT order_item × 4 (含 reserved 字段)
     任一条 item reserve 失败 → 整单回滚, 已 reserve 的全释放, 提示用户"售罄"
  5. 返回 order_id + Stripe checkout URL

[支付 — 第 2 段]
  6. 用户跳 Stripe → 付款 (库存已锁, 30min 内付完都安全)
  7. Stripe webhook → POST /api/orders/webhook
  8. 后端事务:
     UPDATE order_main SET status=PAID, paid_at=now
     遍历 order_item:
       对每条 NEW item 调 activateForItem():
         - IP 状态 RESERVED → OCCUPIED
         - provision xray_client + 建 sub_main (status=ACTIVE, provision_state=PROVISIONING)
         - 派任务到 agent_task 队列 (xray_provision_user) + 调 CF API 创建 DNS
         - 成功后 provision_state → PROVISIONED

[通知 — 第 3 段]
  9. 邮件: "订单已激活, 您新增 4 个节点; 订阅 URL: https://nook.example.com/api/sub/{sub_token}"

[使用]
  10. 客户端定时刷新 → 多 4 个节点
```

**异常路径**:

| 情况 | 处理 |
|---|---|
| 结算时某 item 抢不到 | 整单失败, 已 reserve 全释放, 前端提示"已售罄" |
| 用户进了支付页但 30min 没付 | `OrderTimeoutCleanupJob` 标 EXPIRED + 释放所有 reserved |
| 用户付完, 但激活异常 (xray 推送失败等) | DB 已建 sub (provision_state=ERROR), 由 `agent_task` 队列 retry 机制兜底; 仍失败则在 admin "异常订阅"列表手动处理 |
| 用户付完, 但 Stripe webhook 丢了 | 30min 后调度释放; 用户实付了钱 — 极端情况, 走人工对账退款 |

### 10.2 续费 (按 sub 状态分两条路径)

#### 10.2.1 ACTIVE 期续费 (套餐还活着 — 轻量路径)

```
[用户 portal]
  1. 实例列表里某 sub "3 天后到期" (sub.status=ACTIVE)
  2. 点"续费", 默认续 plan.period_days
  3. 后端校验 plan_sku.enabled = 1; 已下架则拒绝
  4. 用 plan_sku.price_cny 算续费价 (跟随当前价, 不防涨价)
  5. 生成 order(N=1 item, type=RENEW)
[支付完成]
  6. sub_main.expires_at = max(expires_at, now) + plan.period_days
  7. 调 xray api adu 重置 client.totalBytes = plan_sku.traffic_gb × 1GB (JOIN 拿)
  8. 流量统计基线重置 (xray_client_traffic 加 reset marker)
[通知]
  9. 邮件: "节点 「🇯🇵 东京 | 住宅 | 100GB」 已续期至 YYYY-MM-DD"
```

#### 10.2.2 EXPIRED 期续费 (套餐已死 — 重新部署路径)

```
[用户 portal]
  1. 用户某 sub 已 EXPIRED (xray client 已 revoke; IP 仍保留 14 天)
  2. 点"续费", 后端检测 sub.status=EXPIRED
  3. 校验 14 天内 (now < expires_at + 14 days), 否则 sub 已 CANCELLED, 引导买新套餐
  4. 校验 plan_sku.enabled = 1
  5. 生成 order(N=1 item, type=RENEW), 标记 "redeploy=true"
[支付完成 — 重新部署 client]
  6. sub_main.started_at = now, expires_at = now + plan.period_days, status = ACTIVE
  7. 调 clientService.provision() 完整推送一遍 (adu + adr + ado 三段)
     — 这里复用 ACTIVE 期的 provision 逻辑, 不需要新算法
  8. xray_client_traffic 新统计周期开始
[通知]
  9. 邮件: "节点 「🇯🇵 东京 | 住宅 | 100GB」 已恢复服务, 有效期至 YYYY-MM-DD"
```

> **核心区别**: ACTIVE 续费仅 `adu` 重填 totalBytes; EXPIRED 续费要把 user/rule/outbound 三段都重新部署 (因为已 revoke). 续费价格两条路径都取当前 plan_sku.price_cny.

> 批量续费: 用户勾选 N 个 sub 一次性结算, N 个 RENEW item 走购物车流程; 后端按各 sub 当时状态分别走 10.2.1 或 10.2.2.

### 10.3 流量包加购 — **Sprint 5+ 才做**

起步阶段用户流量用完只能买新套餐. 等运营数据出来再决定加购模型 + 引入 `plan_traffic_addon` / `sub_addon` 两张表.

### 10.4 自动切换 (§9.1-9.4 故障) / admin 手动迁移 (§9.4) / 手动换 IP (§9.5) — 用户被动 (详见 [04-监控-流量-告警-切换.md](./04-监控-流量-告警-切换.md))

### 10.5 admin 手动换 IP (详见 [04-监控 §9.5](./04-监控-流量-告警-切换.md))

IP 被墙 / 质量下降时 admin 后台一键触发. 落地 IP 是用户独享资产, 系统不会自动换, 全程 admin 决策.

### 10.6 资源上下架 (admin)

```
[admin 新增线路机]
  1. 后台填 SSH 凭据 + bandwidth_mbps / monthly_traffic_gb / billing_cycle_day / expires_at
  2. resource_server.lifecycle_state = INSTALLING
  3. ServerProvisioner 后台跑装 xray + nook-agent (5-10min)
     - xray 装机时 admin 填 **`xray_node.touchdown_size` (必填)**: 该 server 上能挂多少客户
     - 建议值: 1C1G=50-100, 2C2G=200, 4C4G=500, 8C8G=1000 (admin UI 给出提示)
     - 防止单 server 过载: 1C1G 跑 300 用户时 CPU 必爆, allocator 会自动跳过该 server
  4. 装完自动 → READY (agent 心跳到达后)
  5. admin 在 "待上线" 列表看到, 把该 server 关联到目标 SKU 资源池
  6. admin 点 "上线" → lifecycle_state = LIVE
     (此时 allocator 才把这台机器纳入候选池)

[admin 新增落地 IP]
  1. 后台填 SSH 凭据 + region + ip_type + cost / expires_at
  2. resource_ip_pool.lifecycle_state = INSTALLING
  3. 后台跑装 dante (Socks5Controller)
  4. 装完自动 → READY
  5. admin 点 "上线" → lifecycle_state = LIVE (allocator 可分配)

[admin 下架]
  1. server 下架: lifecycle_state = RETIRED. 已挂客户走 [§9.2 池内切换 04-监控](./04-监控-流量-告警-切换.md)
  2. IP 下架: lifecycle_state = RETIRED. 已挂客户走 [§9.5 手动换 IP 04-监控](./04-监控-流量-告警-切换.md)
```

> v3 装机时还要填: `domain` (LIVE 前置), `cf_zone_id` (Cloudflare zone), 不填不能上线.

### 10.7 SKU 上架 / 下架 (admin 关联资源池)

```
[admin 新增 SKU]
  1. 后台填: name, region, ip_type, traffic_gb, period_days, price_cny
  2. 保存 plan_sku, enabled=0 (默认下架)
  3. "绑定资源" 弹窗:
       - 选 server (从 LIVE 状态的 server 列表勾)
       - 选 IP (从 LIVE+AVAILABLE 状态的 IP 列表勾, IP 类型必须匹配 SKU.ip_type)
  4. INSERT plan_sku_resource × N (按勾选)
  5. 系统提示: "该 SKU 容量 = 关联 IP 数 = X"
  6. admin 点 "上架" → enabled=1, 商品页可见

[admin 加 IP 进已上架 SKU 池]
  1. 后台 SKU 编辑 → "新增 IP 到此池"
  2. 选 LIVE+AVAILABLE 的 IP
  3. INSERT plan_sku_resource
  4. SKU 容量自动 +1

[admin 下架 SKU]
  1. plan_sku.enabled=0
  2. 商品页不再显示
  3. 老用户**不能续费** (§4.6 规则: 续费校验 enabled=1)
  4. 老用户使用不受影响, 到期自然 EXPIRED

[admin 永久删除 SKU]
  起步阶段**不支持** (避免数据完整性问题). 下架即可.
```
