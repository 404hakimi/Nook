## 1. 问题与现状

一次全后端审计发现共 9 个业务状态字段, 仅 1 个有流转治理 (合法流转表 + 校验), 普遍缺失.

> **范围**: 本方案限定在**核心业务状态流转**——服务器装机生命周期、订阅、凭证、额度. 会员、后台用户的账号启停 (正常 / 禁用) 属支撑性账号管理, 逻辑简单, **不在本次范围**, 下表及后文不再涉及.

下表为范围内字段:

| 字段 | 模块 | 流转表 | 守卫 | 变更入口 | 归类 |
|---|---|:--:|:--:|---|---|
| `ResourceServer.lifecycleState` (服务器装机生命周期) | node | ✅ | ✅ | 2 个运维流转接口走校验 + 装机回写**绕过** | 生命周期 |
| `TradeSubscription.status` (订阅) | trade | ❌ | ❌ | 开通 / 退订 + 定时任务(到期/停服/恢复), 全部直接改库 | 生命周期 |
| `TradeCert.certStatus` (凭证) | trade | ❌ | ❌ | 签发 / 改状态 / 吊销 + 定时任务, 两套写法 | 生命周期 |
| `TradeQuota.status` (额度) | trade | ❌ | ❌ | 用量计算驱动; 枚举有 2 个未用值 | 派生 |
| `ResourceServerTraffic.throttleState` (限流) | node | ❌ | ❌ | 流量对账 3 处直写 | 派生 |
| `TradePlan.enabled` (套餐上下架) | trade | ❌ | ❌ | 上 / 下架二态 | 开关 |
| `XrayClientStatus` | node | — | — | 无写入点 (死枚举) | **删除** |

**核心风险 (比"没有流转表"更要紧)**: 订阅状态变更需**连带**改名下凭证状态 + 释放落地机 + 处理额度. 这套联动现**手写散落在 4 处** (定时任务三分支 + 退订), 每处自己配对. 任一入口改了订阅状态却漏改凭证, 就**静默不一致** (订阅显示已取消, 凭证还在跑).

---

## 2. 目标与非目标

**目标**
- 把核心生命周期状态的【流转合法性校验 + 状态写入 + 联动副作用编排 + 事件/审计】收口到单一入口, 杜绝散落与漏配.
- 所有改状态的调用方 (运维接口 / 定时任务 / 跨模块) 统一调该入口, 不再各自直接改库.
- 两模块风格对齐 (trade 补齐 node 已有的枚举流转表模式).

**非目标 (避免过度设计 — 个位数规模)**
- ❌ 不引入通用状态机框架 / 状态模式类爆炸.
- ❌ 不抽跨模块统一流转基类; node 与 trade 各自一套薄封装.
- ❌ 不给"派生态"和"二态开关"上流转表 (见 §3).

---

## 3. 状态分类: 只治理"真·生命周期"

| 类别 | 字段 | 是否治理 | 理由 |
|---|---|:--:|---|
| **真·生命周期** | 服务器装机生命周期、订阅、凭证 | ✅ | 有业务流转语义, 人 / 定时任务推动, 误跃迁有害 |
| **派生 / 对账态** | 限流、额度 | ❌ | 由数据计算驱动, 非人工流转; 强加流转表反而别扭 |
| **二态开关** | 套餐上下架 | ❌ | 仅上 / 下架互转, 流转表零收益 (会员 / 后台用户账号启停已划出本方案范围) |

## 4. 设计

### 4.1 包与类 (流转代码独立成包, 不与 service CRUD 混)

```
com.nook.biz.node.lifecycle
  ├─ ServerLifecycleManager        # 服务器生命周期流转执行 + 编排
  └─ ServerLifecycleValidator      # (从 validator 包迁入) 流转表校验 + 上线/停用前置守卫

com.nook.biz.trade.lifecycle
  ├─ SubscriptionLifecycleManager  # 订阅状态流转 + 凭证/落地机/额度联动编排
  └─ CertLifecycleManager          # 凭证状态流转 (被 SubscriptionLifecycleManager 调度)
```

- 包名 `lifecycle`: 与现有字段 / 方法 / 枚举的"生命周期"命名一脉相承.
- `XxxLifecycleManager`: 流转执行入口. **CRUD 仍留在 service**, service 需要改状态时调它, 不自己改库.
- node 现有 `ServerLifecycleValidator` (纯流转校验) 迁入本包; CRUD 校验类 (`ResourceServerValidator` 等) 留 `validator` 包.

### 4.2 流转方法形态

```java
// 统一签名: 当前实体 + 目标态 + 变更原因 (审计/事件用)
void transition(SomeDO entity, SomeStatusEnum target, TransitionReason reason);
// 内部固定流程: 幂等短路 → 流转表校验 → 前置守卫 → 写状态 → 编排联动 → 发事件/审计
```

调用方一律走 `xxxLifecycleManager.transition(...)`, 替换现有的"直接改库不经校验".

### 4.3 流转表 (写在枚举内, 复用 node 已有写法)

> 表头 / 单元格用中文业务态; 对应英文枚举值见各枚举类定义, 不入注释.

**订阅 `TradeSubscriptionStatusEnum`** (状态值: 生效中 / 已暂停 / 已过期 / 已取消):

| 从 \ 到 | 生效中 | 已暂停 | 已过期 | 已取消 |
|---|:--:|:--:|:--:|:--:|
| 生效中 | — | ✅停服 | ✅到期 | ✅退订 |
| 已暂停 | ✅恢复 | — | ✅到期 | ✅退订 |
| 已过期 | ✅续费\* | | — | |
| 已取消 | (终态) | | | — |

> \* 已过期 → 生效中 为续费复活边, **预留** (决策 3): 流转表列入但暂不接线, 做续费功能时启用.

**凭证 `TradeCertStatusEnum`** (状态值: 应运行 / 应停 / 应移除):

| 从 \ 到 | 应运行 | 应停 | 应移除 |
|---|:--:|:--:|:--:|
| 应运行 | — | ✅ | ✅ |
| 应停 | ✅ | — | ✅ |
| 应移除 | (终态) | | — |

**服务器 `ResourceServerLifecycleEnum`** (状态值: 装机中 / 待上线 / 运行中 / 已退役): 流转表不变 (装机中→待上线、待上线→{装机中, 运行中, 已退役}、运行中→{待上线, 已退役}、已退役→运行中). 装机回写改为置 **待上线**, 与流转表一致; 不增"装机中→运行中"直升边 (决策 1).

### 4.4 联动编排矩阵 (订阅流转 → 连带动作, 收口进 SubscriptionLifecycleManager)

| 订阅流转 | 凭证动作 | 落地机 | 触发方 |
|---|---|---|---|
| → 已暂停 (流量耗尽) | 应运行 → 应停 | 保留占用 | 定时任务 |
| → 生效中 (周期重置) | 应停 → 应运行 | 对账装回 | 定时任务 |
| → 已过期 (到期) | → 应移除 + 清分配 | 释放 | 定时任务 |
| → 已取消 (退订) | → 应移除 + 清分配 + 发释放事件 | 释放 | 运维 |

现在这 4 行散在定时任务与退订里; 收口后调用方只调 `transition(订阅, 目标态, 原因)`, 联动由 Manager 内部按本矩阵原子执行.

## 5. 改造分级

| 级别 | 项 | 说明 |
|---|---|---|
| **P0** | ① `finalizeInstall` 装机回写改置 待上线 | 合法流转; 上线由运维手动 待上线→运行中 (决策 1) |
| | ② 订阅↔凭证联动收口到 `SubscriptionLifecycleManager` | 消除漏配不一致 (最高价值) |
| **P1** | ③ `TradeSubscriptionStatusEnum` 补流转表 + 校验方法 | 对齐 node, 几行 |
| | ④ node 两处 `transitionLifecycle` 去重 | 线路机/落地机合并为公共流转 + 按类型分派守卫钩子, 迁入 `ServerLifecycleManager` |
| | ⑤ 落地机 上线守卫: 经检查站点拨测 SOCKS5 可用 | 复用现有检查站点配置 + 默认值 (决策 2) |
| **P2** | ⑥ `CertLifecycleManager` 独立 / 凭证·额度流转表 | 按需 |
| | ⑦ 删除 `XrayClientStatus` 死枚举 | 已确认无写入点 (决策 4) |
| | — | 套餐上下架二态开关 **保持现状, 不上流转表** |

## 6. 决策记录 (已定)

1. **落地机装机终态 = 待上线**. `finalizeInstall` 装机成功置 装机中→待上线 (合法流转); 上线由运维手动 待上线→运行中. 不增"装机中→运行中"直升边, 保留"待上线"人工关卡.
2. **落地机 上线前置守卫 = SOCKS5 可用性拨测**. 上线前经"检查站点"拨测 SOCKS5 实际出网可用 (对称线路机上线查域名). 检查站点管理端已可手动填 + 带默认值 (现有能力), 守卫复用之.
3. **续费复活 (已过期→生效中) 预留**. 流转表先列入合法边, 当前无续费入口暂不接线; 后续做续费功能直接复用.
4. **`XrayClientStatus` 死枚举, 删除**. 服务端无任何写入点, 确认废弃.

## 7. 风险与回滚

- 改造集中在状态写入路径, 不动数据库表结构; 分级独立, 可只做 P0.
- 收口后行为应与现状等价 (除决策 1 刻意修正的不一致); 每级改完按后端规范 build + 重启验证.
