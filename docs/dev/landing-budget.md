# 落地机流量预算 方案

> 打磨稿(未定)。解决「落地机(用户独占出口)的物理流量是花钱买的有限预算, 但售卖、续费、运行全程都不感知它, 超了只能听机房处置」的问题。
> **独占 IP 的产品价值 = 长期稳定**(账号风控 / 白名单 / 登录态常绑死 IP), 所以**不为省流量自动换 IP**(原自动换机 LandingFailoverJob 已撤, 见 §4 闸三 / rollout-plan A4); 预算靠「填报留冗余 + 不够不卖」预防, 真到顶靠停售 + 运营补量兜底。
> 账的正确性(不串号、不丢量)已由 [traffic-model.md](traffic-model.md) 解决, 本文只管「预算够不够、不够怎么办」。
> 入口侧(线路机)的故障切换见 [frontline-failover.md](frontline-failover.md); 两侧正交、可独立落地, 计量锚点(落地机计量)谁都不动。

---

## 0. 一页白话(只记这段就够)

落地机的月流量是花钱买的, 用光了机房就限速 / 断网 / 收超额费。本方案三件事:

1. **闸一·把账算对**(字段化阈值 + 填写规范): 机器流量按「进 + 出」双向计, 用户用 1G 机器要走 2G。给机器填月流量**照抄厂商面板数字**(厂商明示单向计费的 ×2); 冗余比例配在配额上: **`usable_percent`(管理端默认 90, 可按机器调), 限流阈值 = 配额 × 该比例**, 默认 10% 冗余吃掉换机反应延迟 / agent 与装机流量 / 统计口径误差。
2. **闸二·不够就别卖**(代码): 卖套餐 / 显示库存时, 看机器**这个月还剩多少**; 剩的不够撑一个套餐就不卖, 下月流量重置自动恢复可卖。
3. **闸三·用光不换 IP, 告警靠运营**(策略, 基本无新代码): 机器到顶 → 自动停止接新用户 + 停售这台机(闸二已做), **存量用户保留原机原 IP 继续用**(独占 IP 不动是产品承诺); 短时小超支由闸一冗余吸收, 运营收到告警后补流量包 / 调高该机比例 / 协调换套餐 —— 绝不自动换用户 IP。

口径默认**双向**(§6 #1 已定) → **500G 机器最多配 ~225G 套餐; 要卖 300G 套餐, 得采购 ≥670G(700G 档)的机器**(算术见 §2.1)。

## 1. 名词(先对齐)

- **物理预算**: 落地机向机房买的月流量(例 500G/月), 按机房重置日翻篇。花钱买的硬资源。
- **机器口径(rx+tx)**: 本机网卡进站 + 出站。`resource_server_traffic.used_bytes` 与 `total_gb` 限流判定用的就是它。
- **用户口径(up+down)**: 用户实际上下行(代理负载), 扣套餐用的口径。
- **口径系数**: 用户 1G = 机器 2G(转发要收一次再发一次), 见 §2.1。
- **承载力**: 一台落地机一个预算周期内能服务的用户流量 = 机器口径预算 ÷ 2。

---

## 2. 为什么会超(三个事实)

### 2.1 口径放大 ×2(全链路算清楚)

以「用户上行 50G + 下行 50G(用户口径 100G)」为例:

```
下行 50G:  网站 ──50G──> 落地机 ──50G──> 线路机 ──50G──> 客户端
上行 50G:  网站 <──50G── 落地机 <──50G── 线路机 <──50G── 客户端

每台中转机「每字节进一次、出一次」, 与上/下行方向无关:
  落地机: rx(50下+50上) + tx(50下+50上) = 200G(双向口径)
  线路机: rx(50下+50上) + tx(50下+50上) = 200G(双向口径)
```

→ 用户 100G ⇒ 落地机 200G、线路机 200G, **各自 ×2(不是合计 ×2)**; 协议封装另有几个百分点开销, 由闸一 10% 冗余吃掉。

> 线路机同理 ×2: 线路机共享池的容量 / 采购规划按 `Σ用户流量 × 2` 估; 但线路机到顶已有渐进疏散兜底(frontline-failover), 本文预算三道闸只管落地机。

由此, 「500G 落地机配 300G 套餐」是否成立**取决于机房计费口径**:

| 机房计费口径 | 机房配额 | `total_gb` 填法 | 限流阈值(默认 90%) | 承载力(用户口径, 阈值÷2) | 配 300G 套餐 |
|---|---|---|---|---|---|
| 双向(进+出都计, **默认**) | 500G | **500**(照抄面板) | 450G | 225G | ✗ 配不上 |
| 单向(只计出站) | 500G | **1000**(×2) | 900G | 450G | ✓ 余 150G |

> ⚠️ 现状两个问题, 是缺口一/二的根源: ① 限流阈值 **100% 触发、无冗余**, 且单向厂商无 ×2 填写规则; ② `meetsPlanSpec` 拿机器口径的 `total_gb` 直接和用户口径的套餐 GB 作 1:1 比较。
>
> **已定(§6 #1/#2)**: 采购默认双向计费, `total_gb` 照抄厂商面板, 冗余比例配在配额上(`usable_percent` 默认 90%) → 上表第一行是常态, **「500G 机器 + 300G 套餐」不成立**。二选一: 套餐降到 ≤225G 档, 或 300G 套餐只配 ≥670G 的双向机器(300 × 2 ÷ 0.9 ≈ 667G)。单向厂商 500G → 填 1000, 可配 ≤450G 套餐。

### 2.2 周期错配: 「售卖时算准」在数学上不存在

套餐周期从**购买日**起每 30 天一段(每个用户都不同); 机房预算按**自然月**翻篇。两套周期永远对不齐:

```
套餐周期(购买日起30天):  ├── 5/15 ~ 6/14 · 300G ──┼── 6/14 续费 ~ 7/13 · 又300G ──┤
机房预算(自然月):               ├────── 6/1 ~ 6/30 · 预算500G ──────┤
同一自然月最坏: 旧期尾 ≤300G + 新期头 ≤300G = 600G > 500G
```

→ 一个自然月最多横跨同一用户的**两个**套餐周期, 理论消耗上限 = 2 × 套餐额度。任何分配算法都算不准未来, 所以思路只能是「**少发生**(售卖看剩余) + **兜得住**(到顶停售 + 运营补量, 不换 IP)」, 不是精确预测。

### 2.3 二手回池: 当月被用过的机器再分给新用户

A 用掉一部分后退订 / 换走 → 机器回池 → B 接手, 但 B 的套餐额度 > 机器本月剩余预算 → B 月中遇到机器到顶。现状分配只看标称(500 ≥ 300 通过), 不看本月已耗。

## 3. 现状与缺口

- ✅ **计量层不动**: 换手 / 换机 / 归零 / 重装都不串账([traffic-model.md](traffic-model.md) §八), 本方案完全复用「换落地机 → 重打游标」的既有能力。
- ❌ **缺口一(口径)**: 限流阈值 100% 触发、无冗余, 单向厂商无 ×2 填写规则; `meetsPlanSpec` 机器口径与用户口径 1:1 混比(§2.1)。
- ❌ **缺口二(售卖)**: 分配与库存只看标称规格, 不看本月剩余预算(§2.3); 到顶的机器会被准入排除, 但「用了一半还没到顶」的照分。
- 🅾️ **缺口三(运行) —— 已定为「不自动处置 IP」**: 落地机到顶 / 掉线时, 现状是「停止接新用户 + 存量留原机」。曾设计自动换机(LandingFailoverJob)补这个口, 但**评审撤销**: 独占 IP 必须稳定, 自动换 IP 砸招牌(见 §4 闸三)。所以缺口三**有意不补自动换机**, 改为「闸一闸二预防 + 到顶停售 + 运营告警补量」。唯一未落地的是「告警」(可选, 后做); 到顶停新分配 / 停售是现状已有行为。

## 4. 方案: 三道闸

### 闸一 口径钉死(已定: 字段化阈值 + 填写规范)

- `total_gb` = **厂商面板原值照抄**(双向计费默认, 我们统计同为 rx+tx, 口径天然一致, 与厂商面板直接可对); 厂商明示单向计费的 **×2 放大填**(500G 单向 → 填 1000)。填写零心算, **永远不为留冗余而改小它**(那会失去对账锚点)。
- **冗余比例配在配额上, 不写死代码**: `resource_server_quota` 加 **`usable_percent`**(月配额实际可用比例, 管理端默认 90, 按机器可调); 限流阈值从现状的 `used_bytes ≥ total_gb`(100% 触发)改为 **`used_bytes ≥ total_gb × usable_percent`**(空值兜底 90)。默认 10% 冗余统一吃掉: 换机反应延迟的超支(见闸三校验)、agent / 装机部署流量、我们统计与厂商面板的口径误差; 个别机器要更保守, 调它的 `usable_percent`, 不动 `total_gb`。
- 承载力(能卖多大套餐, 用户口径) = `total_gb × usable_percent ÷ 2`; 套餐配套: 上架前按 `total_gb × usable_percent ≥ 套餐GB × 2` 反推机器是否配得上; 套餐管理页本就显示库存(`countCapacityForPlans`), 建完显示 0 台即规格配不上, 不必加码。
- 落点: `resource_server_quota` 加 `usable_percent`(DDL 默认 90 + admin 表单) + `markQuotaReached` 改读字段判阈值 + `total_gb` 字段注释(填厂商面板原值, 单向 ×2) + 运营手册一段。

### 闸二 售卖准入按「本月剩余」(防患)

- 规则一处定义: `ResourceServerRules.hasTrafficBudget(totalGb, currentUsedBytes, planTrafficGb)`:

  ```
  total_gb × usable_percent(同闸一阈值, 读同一字段) − 当周期 used_bytes ≥ 套餐GB × 2(口径放大) 换字节
  ```

  不另设售卖缓冲(早期草案的 ×1.1 取消): 杂项流量已由闸一冗余(默认 10%)覆盖, 贴线风险由闸三兜底, 一道冗余够了。

- **替代** `meetsPlanSpec` 的流量比较项(带宽项保留), 于是两条消费路径同时生效, 不另加判断点:
  - `findMatchingForPlan`(下单分配): 剩余不足的机器选不到;
  - `countCapacityForPlans`(库存展示): 剩余不足的机器不算「可售」→ 全区不足时套餐显示**售罄、不可下单**。
- 效果(全自动, 无人工状态):

  | 场景 | 行为 |
  |---|---|
  | 全新机器 / 月初机器 | used ≈ 0, 判定退化为标称检查, 行为同现状(但口径已修正) |
  | 当月被用过一半后回池 | 本月自动退出可售池; **下月预算翻篇 used 归零, 自动回池**, 无需「冷却」状态 |
  | 区域内全部空闲机剩余不足 | 库存算 0 → 售罄; 月初自动恢复有货(库存每次现算) |
  | 占用中的机器 | 不参与可售判定(现状已对, 计入 occupied) |

- **管不到续费**: 续费 / 翻篇不走售卖分配路径(机器已绑定), 老用户跨周期超支(§2.2)闸二无能为力 → 闸三必须存在。

### 闸三 到顶/掉线兜底: 保 IP 不变 + 运营补量(不自动换机)

> **决策(已定, 评审撤销自动换机)**: 独占 IP 的产品价值就是**长期稳定**(账号风控 / 白名单 / 登录态常绑死 IP)。为省流量自动换 IP = 砸独占 IP 的招牌, 本末倒置。故**不做 LandingFailoverJob 自动换机**(已撤, rollout-plan A4)。

到顶 / 掉线时的行为(到顶停售 / 停新分配是**现状已有**, 无新代码):

- **到顶**(used ≥ 闸一阈值 `total_gb × usable_percent`): `markQuotaReached` 置 THROTTLED → 准入网关 `filterAllocatable` 自动**停止接新用户** + 闸二库存自动**停售**这台机; **存量用户保留原机原 IP 继续用**。短时小超支由闸一冗余(默认 10%, 机房账面到 100% 才真超)+ 运营及时补量吸收。
- **掉线**(心跳 OFFLINE): 同样停新分配; 存量用户等机器恢复(独占机换机=换 IP, 不做)。
- **告警 + 运营决策**(唯一未落地项, 可选后做): 监控持续 THROTTLED / 长 OFFLINE 的落地机 → 通知运营 → 运营按情况**给机房补流量包 / 调高该机 `usable_percent` / 协调更换套餐**, 全程不动用户 IP。当前先靠 admin 看板的限流态人工巡检, 告警 job 留作后续可选项。

为什么够: 闸一(留冗余) + 闸二(挡住二手 / 不够的机器卖出去) 把「到顶」压成小概率; 真发生时, 牺牲一点机房超支换 IP 稳定, 符合独占 IP 的产品承诺。

> **运营手动换机仍支持**: 运营在 admin 主动换落地机(如机房永久收回某 IP)的场景仍走「改 `ip_id` → agent 对账收敛 → `meterCert` 重打游标账连续」。A3 已把 xray 出站/路由 tag 编入 ipId(`out_<certId>_<ipId>`), 正是为这种**手动**换机的对账收敛服务 —— 即「换机能力」保留, 只是不再由系统**自动**触发。

## 5. 场景推演(三道闸全开后)

| 场景 | 行为 | 账 |
|---|---|---|
| 新用户买 300G, 区域只剩「被用过半」的机器 | 售罄, 不可下单(闸二) | — |
| 月初机房预算翻篇 | 机器自动回池, 套餐自动恢复有货 | — |
| 老用户月中续费, 机器撑不住新周期(§2.2) | 用到顶 → 停售该机 + 保 IP 不变, 运营补流量包 / 调比例 | 不变, 不换机 |

| 机房账面与我方对不上 | 不对账(traffic-model §九); 闸一 10% 代码冗余吃口径差, 永远先于厂商到顶 | — |

## 8. 边界(本方案不做)

- 机房 API 精确对账(traffic-model §十二 既有边界)。
- 多接入点的预算精细分摊(先按全额度保守判, 表已就绪)。
- 跨区域换落地机(同区无可用 → 告警 + 运营补机)。
- 落地机间数据迁移(独占出口无状态, 换机即换 IP, 无数据可迁)。
- 共享落地机(traffic-multiuser.md, 评估中)的预算准入: 真开共享时, 闸二须从「空闲机才可售」升级为「剩余预算 ≥ 新套餐需求 + 已挂接入点的剩余承诺」; 本方案与 multiuser 正交(闸三换机在每接入点端口计量下同样成立), 先后落地顺序无碍。
