# 线路机故障切换 / 客户端多节点 方案

> 评审稿(未定)。解决「线路机(用户连接的入口机器)故障或流量耗尽时,用户连接中断、恢复慢」的问题。
> 思路:一个用户的订阅里放**一组**健康线路机入口,客户端自己在里面挑能连的切换;整组都挂再由后台改派 + 用户刷新订阅。
> **计量在落地机(见 [traffic-model.md](traffic-model.md) / [traffic-multiuser.md](traffic-multiuser.md)),本方案只动线路机侧,计量一行不改** —— 这是方案成立的基石。
> **分工**:候选组数据模型 / 选址 / 下发 / 故障切换(§10 步骤 1/2/4/5)走**核心线**;订阅渲染格式细节(决策 7/9 的组结构与参数、§10 步骤 3)与 §11 客户端走**客户端线**(另一会话);两线契约点 = cert 候选组字段(决策 8)。

---

## 1. 名词(先对齐,后面不再解释)

- **线路机**:用户客户端连接的**入口**机器(跑 xray,有域名)。多个用户共享一台。**会故障、要切换的就是它。**
- **落地机**:真正**出口**上网的机器(跑 socks5),用户**独占**一台 = 他买的那个 IP。**流量在这里计量。**
- **订阅**:用户导入客户端的那串链接 / 二维码,里面是一个或多个**节点**。
- **接入点(凭证)**:用户的一条连接身份,绑了「一个连接密钥 + 一个出口落地机」。一个订阅可以有多个接入点(多个出口 IP)。
- **对账**:后台改了数据库的「期望状态」,落地机/线路机上的 agent 把远端实际配置收敛成期望状态。

---

## 2. 现状与痛点

- 现在一个接入点**只绑一个线路机入口**(`cert.server_id`)。订阅里一个接入点 = 一个入口节点。
- 线路机挂了,后台定时任务把这个接入点**改派**到另一台线路机,但:
  - 客户端要**重新拉订阅**才知道换了入口;
  - 在重拉之前,**正在连的会话是断的**,客户端还在死磕那台挂掉的线路机。
- 即:现在的恢复是「最终恢复(等刷订阅)」,**救不了正在连的用户**,体感是一段明显的断网。

---

## 3. 为什么不做 DNS 切换(已否决,留档)

评审前讨论过「浮动域名 + DNS 健康切换」,否决,理由记录在此免得重提:

- 要多台同时分摊,得一个域名挂多条 A 记录(多 IP 轮询)→ 一个用户会被撒到多台 → **计量被迫跨机聚合**,复杂。
- DNS 有**生存时间(TTL)缓存延迟**,切换不是瞬时,而且同样**救不了正在连的会话**(客户端连接时只解析一次 IP)。
- 要让任意线路机都能接同一个域名,得上**通配符证书 + 全区证书互换**,是一次证书/域名架构重构。
- 还得依赖 Cloudflare(付费/锁定)或自建 DNS(又成单点)。

**结论**:用「客户端多节点」替代 DNS,**更省、能救正在连的会话、且完全不动证书/域名/DNS**。

---

## 4. 方案总览(链路)

一句话:**一个接入点(同一个连接密钥、同一个出口落地机)在订阅里渲染成 N 个线路机入口节点;客户端自动在这 N 个里挑能连的;后台定时任务维护这组线路机的健康(死一个补一个)。**

```
现状(单入口):
  客户端 ──连接──> 线路机A ──转发──> 落地机(独占出口) ──> 外网
  线路机A 挂 → 后台改派到线路机B → 用户重拉订阅才恢复
              └ 正在连的会话: 断, 直到重拉                      ✗ 痛点

方案(多入口):
  客户端 ┌──> 线路机A(主) ┐
         ├──> 线路机B(备) ┼──转发──> 落地机(独占出口) ──> 外网
         └──> 线路机C(备) ┘                    └ 计量在这里, 不随入口变
  线路机A 挂 → 客户端自己秒切到 B(节点已在手, 不用重拉, 救在连)  ✓
              后台: 发现 A 死 → 给这组补一台健康线路机
                    (改数据库期望状态 → agent 对账 → 配好该机 xray)
```

「一主两从」只是名义说法,实际就是**三个对等的健康入口**,客户端连得上哪个用哪个。

---

## 5. 关键澄清:为什么不影响流量统计(方案基石)

这是你最担心的点,结论是**不受影响**,原因:

- nook 的用户流量是从**落地机**(用户独占的出口)上读的,按接入点的出口落地机(`cert.ip_id`)记账 —— **从来不在线路机上按用户拆分**(线路机是共享入口,本来也拆不干净)。
- 你切换的是**线路机(入口)**;一个用户的**出口落地机始终是同一台**。
- 所以计量点(落地机)**纹丝不动**,用量游标继续累加,**不需要任何跨机聚合,计量代码一行不改**。

```
              切换发生在这一段(入口)
              ┌────────────────┐
  客户端 ════>│ 线路机 A/B/C 任切 │════> 落地机(独占·出口) ──> 外网
              └────────────────┘         │
                                          └─ 计量锚在这里; 入口怎么切, 这里不变
```

**必须守住的约束**:这 N 个节点是**同一个接入点**(同一个连接密钥、同一个出口落地机),只是 N 个不同的线路机入口 —— **不是 N 个独立接入点**。否则「一落地机 : 一接入点」的计量假设被打破,记账会乱。

> 一句话:**一接入点 · 一密钥 · 一出口落地机 · 多线路机入口**。

## 6. 各种故障下会发生什么(场景)

| 场景 | 结果 | 救正在连的会话? |
|---|---|---|
| 组里挂 **1 台**线路机 | 客户端秒切到组内其他健康入口 | ✅ 能 |
| 组里挂到**只剩 1 台** | 还能用那台幸存的 | ✅ 能 |
| **整组都挂 / 限流** | 后台改派到新健康线路机 + 用户**手动刷新订阅**恢复 | ❌ 要重连 |
| **全区没有可用线路机** | 告警,由**运营及时补线路机**;补一台自动进健康池 | ❌ 容量问题 |
| **落地机(出口 IP)挂了 / 预算耗尽** | **不自动换 IP**(保独占 IP 稳定):到顶→停售+留原机+运营补量;机房永久收回→运营手动换机。方案见 [landing-budget.md](landing-budget.md) | —(另文) |
| 用户**自己流量用完** | 挂起订阅(与故障切换无关,现有逻辑) | — |

边界要认清:**整组/全区都挂是容量问题,不是任何切换方案能解的** —— 靠告警 + 运营补机,不是堆更多备机(小区域里备机很快等于整区)。

## 7. 工程触点(给落地用,评审可略过)

要改的全在**线路机侧**,计量/额度/证书/DNS 都不动:

| 触点 | 现状 | 改成 |
|---|---|---|
| 接入点绑定 | `cert.server_id` 单线路机 | 加一个**候选线路机组**(单字段存,主排在前;同一密钥、同一落地) |
| 选址 | `TradeAllocator.pickFrontline` 选 1 台 | 选 **主 + N 备**(复用同一余量打分) |
| 下发 | agent 对账下发绑定那台 | 对账把「密钥 → 同一落地」的路由下发到**候选组每台**线路机 |
| 订阅渲染 | `renderSubscription` 每接入点 1 节点 | 每接入点渲染**一组**节点(各线路机各自域名,**同一密钥**) |
| 订阅格式 | base64 vmess 列表 + Clash YAML(`?format=clash`,**select 手选组**) | Clash 组 select → **fallback**(决策 7/9);sing-box 视客户端需要再加;**保留** vmess 列表给基础客户端 |
| 故障切换 | `FrontlineFailoverJob` 改 `cert.server_id`(开关默认关,未实开) | 升级成**维护健康候选组**(死一个补一个) |
| 计量 | 落地机计量 | ✅ **不动** |
| 证书 / 域名 / DNS | 一机一域名一证书 | ✅ **不动** |

---

## 8. 优缺点 / 边界

**优点**
- 救得了正在连的会话(客户端秒切),比现状「等刷订阅」体感好一档。
- **不动计量、不动证书/域名/DNS**,改动集中在线路机侧。
- 复用现成的「选址打分、对账下发、订阅渲染、定时任务」框架。
- 零中心件、零额外依赖(不引入 Cloudflare / 网关 / 自建 DNS)。

**缺点 / 代价**
- 自动切换的**好坏取决于客户端**:Clash / sing-box 类原生自动切;基础客户端可能要手动点另一个节点(见决策 3)。
- 一个密钥要**下发到 N 台线路机**,对账要给候选组每台配 xray(N=3 时是 3 倍下发,但都是轻量配置)。
- 备用线路机的**带宽余量**要不要预留,涉及容量取舍(见决策 4)。

**本方案不解决(边界)**
- 整组 / 全区都挂 → 容量问题,靠告警 + 运营补机。
- 落地机(出口 IP)故障 / 预算耗尽 → 不自动换 IP(保独占 IP 稳定),靠停售 + 运营补量,见 [landing-budget.md](landing-budget.md)。
- 用户自身流量耗尽 → 挂起订阅,现有逻辑。

## 9. 已定决策(评审锁定)

| # | 决策 | 选择 | 说明 |
|---|---|---|---|
| 0 | 切换速度目标 | **做多节点,允许短暂断一下** | 不追求零感知;客户端检测到挂掉再切,容忍几秒重连 |
| 1 | 候选组大小 | **3 台(一主两从)** | 名义主从,实为 3 个对等健康入口;区域不足 3 台时**有几台用几台**(≥1 即可开通),failover 巡检在出现新健康机后自动补满 |
| 2 | 候选集存储 | **接入点上单字段** | 不另建表;存这组线路机,主排在前 |
| 3 | 订阅格式 | **增 Clash / sing-box 格式** | 让主流客户端开箱自动切;同时保留 vmess 列表给基础客户端 |
| 4 | 备机带宽预留 | **软预留** | 不预占备机余量;某线路机余量不够时,后台把用户改派到更空的线路机 + 对账兜底 |
| 5 | 整组 / 全区全挂 | **运营补机** | 不做跨区降级;告警 + 及时补线路机 |
| 6 | 故障检测 | **沿用心跳** | 不加主动探活 |
| 7 | Clash 组类型 | **fallback(主从)** | `select` 是手选组,挂了不自动切(现状代码即 select,等于自动切换未开);`url-test` 平时就把流量散到备机,与决策 4 软预留「容量只按主记账」冲突;`fallback` 平时压主、主挂走备、主复活切回,与软预留自洽 |
| 8 | 候选集字段形态(细化决策 2) | **`server_id` 仍 = 主,新增 `standby_server_ids`(有序)** | 容量/计量「按主」的现有语义零改动;「查某线路机的凭证」拆三口径:对账 = 主+备(组内每台都下发) / 容量 = 只主(软预留) / 疏散 = 主+备(是备也要换) |
| 9 | 多接入点组结构 | **出口 select × 入口 fallback 两层** | 外层选出口 IP(用户手选),内层每接入点一组 3 入口自动容灾;单接入点套餐退化为 1 select 包 1 fallback |

> 决策 7 注(探测流量):fallback 组渲染须带健康检查参数 `url` + `interval`(取 120–300 秒,切换速度与探测开销的折中)。健康探测走「线路机 → 落地机 → 外网」**完整链路,计入用户账、也耗落地机预算**:约 10KB/次 × 3 节点,300 秒间隔 ≈ 每月几百 MB,60 秒间隔 ≈ 1.3G/月。量级可接受,订阅条款 / FAQ 写明「不使用也有微量流量」。

> 否决留档(免重提):① **客户端 API 直连层**(登录后拉结构化节点组 + 短轮询热重载)——mihomo 原生「订阅自动更新」把间隔调到 5–10 分钟即可逼近,多一个协议耦合面不值;留作日后运营发现「刷新级同步太慢」时的可选薄层。② **组中心化重构**(独立候选组表 + 版本号 + 推送通道)——「渐进生效」已被 3 台冗余兜底(见 §9.1),个位数 server 规模下复杂度盖过收益。

### 9.1 由决策推出的运行原则(重要)

决策 0 与 6 合在一起,得到一个清晰的**分工**:

- **快的一半交给客户端**:线路机挂了,**客户端自己**检测连接失败、秒级切到组内其他健康入口(允许几秒重连)。这一步**不依赖后台检测速度**。
- **慢的一半交给后台**:后台按**心跳(分钟级)**发现某线路机持续不健康 → 把它从受影响用户的候选组里**换成一台新的健康线路机**(改数据库 → agent 对账 → 配 xray)。

两者互补,且被决策 1(3 台冗余)兜底:

- 后台慢慢愈合期间,用户**还有另外 2 台**可用,体验不受影响。
- 只有 3 台**接连都挂**才会真正掉线 —— 那就是决策 5 的「整组挂 → 运营补机」。

> 注:后台改了候选组后,客户端要**下次刷订阅**才拿到新组;在那之前客户端用手里旧组里仍健康的节点照样能连。所以「改组」(无论故障愈合还是决策 4 的容量再平衡)是**渐进生效**的,不要求即时。

## 10. 改造小步(执行切分以 [rollout-plan.md](rollout-plan.md) 线 B 为准,本节为目标态摘要)

1. 接入点新增 `standby_server_ids`(有序备机,主仍在 `server_id`,决策 8);「查某线路机的凭证」拆三口径(对账 = 主+备 / 容量 = 只主 / 疏散 = 主+备);`pickFrontline` 扩成返「主 + N 备」一次选齐(同一余量打分取前 N)。
2. 下发:对账把「密钥 → 同一落地」路由下发到候选组每台线路机;退订全清(声明式对账天然收敛,无需显式清理)。
3. 订阅:`renderSubscription` 每接入点渲染候选组 N 个节点,Clash 按决策 7/9 出「出口 select × 入口 fallback」两层组(`?format=` 与 clash 渲染**已有**,本步实质是 select 改 fallback + 两层化);sing-box 视客户端需要再加;URL / 二维码界面已有。
4. 切换:`FrontlineFailoverJob`(现有)从「改单个绑定」升级成「组内换血」:某台 **挂/限流(沿用现检测)或余量不够(决策 4)** → 移出组、幸存备机顺位、新机补队尾(客户端手里旧订阅的幸存机还在,平滑)。
5. 联调:杀一台线路机 → 客户端 fallback 自动切;后台补组 → 对账下发;客户端刷订阅拿到新组;**全程核对落地机计量游标无跳变**。

## 11. 客户端侧(选做):最小换皮 fork

> 与上面后端方案**独立**:多节点 fallback 在任何 Clash 客户端都生效;本节是「要不要自有客户端」这条产品线,评审选 **B(最小换皮 fork)**。

**做什么(最小集)**

- fork **clash-party**(原 mihomo-party,Electron + Mihomo 核心,桌面端),换皮(名字 / 图标 / 配色)。
- 内置**登录**:调 `POST /portal/member/auth/login`(邮箱 + 密码,现成)。
- 登录后**自动拉节点**:取本人 `sub_token` → 拉 `/portal/sub/{token}?format=clash` → 导入为当前配置(替代手动导入),订阅自动更新间隔设 5–10 分钟(后台改组后靠这步渐进拿到新候选组,见 §9.1 注)。
- (可选)一个**套餐余量**页。
- 其余保持 clash-verge 原样,**不大改 UI**。

**后端触点(大多现成)**

| 触点 | 现状 |
|---|---|
| 登录 | ✅ `POST /portal/member/auth/login` 返 token + 会员信息 |
| 当前会员 | ✅ `GET /portal/member/profile/me` |
| 订阅 token | ✅ `/me` 已带 `subToken`;✅ `POST /portal/member/profile/sub-token/reset` 现成 |
| 订阅格式 | ✅ `?format=clash` 已有;⚠️ 差 select → fallback(决策 7,后端候选组改造的一部分) |
| 套餐余量 | ✅ `GET /portal/member/subscriptions` 已有(套餐 + 剩余额度 + 组名,组名与 clash 渲染同源) |

**选 B 即接受的硬约束**

- **GPL-3.0**:fork 必须开源(含登录 / 账户代码),不能闭源。
- **只覆盖桌面**(Win / Mac / Linux);**手机端不在内**,要的话另立线。
- **永续维护**:周期性追 clash-party + Mihomo 上游 + 重新签名发版。

**B 还要定的小项**

- 桌面先发哪几个平台(Win 必,Mac / Linux 看需求)。
- 手机端要不要、什么时候(独立决策)。
- 谁负责上游同步 + 发版(归属 / 流程)。
